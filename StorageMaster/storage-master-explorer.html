Storage Master Lab, from the inside

The Storage Master Lab is the deepest reference in this teaching set: 21 sections across six categories, each with a hand-built interactive playground and real generated Kotlin/SQL, all in a single offline HTML file. That file isn't hand-written — it's generated from three small source files. This page lets you do two things: use a live storage-decision chooser to learn what the lab teaches, and click through the build pipeline and source to learn how the lab is made.

One sentence to hold onto The lab is data + a renderer, baked into one file: sections.json holds the content, engine.js turns each section into interactive DOM, and build.mjs inlines both into a self-contained storage-tutorial.html that runs offline with no server and no JavaScript libraries — it only fetches optional web fonts, falling back to system fonts offline.

What the lab teaches · hands-on

Storage chooser — which API should I use?

The lab's whole premise is that most storage bugs are category errors — the right tool for the wrong kind of data. Answer a couple of questions and get a recommendation, with a one-line why and a jump into the matching lab section. It's inspired by the lab's own storage-chooser playground.

engine.js — DOM + a copyable code panel
function h(tag, cls, html) { const e = document.createElement(tag); if (cls) e.className = cls; if (html != null) e.innerHTML = html; return e; }
function esc(s) { return String(s == null ? "" : s).replace(/&/g, "&amp;").replace(/</g, "&lt;").replace(/>/g, "&gt;"); }

function codePanel(title, getCode, lang) {
  const panel = h("div", "codepanel");
  const pre = h("pre");
  const hl = lang === "sql" ? sql : lang === "xml" ? xml : kt;   // pick a highlighter
  panel.update = function () { pre.innerHTML = hl(getCode()); };  // re-render on demand
  panel.update();
  return panel;
}

The interactive part

The 10 playgrounds

Each section can carry one playground — a hand-built simulation of real Android behavior, no emulator required. The engine dispatches on the spec's kind; here is the full family.

🧭 storage-chooser

A decision tree: answer "which storage do I use?" and get a recommendation.

📍 device-map

Where each store lives on disk + what survives clear-cache / clear-data / uninstall.

🧵 threading

Main-thread blocking (ANR) vs a suspend call on Room's executor.

⚖️ prefs-datastore

SharedPreferences (sync, ANR-prone) vs DataStore (async, Flow-based).

🧬 entity-schema

Flip @Entity annotations → watch the generated Kotlin + CREATE TABLE change.

📇 dao-builder

Pick a DAO operation → see the generated method, its SQL, and a sample result.

💧 db-live

Insert / update / delete → watch a Flow<List<Note>> re-emit and the UI recompose.

🔄 type-converter

Feed a non-primitive Kotlin type → see the column value Room actually stores.

🪜 migration

Bump the schema version and watch it preserve, crash, or wipe the data.

☁️ backup-rules

Pick which buckets to back up → generate a data_extraction_rules.xml.

Adding one is two edits A new playground is one w… function in engine.js plus one entry in the WIDGETS map — then any section can request it by kind. The walkthrough shows the dispatch table.
