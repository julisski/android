A tutorial that builds itself

How the Master Lab is built

The Storage Master Lab (storage-tutorial.html) is the longest, deepest piece in this teaching set: 21 sections across six categories, each with a hand-built interactive playground and real generated Kotlin/SQL — all in a single offline HTML file. Unlike the other explainers, that file is not hand-written. It is generated from three small source files. This page follows that build, in order, as blocks of real code.

Node.js build 21 sections · 6 categories 10 interactive playgrounds no JS libraries single offline HTML
The one sentence to hold onto The lab is data + a renderer, baked into one file: sections.json holds the content, engine.js knows how to turn a section into interactive DOM, and build.mjs inlines both into a self-contained storage-tutorial.html that runs offline with no server and no JavaScript libraries — it only fetches optional web fonts, falling back to system fonts offline.

Why generate it instead of writing the HTML by hand?

  • Content stays separate from presentation. Editing a lesson means editing JSON, never touching layout code.
  • Every section is consistent. One renderSection function lays out all 21 — heading, summary, key points, playground, gotchas, canonical code — so nothing drifts.
  • It ships as one file. The build inlines the CSS, the engine, and the data, so the result is a single document you can email, host, or open from disk.

The shape of the build

Data → engine → build → one HTML file

Three inputs flow into one output. The data describes what to teach, the engine describes how to render and animate it, and the assembler welds them together with the stylesheet into the finished page.

Source file 1 of 3 · the curriculum

The data — a section is a JSON object

sections.json is an array of 21 objects, one per lesson. Each object is pure content — no markup decisions — so authoring a lesson never means touching layout code. Here is the shape, using the first section (The storage landscape) trimmed for length.

Why HTML lives in the JSON explanation_html, key_points, and gotchas_html hold real HTML and are inserted with innerHTML. That keeps rich formatting (links, <code>, lists) in the content layer — fine here because the author is the data source. If this content ever came from untrusted users, those fields would need sanitizing.

Source file 2 of 3 · the assembler

The build — inline everything into one file

build.mjs is a ~70-line Node script. It reads the three pieces, drops them into one HTML template, and writes the result. The key trick is on one line: const SECTIONS = ${JSON.stringify(sections)} turns the entire data file into a JavaScript global the engine can read with no fetch — which is what makes the page work offline from file://.

build.mjs · read, template, write
import { readFileSync, writeFileSync } from "node:fs";
import { dirname, join } from "node:path";
import { fileURLToPath } from "node:url";

const here = dirname(fileURLToPath(import.meta.url));
const css = readFileSync(join(here, "styles.css"), "utf8");
const engine = readFileSync(join(here, "engine.js"), "utf8");
const sections = JSON.parse(readFileSync(join(here, "sections.json"), "utf8"));
const out = join(here, "..", "storage-tutorial.html");

const html = `<!DOCTYPE html>
<html lang="en" data-theme="light">
...
<style>${css}</style>
...
<div id="sections"></div>   <!-- engine fills this on load -->
...
<script>
const SECTIONS = ${JSON.stringify(sections)};   // the whole curriculum, inlined
${engine}                                        // the renderer, inlined
</script>`;

writeFileSync(out, html);
console.log("wrote", out, "(" + html.length + " bytes,", sections.length, "sections)");
Same idea as this page The four explainer pages in this teaching set use the same "inline everything" approach — a small generator welds shared CSS and JavaScript into standalone HTML. One file, no build step to view it.

Source file 3 of 3 · the engine, bottom-up

The engine toolkit — tiny DOM + code helpers

engine.js is one IIFE. At the bottom are small, dependency-free helpers everything else is built from: element creators, an HTML escaper, three single-pass syntax highlighters, a copyable code panel, and a couple of form controls.

engine.js · DOM + escape helpers
// h(tag, class, html): the quick one. el(tag, attrs, kids): the structured one.
function h(tag, cls, html) { const e = document.createElement(tag); if (cls) e.className = cls; if (html != null) e.innerHTML = html; return e; }
function esc(s) { return String(s == null ? "" : s).replace(/&/g, "&amp;").replace(/</g, "&lt;").replace(/>/g, "&gt;"); }
function el(tag, attrs, kids) {
  const e = document.createElement(tag);
  if (attrs) for (const k in attrs) { if (k === "class") e.className = attrs[k]; else if (k === "html") e.innerHTML = attrs[k]; else if (k === "text") e.textContent = attrs[k]; else e.setAttribute(k, attrs[k]); }
  if (kids) kids.forEach(function (k) { if (k) e.appendChild(k); });
  return e;
}
engine.js · a copyable, highlighted code panel
function codePanel(title, getCode, lang) {
  const panel = h("div", "codepanel");
  const ch = h("div", "ch", "<span>" + esc(title) + "</span>");
  const copy = h("button", null, "Copy"); ch.appendChild(copy);
  const pre = h("pre");
  panel.appendChild(ch); panel.appendChild(pre);
  const hl = lang === "sql" ? sql : lang === "xml" ? xml : kt;     // pick a highlighter
  panel.update = function () { const c = getCode(); pre.innerHTML = hl(c); panel._raw = c; };
  copy.addEventListener("click", function () {
    navigator.clipboard && navigator.clipboard.writeText(panel._raw || getCode());
    copy.textContent = "Copied!"; setTimeout(function () { copy.textContent = "Copy"; }, 1200);
  });
  panel.update();
  return panel;
}
Three single-pass highlighters kt() (Kotlin), sql(), and xml() each run one regular expression over the code and wrap matches in <span class="tok-…">. A single pass means a string or comment can never be re-highlighted from inside — the source of most naive-highlighter bugs. The codePanel's update() re-renders on demand, which is how playgrounds show code that changes as you flip options.

The engine · one section

Rendering a section — the consistent layout

renderSection(s) turns one data object into one DOM <section>. It assembles the same parts in the same order every time, and each part is optional — present it only if the data has it. That single function is why all 21 sections look identical in structure.

engine.js · renderSection (condensed)
function renderSection(s) {
  const sec = h("section", "section"); sec.id = s.id;            // id = the anchor
  const head = h("div", "section-head");
  head.appendChild(h("div", "emoji", esc(s.emoji || "▫️")));
  const ht = h("div"); ht.appendChild(h("h2", null, esc(s.title)));
  if (s.category) ht.appendChild(h("span", "cat-badge", esc(s.category)));
  head.appendChild(ht); sec.appendChild(head);

  if (s.summary) sec.appendChild(h("p", "summary", esc(s.summary)));
  if (s.explanation_html) sec.appendChild(h("div", "explain", s.explanation_html));
  if (s.key_points && s.key_points.length) { /* build a "Key points" list */ }
  if (s.playground) sec.appendChild(buildPlayground(s.playground));   // the interactive part
  if (s.gotchas_html) sec.appendChild(h("div", "gotchas", "<h4>⚠️ Common mistakes</h4>" + s.gotchas_html));
  if (s.canonical_blocks) { /* a codePanel per {label, lang, code} */ }
  return sec;
}
  • Order is fixed: head → summary → explanation → key points → playground → gotchas → canonical code. Authors get a uniform reading rhythm for free.
  • Everything is guarded: a section with no playground or no gotchas simply omits that block — no empty shells.
  • Trusted vs escaped: titles/summaries go through esc(); the _html fields and canonical code are inserted as markup on purpose.

The engine · the interactive part

The playground system — a spec picks a widget

Each section's playground spec has a kind. buildPlayground looks that kind up in a WIDGETS map and calls the matching builder, wrapping it in a try/catch so one broken widget never blanks the page. This is a small dispatch table — adding a new playground type is one new function plus one map entry.

engine.js · the dispatch table + builder
const WIDGETS = {
  "db-live": wDbLive,                 "entity-schema": wEntitySchema,
  "dao-builder": wDaoBuilder,         "storage-chooser": wStorageChooser,
  "prefs-datastore": wPrefsDatastore, "device-map": wDeviceMap,
  "migration": wMigration,            "threading": wThreading,
  "type-converter": wTypeConverter,   "backup-rules": wBackupRules,
};

function buildPlayground(pg) {
  const root = h("div");
  root.appendChild(h("div", "pg-label", pg.title || "Interactive playground"));
  if (pg.intro) root.appendChild(h("p", "pg-intro", pg.intro));
  const board = h("div", "playground");
  const builder = WIDGETS[pg.kind];
  if (builder) { try { builder(pg, board); } catch (e) { board.appendChild(h("div", "hint", "playground error: " + esc(e.message))); } }
  else board.appendChild(h("div", "hint", "Unknown playground: " + esc(pg.kind)));
  root.appendChild(board);
  return root;
}

The engine · on load

init() — build the nav, the sections, and the chrome

When the page loads, init() reads the global SECTIONS, groups them by category to build the sidebar, renders every section into #sections, then wires a scroll-spy and the theme toggle. Everything you see is produced here from data.

engine.js · init (condensed)
function init() {
  const nav = document.getElementById("nav");
  const main = document.getElementById("sections");

  // 1) group sections by category, build the sidebar
  const cats = []; const byCat = {};
  SECTIONS.forEach(function (s) { const c = s.category || "Sections"; if (!byCat[c]) { byCat[c] = []; cats.push(c); } byCat[c].push(s); });
  cats.forEach(function (cat) { /* a nav-group heading + a link per section */ });

  // 2) render every section
  SECTIONS.forEach(function (s) { main.appendChild(renderSection(s)); });

  // 3) scroll-spy: highlight the nav link of the section in view
  const io = new IntersectionObserver(/* toggle .active */, { rootMargin: "-10% 0px -75% 0px" });
  SECTIONS.forEach(function (s) { const e = document.getElementById(s.id); if (e) io.observe(e); });

  // 4) theme toggle: flip data-theme on <html>
}
if (document.readyState === "loading") document.addEventListener("DOMContentLoaded", init); else init();
Data-driven all the way down The sidebar's groups, the section order, the anchors, and the scroll-spy targets are all derived from SECTIONS. Add an object to sections.json and a fully wired section + nav entry appears after the next build — no other code changes.

All together

The full build, step by step

From running one command to a rendered, interactive section in your browser — here is the whole chain, in order.

Scenario: you run node build.mjs, then open the file
1 / 6
  1. 1
    Read. build.mjs reads styles.css, engine.js, and parses sections.json into an array.build.mjs · readFileSync / JSON.parse
  2. 2
    Inline. It drops the CSS into a <style>, then writes const SECTIONS = […] and the whole engine into one <script>.build.mjs · the html template literal
  3. 3
    Write. writeFileSync emits ../storage-tutorial.html — a single self-contained document. The build is done.storage-tutorial.html (≈ one directory up)
  4. 4
    Open. You open the file in a browser. The inlined <script> runs; init() fires on DOMContentLoaded.engine.js · init()
  5. 5
    Render. init() groups SECTIONS into the sidebar, then calls renderSection for each, appending live DOM into #sections.engine.js · renderSection()
  6. 6
    Animate. Each section with a playground spec gets its widget built by buildPlayground → WIDGETS[kind], and the scroll-spy lights the nav as you read.engine.js · buildPlayground + IntersectionObserver
The point Steps 1–3 happen once, at build time, in Node. Steps 4–6 happen every time someone opens the file, in the browser — with no server and no JavaScript libraries (only an optional web-font fetch). Data and renderer, baked into one document.

Try it yourself

Build & extend

The build needs only Node — no install step, no dependencies.

terminal · from StorageMaster/storage-tutorial/
node build.mjs        # reads css + engine + json, writes ../storage-tutorial.html
# then open ../storage-tutorial.html in any browser (works from file://)

Adding a new lesson

  • Append an object to sections.json with a unique id, a category (reuse an existing one to slot it into that group), a title, summary, and explanation_html.
  • Pick a playground by setting playground.kind to one of the ten existing widgets — or add a new w… function in engine.js and one entry in the WIDGETS map.
  • Re-run node build.mjs. The new section, its nav link, and its scroll-spy target all appear automatically.
