QUESTION 1 1 pt · MC

What is the result of:

val nums = listOf(3, 1, 4, 1, 5); val r = nums.fold(1) { acc, n -> acc * n }
ANSWER ▸60 (Option 3)

How fold works

fold(initial) walks the list left to right, carrying a running accumulator. The lambda takes (acc, n) and returns the new accumulator. The seed you pass — here 1 — is the accumulator's starting value before any element is seen. Because the operation is multiplication, seeding with 1 (the multiplicative identity) means the seed doesn't distort the product.

Trace:
start   acc = 1
n=3  →  acc = 1 × 3 = 3
n=1  →  acc = 3 × 1 = 3
n=4  →  acc = 3 × 4 = 12
n=1  →  acc = 12 × 1 = 12
n=5  →  acc = 12 × 5 = 60

Why the distractors are there: 15 is the sum (3+1+4+1+5) — the trap for anyone who misreads * as +. 0 would be the answer only if the seed were 0 (anything ×0 = 0), the classic mistake of seeding a product with the additive identity. 14 and 61 are just off-by-one noise.

💡 Teaching point: the seed must match the operation's identity — use 1 for products, 0 for sums, emptyList() for list-building. Seeding a product with 0 is the single most common fold bug.
QUESTION 2 1 pt · MC

Given val count: Int = 42, which single line compiles in Kotlin?

ANSWER ▸val asLong: Long = count.toLong() (Option 3)

Why each option does / doesn't compile

✗ O1val asLong: Long = count — Kotlin does not perform implicit widening. Unlike Java, an Int will not silently become a Long. Type mismatch, won't compile.
✗ O2count as Long — as is a reference cast, checked against the class hierarchy. Int and Long are unrelated types (neither is a subtype of the other), so this can't convert and won't compile/throws.
✓ O3count.toLong() — the explicit conversion function. Every Kotlin numeric type exposes toByte(), toShort(), toInt(), toLong(), toFloat(), toDouble(). The idiomatic, type-safe way to widen.
✗ O4(Long) count — C/Java-style cast syntax. Kotlin has no such syntax; it's a parse error.
✗ O5Long(count) — Long is not a constructor you can call. There's no Long(...) factory.
💡 Teaching point: Kotlin deliberately removed implicit numeric conversions to prevent silent precision/range surprises. The only way up or down the numeric ladder is an explicit toXxx() call.
QUESTION 3 1 pt · MC

In which situation does a Kotlin when REQUIRE an else branch?

ANSWER ▸Expression assigned to a value, subject is an Int (Option 1)

The rule

Two conditions must both hold for else to be mandatory:

① The when is used as an expression — its value is consumed (assigned, returned, passed as an argument). A when used purely as a statement never requires else, because there's no result that could be "missing."

② The branches are not exhaustive — the compiler can't prove every possible subject value is handled. If it can't prove exhaustiveness, it forces else so the expression always produces a value.

Applying it to the options

✓ O1Expression + Int subject. An Int has ~4 billion values; you can't enumerate them, so the compiler can never prove exhaustiveness. else required.
✗ O2Sealed interface with every subtype covered → already exhaustive. The compiler knows the complete set of subtypes at compile time, so no else needed.
✗ O3Enum with every constant covered → already exhaustive, same reasoning. No else needed.
✗ O4Used as a statement, result ignored → exhaustiveness is irrelevant. No else needed.
✗ O5Subjectless when used as a statement → again a statement. No else needed.
💡 Teaching point: "expression + can't-be-proven-exhaustive ⇒ else." Sealed classes/interfaces and enums are special precisely because they give the compiler a closed set of possibilities to check against. Adding a new sealed subtype later turns a previously-exhaustive when into a compile error — a feature, not a bug, since it forces you to handle the new case.
QUESTION 4 1 pt · MC

user: User?; its address is a non-null Address; address.city is a non-null String. Inferred type of:

val c = user?.address?.city
ANSWER ▸String? (Option 2)

Why the chain is nullable

The safe-call operator ?. has one defining behavior: if the left side is null, the whole expression short-circuits to null and the right side is never evaluated.

Here, user is User?. If user is null, the entire expression evaluates to null immediately — it never even looks at address or city. So the result must be able to hold null, which means the type is nullable regardless of how non-null the deeper properties are.

The non-nullness of address and city only tells you that if the chain reaches them, they won't add additional null sources. But the very first ?. already poisoned the result type with nullability. One nullable link is enough.

Type derivation:
user        : User?
user?.address : Address?  (? propagates from user)
...?.city    : String?  (? still propagating)

Distractors: String (O1) would be right only if user were non-null. Address? (O3) is the type of the intermediate user?.address, not the final result. User? (O4) is just the starting type. And ?. chaining absolutely compiles (O5) — that's its whole purpose.

💡 Teaching point: to collapse the result back to non-null, end the chain with the Elvis operator — user?.address?.city ?: "unknown" infers String, because Elvis guarantees a non-null fallback.
QUESTION 5 1 pt · MC

Four are equivalent ways to pass the same predicate to filter. Which is NOT valid Kotlin?

ANSWER ▸nums.filter(it > 2) (Option 5)

filter's signature

filter expects a function: filter(predicate: (T) -> Boolean). You must hand it something callable that takes an element and returns a Boolean — i.e. a lambda. The four valid options are just different surface syntaxes for handing over the same lambda { it > 2 }.

valid O1filter({ it > 2 }) — lambda passed normally inside the parentheses.
valid O2filter() { it > 2 } — empty parens + trailing lambda. Legal but unusual.
valid O3filter { it > 2 } — trailing-lambda syntax: when the lambda is the last (here, only) argument, it can move outside and the parens drop. The idiomatic form.
valid O4filter(predicate = { it > 2 }) — named argument, lambda as the value.
✓ O5filter(it > 2) — this is the answer. No braces, so it > 2 is not a lambda — it's an attempt to evaluate an expression. Worse, it has no meaning here (no lambda scope to define it), and even if it did you'd be passing a Boolean where a function is required. Two errors at once.
The trap: the missing braces are easy to miss when skimming. { it > 2 } is a function value; it > 2 is a (broken) boolean expression. The braces are what create the lambda and bring the implicit it parameter into scope.
QUESTION 6 2 pts · Code

Return the length of s, or 0 if s is null. Use ?. and ?: — one line.

fun safeLength(s: String?): Int = s?.length ?: 0

How it reads

Evaluate left to right:

s?.length — safe-call. If s is non-null, this is its length (an Int). If s is null, the safe-call short-circuits to null, giving an intermediate type of Int?.

?: 0 — Elvis. If the left side is non-null, use it; if it's null, fall back to 0. This collapses the Int? back down to a plain Int, satisfying the declared return type.

Cases:
safeLength("hello") → "hello".length = 5 → 5
safeLength("")     → "".length = 0 → 0
safeLength(null)  → null ?: 0 → 0
Common wrong answers: s.length won't compile (s is nullable, no safe-call). if (s != null) s.length else 0 works but ignores the instruction to use ?. and ?:. s?.length ?: -1 returns the wrong fallback.
QUESTION 7 2 pts · Code

Return a new list of only the even numbers, each doubled, in original order. Chain filter then map — no manual loop.

fun evensDoubled(nums: List<Int>): List<Int> =
    nums.filter { it % 2 == 0 }.map { it * 2 }

Why this order

filter { it % 2 == 0 } keeps only elements where the predicate is true — the evens. map { it * 2 } then transforms each surviving element by doubling it. Both operations preserve input order and both return new lists (the original nums is untouched), which is what "return a new list" demands.

Order of operations matters — and the question says so explicitly. Filtering first means map runs on fewer elements (cheaper), and it produces the right answer: you want "evens, doubled," not "all values doubled, then keep the ones that are even." Reversing them would double odds too before discarding them — wasted work and a different intent.

Trace:  listOf(1, 2, 3, 4, 5, 6)
after filter (evens) → [2, 4, 6]
after map (×2)     → [4, 8, 12]
Watch for: it % 2 == 1 catches odds, not evens (and breaks on negative numbers, where the remainder can be -1). Use == 0. A manual for loop with an accumulator list would produce the same output but is explicitly disallowed here.
QUESTION 8 2 pts · Code

Implement code that uses four functions chained together.

fun fancySum(nums: List<Int>): Int =
    nums.filter { it > 0 }.map { it * 2 }.distinct().sum()

What each link does

filter{ it > 0 } — keeps positives, drops zero and negatives. Returns List<Int>.
map{ it * 2 } — doubles each survivor. Still List<Int>.
distinctremoves duplicate values, keeping the first occurrence and preserving order. Returns List<Int>.
sumadds all elements, returning a single Int — matches the function's return type and is what makes the chain terminate in a value rather than a list.
Trace:  listOf(-1, 2, 2, 3, 0, 3)
filter >0  → [2, 2, 3, 3]
map ×2    → [4, 4, 6, 6]
distinct → [4, 6]  (dupes from doubling collapse)
sum      → 10

Note the behavioral wrinkle: distinct() here actually changes the result, not just tidies the list. Two different positive inputs that double to the same value contribute only once to the sum. Worth flagging to students as the substantive consequence of including distinct in the chain.

Any chain of four standard collection functions that ends in an Int satisfies the prompt — e.g. filter().map().take(3).sum(), or filter().map().sorted().first(). The terminal operation must collapse the list to a scalar Int.

⚠ Grading note: Q8 is open-ended as written, so many valid answers exist. To make student submissions comparable, consider pinning the spec down (e.g. "filter positives, map ×2, take the first 3, sum"). Otherwise grade on: (a) exactly four chained calls, (b) the chain type-checks, and (c) it returns an Int.
