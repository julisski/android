Jetpack Compose Quiz — Answer Key & Explanations

Five multiple-answer questions. Each item explains why every option is correct or incorrect, with special focus on the wrong answers.

Green = correct answer (should be selected) Red = incorrect (should be left unselected) These are “select ALL that apply” questions — picking a red option or missing a green one loses credit.
QUESTION 1
Which of the following are valid Jetpack Compose Text() calls? Assume all required imports (Text, FontWeight, Color, sp, MaterialTheme) are present. Select ALL that are valid.
✓ VALID — correct answerOption 1
Text("Hello")
Why it's valid: text is the only required parameter of Text. Every other parameter (modifier, color, fontSize, style, …) has a default value, so the simplest legal call is just a string.
✓ VALID — correct answerOption 2
Text("Body text in bodyMedium", style = MaterialTheme.typography.bodyMedium)
Why it's valid: style is a real Text parameter of type TextStyle, and MaterialTheme.typography.bodyMedium returns a TextStyle from the Material 3 type scale — a type-correct argument.
✓ VALID — correct answerOption 3
Text("Bold + primary color", fontWeight = FontWeight.Bold, color = MaterialTheme.colorScheme.primary)
Why it's valid: fontWeight (FontWeight?) and color (Color) are both genuine parameters. FontWeight.Bold and MaterialTheme.colorScheme.primary are the right types, passed as comma-separated named arguments.
✗ INVALIDOption 4
Text("Big and red", fontSize = 22.sp, color = Color.Red
Why it's wrong: the closing parenthesis is missing. The call opens ( but never closes it, so it fails to compile with “Expecting ')'”. Note that the arguments themselves are all perfectly valid — this is purely a syntax typo, and it is the most common one in real code.
Fix: add the ) → Text("Big and red", fontSize = 22.sp, color = Color.Red)
✗ INVALIDOption 5
Text("Sized", size = 20.sp)
Why it's wrong: Text has no parameter named size. The compiler reports “Cannot find a parameter with this name: size.” The parameter that controls text size is fontSize. (size sounds right because Modifier.size() exists for layout — but that is a modifier, not a Text parameter.)
Fix: Text("Sized", fontSize = 20.sp)
✗ INVALIDOption 6
Text(color = Color.Blue)
Why it's wrong: the required text argument is missing. Because text has no default, omitting it gives “No value passed for parameter 'text'.” A Text with a color but no string has nothing to render.
Fix: supply a string → Text("Blue", color = Color.Blue)
QUESTION 2
A single Text("Hello") is placed directly inside setContent { } with no Modifier, Scaffold, Surface, or other layout wrapper. Which of the following are TRUE about where and how it is displayed by default? Select ALL that apply.
✓ TRUE — correct answerOption 1

It appears in the top-left corner of the screen (the top-start position in a left-to-right layout).

Why it's true: a single root composable is placed at the origin (0, 0) of the available space, which is the top-left in an LTR layout — “top-start” in Compose terms (start = left in LTR, right in RTL). Compose applies no implicit centering to root content.
✓ TRUE — correct answerOption 2

It sits at the very top and can overlap the status bar, because no padding or window insets are applied by default.

Why it's true: nothing applies insets to raw content, so the text draws from the very top of the window and can sit under the status bar. This is exactly why apps add a Scaffold (which hands you innerPadding), a Surface, or Modifier.statusBarsPadding()/safeDrawingPadding().
✓ TRUE — correct answerOption 3

It takes up only as much width as its text needs, wrapping its content instead of filling the screen width.

Why it's true: Text measures itself to fit its content (wrap-content). With no width modifier it is only as wide as “Hello”. Filling the width is opt-in via Modifier.fillMaxWidth().
✗ FALSEOption 4

It is automatically centered horizontally and vertically on the screen.

Why it's wrong: Compose does not center root content. Centering is explicit — e.g. Box(Modifier.fillMaxSize(), contentAlignment = Alignment.Center). With none of that present, the text stays at the top-start, not the center. (This is the most tempting wrong answer because many other tools center by default.)
✗ FALSEOption 5

It appears in the bottom-left corner of the screen by default.

Why it's wrong: the default origin is the top, not the bottom; content grows downward from the top-start. Bottom placement would need something explicit like Box(Modifier.fillMaxSize(), contentAlignment = Alignment.BottomStart).
✗ FALSEOption 6

It automatically stretches to fill the entire width of the screen.

Why it's wrong: the opposite of Option 3 — Text wraps its content and does not stretch. Filling width requires Modifier.fillMaxWidth(). (Options 3 and 6 directly contradict each other; only the wrap-content behavior is correct.)
QUESTION 3
Which of the following statements about the Jetpack Compose Column composable are TRUE? Select ALL that apply.
✓ TRUE — correct answerOption 1

Column places its children vertically, one above the next, in the order they are declared (first child at the top).

Why it's true: Column is the vertical layout primitive. It stacks children along the vertical (main) axis in declaration order — first child on top, next below it.
✓ TRUE — correct answerOption 2

The verticalArrangement parameter controls how children are positioned and spaced along the vertical (main) axis.

Why it's true: a Column's main axis is vertical, so verticalArrangement (an Arrangement.Vertical such as Arrangement.Top or Arrangement.spacedBy(8.dp)) governs top-to-bottom positioning, spacing, and distribution of leftover vertical space.
✓ TRUE — correct answerOption 3

The horizontalAlignment parameter controls how children line up along the horizontal (cross) axis, for example Alignment.CenterHorizontally.

Why it's true: a Column's cross axis is horizontal, so horizontalAlignment (an Alignment.Horizontal such as Alignment.Start or Alignment.CenterHorizontally) decides how each child lines up across the column's width.
✗ FALSEOption 4

Column places its children horizontally, side by side.

Why it's wrong: horizontal, side-by-side layout is what Row does. A Column is vertical by definition. Swapping Column and Row is the classic mistake this option tests.
✗ FALSEOption 5

A Column scrolls vertically by default, with no extra modifier needed.

Why it's wrong: a plain Column is not scrollable — overflowing children are clipped. You must add Modifier.verticalScroll(rememberScrollState()). The composable that scrolls automatically (and lazily) is LazyColumn, which is a different thing — confusing the two is what makes this tempting.
✗ FALSEOption 6

You space the children of a Column apart by setting a horizontalArrangement parameter.

Why it's wrong: Column has no horizontalArrangement parameter (passing one is a compile error). Spacing along a Column's main axis uses verticalArrangement, e.g. verticalArrangement = Arrangement.spacedBy(8.dp). horizontalArrangement belongs to Row.

Key concept — Column vs. Row (ties Q3 & Q4 together)

ColumnRow
Main axisVertical (top → bottom)Horizontal (start → end)
Position/space on main axisverticalArrangementhorizontalArrangement
Align on cross axishorizontalAlignmentverticalAlignment
Scrolls by default?No — use verticalScroll or LazyColumnNo — use horizontalScroll or LazyRow

Mnemonic: the Arrangement word matches the main-axis direction; the Alignment word matches the cross-axis direction.

QUESTION 4
Which of the following statements about the Jetpack Compose Row composable are TRUE? Select ALL that apply.
✓ TRUE — correct answerOption 1

Row places its children horizontally, side by side, in the order they are declared (first child at the leading edge).

Why it's true: Row is the horizontal layout primitive. It places children along the horizontal (main) axis in declaration order — first child at the leading/start edge (left in LTR), each next child to its right.
✓ TRUE — correct answerOption 2

The horizontalArrangement parameter controls how children are positioned and spaced along the horizontal (main) axis, for example Arrangement.SpaceBetween.

Why it's true: a Row's main axis is horizontal, so horizontalArrangement (an Arrangement.Horizontal such as Arrangement.SpaceBetween or Arrangement.spacedBy(8.dp)) governs left-to-right positioning, spacing, and distribution of leftover horizontal space.
✓ TRUE — correct answerOption 3

The verticalAlignment parameter controls how children line up along the vertical (cross) axis, for example Alignment.CenterVertically.

Why it's true: a Row's cross axis is vertical, so verticalAlignment (an Alignment.Vertical such as Alignment.CenterVertically) decides how each child lines up within the row's height.
✗ FALSEOption 4

Row stacks its children vertically, top to bottom.

Why it's wrong: vertical, top-to-bottom stacking is what Column does. A Row is horizontal by definition — the Column/Row swap again.
✗ FALSEOption 5

A Row scrolls horizontally by default, with no extra modifier needed.

Why it's wrong: a plain Row is not scrollable; overflow is clipped. Add Modifier.horizontalScroll(rememberScrollState()), or use the lazy, auto-scrolling LazyRow — a different composable.
✗ FALSEOption 6

You align the children of a Row on the cross axis by setting a horizontalAlignment parameter.

Why it's wrong: Row has no horizontalAlignment parameter. On a Row the cross axis is vertical, so cross-axis alignment uses verticalAlignment; horizontalAlignment belongs to Column. This option mixes up the two composables' parameters.
QUESTION 5
Which of the following statements about the Jetpack Compose Material 3 Card composable are TRUE? Select ALL that apply.
✓ TRUE — correct answerOption 1

Card is a Material 3 container (built on top of Surface) used to group related content, and it applies a themed background color and rounded-corner shape by default.

Why it's true: Card is implemented on top of Surface. Out of the box it applies a default rounded shape (CardDefaults.shape), a themed container/content color (CardDefaults.cardColors()), and a small elevation — so you get a visible card without styling it yourself.
✓ TRUE — correct answerOption 2

The content you pass to a Card is a ColumnScope lambda, so by default its children are stacked vertically, the same way a Column lays them out.

Why it's true: the card's content is declared as @Composable ColumnScope.() -> Unit. Because the receiver is ColumnScope, children stack vertically by default and you can use ColumnScope modifiers like Modifier.align() and Modifier.weight(). (For horizontal content, put a Row inside the card.)
✓ TRUE — correct answerOption 3

A Card can handle taps directly by using the Card overload that takes an onClick parameter (and an enabled flag).

Why it's true: Card has an overload whose first parameter is onClick: () -> Unit (plus enabled: Boolean = true). That makes the whole card a clickable surface with the proper ripple and accessibility semantics — no wrapper needed. A separate display-only overload (no onClick) also exists.
✗ FALSEOption 4

By default a Card lays out its children horizontally, the same way a Row does.

Why it's wrong: a card's content scope is ColumnScope, so the default is vertical stacking (like a Column), not a Row. You only get horizontal content if you explicitly place a Row inside. This is the Card-specific version of the Column-vs-Row trap.
✗ FALSEOption 5

A Card has no default shape or container color, so nothing is drawn until you set shape and colors explicitly.

Why it's wrong: the opposite of Option 1. Card does have defaults — a rounded shape, a themed container color, and elevation from CardDefaults. It is fully visible with zero styling; you override shape, colors, or elevation only when you want something different.
✗ FALSEOption 6

A Card cannot respond to clicks on its own; you must wrap it in a Button or attach a separate clickable to make it tappable.

Why it's wrong: as Option 3 shows, the onClick overload makes a card clickable on its own — no Button wrapper required (and wrapping a whole card in a Button would be the wrong pattern). You may add Modifier.clickable, but it is not required.
