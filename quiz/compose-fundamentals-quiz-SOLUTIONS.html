QUESTION 1
Which of the following are valid Jetpack Compose Text() calls? Assume all required imports (Text, FontWeight, Color, sp, MaterialTheme) are present. Select ALL that are valid.
At a glance Options 1, 2, and 3 are valid; they all pass a non-null String as the required text argument and only add parameters that genuinely exist on the Text() signature (style, fontWeight, color), while options 4, 5, and 6 each break a hard rule (missing closing paren, a non-existent size parameter, and a missing required text argument).
✓ VALID — correct answerOption 1
Text("Hello")
Why: The Material 3 Text() composable has exactly one required parameter, text: String (a separate overload takes text: AnnotatedString); every other parameter such as modifier, color, fontSize, fontWeight, and style has a default value. Passing a single string literal therefore satisfies the signature, and the omitted parameters fall back to their defaults: color = Color.Unspecified, fontSize = TextUnit.Unspecified, fontWeight = null, style = LocalTextStyle.current. When a value is Unspecified, Text resolves it from the effective style (which by default is LocalTextStyle.current, supplied by MaterialTheme), so the call is both syntactically and semantically complete. This is the canonical minimal Compose text call.
✓ VALID — correct answerOption 2
Text("Body text in bodyMedium", style = MaterialTheme.typography.bodyMedium)
Why: The first argument is supplied positionally as the required text, and style is supplied as a named argument of type TextStyle. MaterialTheme.typography returns the active Typography object, and bodyMedium is one of its predefined TextStyle roles, so the type matches style: TextStyle = LocalTextStyle.current. The supplied style becomes the baseline, and any explicit per-call parameters (none here) would override individual fields on top of it. Mixing one positional argument with later named arguments is legal Kotlin as long as the positional arguments come first.
✓ VALID — correct answerOption 3
Text("Bold + primary color", fontWeight = FontWeight.Bold, color = MaterialTheme.colorScheme.primary)
Why: All three parameters exist on Text() and the types match: fontWeight: FontWeight? accepts FontWeight.Bold, and color: Color accepts MaterialTheme.colorScheme.primary, which returns a Color from the active color scheme. These are independent style axes that override only their own field of the resolved TextStyle; setting weight and color does not disturb font size or family. Using colorScheme.primary instead of a hardcoded literal is the theme-aware, dark-mode-safe way to color text. Once the positional text comes first, the named arguments may appear in any order.
✗ INVALIDOption 4
Text("Big and red", fontSize = 22.sp, color = Color.Red
Why: Conceptually every part of this call is fine: fontSize is a real parameter of type TextUnit and 22.sp produces a TextUnit via the Int.sp extension, while color = Color.Red is a valid Color. The failure is purely mechanical: the closing parenthesis of the call is missing. Kotlin cannot find the end of the argument list, so the compiler reports a syntax error such as Expecting ')'. A balanced-delimiter mistake aborts parsing before any type checking matters.
Fix: Add the closing parenthesis: Text("Big and red", fontSize = 22.sp, color = Color.Red)
✗ INVALIDOption 5
Text("Sized", size = 20.sp)
Why: There is no parameter named size on the Text() composable, so the compiler fails with Cannot find a parameter with this name: size. The confusion comes from two separate concepts: the text's character size is controlled by fontSize: TextUnit, whereas size is a Modifier function (Modifier.size(...)) that sets the layout bounds of a composable, not its glyph size. The value 20.sp is itself a valid TextUnit; it is simply being bound to a name that does not exist. Naming a parameter that the signature does not declare is a name-resolution error, distinct from a type mismatch.
Fix: Use the real text-size parameter: Text("Sized", fontSize = 20.sp)
✗ INVALIDOption 6
Text(color = Color.Blue)
Why: color is a real, optional parameter, but the call never supplies the one required parameter, text. Because text has no default, the compiler reports No value passed for parameter 'text'. Optional parameters can be omitted; required ones cannot, and naming an optional parameter does not satisfy the requirement for the mandatory one. There is also no meaningful way to render colored text when there is no text content to color.
Fix: Provide the required text argument: Text("Blue text", color = Color.Blue)

⚠ Common traps

  • Confusing size with fontSize. Modifier.size() sets the composable's layout box; the character size is the fontSize: TextUnit parameter. There is no size parameter on Text(), so size = 20.sp fails with Cannot find a parameter with this name: size.
  • Forgetting that text is the only required argument. Naming optional parameters like color alone does not satisfy the call; you still get No value passed for parameter 'text'.
  • Unbalanced parentheses. A perfectly correct argument list still fails to compile if the closing ) is missing. This is a syntax error caught during parsing, before type checking, so the message points at punctuation (Expecting ')'), not types.
  • Hardcoding colors instead of using the scheme. Color.Red compiles, but MaterialTheme.colorScheme.primary is the theme-aware, dark-mode-safe choice; both are valid Kotlin, only one survives a theme change gracefully.

Key concept

Memorize the Text() contract: the only required argument is text (a String, or an AnnotatedString via a separate overload); everything else (modifier, color: Color, fontSize: TextUnit, fontWeight: FontWeight?, style: TextStyle, ...) is optional with a default, and the Unspecified defaults resolve from the effective style / ambient MaterialTheme. To validate any Text() call, check three things in order: (1) syntax — are the parentheses balanced? (2) the required arg — is text supplied (positionally first, or named)? (3) parameter names and types — does each named argument actually exist on the signature with a matching type? Remember the two classic gotchas: the text-size parameter is fontSize (fed by Int.sp), not size (which is a Modifier function affecting layout bounds), and an explicit style sets a baseline TextStyle that individual parameters then override field-by-field.

QUESTION 2
A single Text("Hello") is placed directly inside setContent { } with no Modifier, Scaffold, Surface, or other layout wrapper. Which of the following are TRUE about where and how it is displayed by default? Select ALL that apply.
At a glance Options 1, 2, and 3 are TRUE — because raw content passed to setContent is placed at the composition root's origin (top-start, i.e. top-left in LTR) with no implicit centering, no window-inset padding, and a Text measures to wrap-content rather than filling the screen.
✓ TRUE — correct answerOption 1

It appears in the top-left corner of the screen (the top-start position in a left-to-right layout).

Why: setContent installs your composables into an AndroidComposeView whose root acts like a single-child container that places its child at coordinate (0, 0) — the top-start corner. In a left-to-right (LTR) layout direction, top-start resolves to top-left; in an RTL locale the same code would render top-right because Compose expresses placement in start/end terms, not hard left/right. There is no parent Box with contentAlignment, no Column/Row arrangement, and no gravity, so nothing shifts the child away from the origin. The Text is therefore laid out flush against the top-left of the available space. This is the single most important default to memorize: unwrapped root content lands at the top-start origin.
✓ TRUE — correct answerOption 2

It sits at the very top and can overlap the status bar, because no padding or window insets are applied by default.

Why: Window insets (the status bar, navigation bar, display cutout, IME, etc.) are not applied automatically to raw content handed to setContent — they are only consumed when you opt in via something like Scaffold (which hands you innerPadding) or modifiers such as Modifier.statusBarsPadding() / Modifier.safeDrawingPadding(). Because modern apps draw edge-to-edge (and enableEdgeToEdge() makes the system bars transparent), the composition fills the whole window including the area behind the status bar. With the Text pinned at the top origin and no inset padding, its glyphs can render underneath the status-bar clock and icons. The fix is to consume insets yourself; the default does not do it for you.
✓ TRUE — correct answerOption 3

It takes up only as much width as its text needs, wrapping its content instead of filling the screen width.

Why: During the measure pass, Text reports a measured size just large enough to hold its rendered string (subject to the incoming constraints) — this is wrap-content behavior, the default for nearly every leaf composable. Without a width modifier, the measured width equals the text's line width, not the parent's max width, so the node occupies only the leftmost slice of the available space. The parent's maxWidth constraint is the screen width, but a child is free to choose any size between the min and max constraint, and Text chooses the minimum it needs. To make it span the screen you must explicitly opt into a larger size with Modifier.fillMaxWidth(). Wrap-content is the rule; fill is the exception you request.
✗ FALSEOption 4

It is automatically centered horizontally and vertically on the screen.

Why: There is no implicit centering anywhere in Compose's layout system — centering is always an explicit opt-in. The root behaves like a single-child container placing its child at the top-start origin, with no contentAlignment applied. To actually center on both axes you must wrap the text in a Box that fills the window and set its contentAlignment, or use a Column/Row with matching arrangement (main axis) and alignment (cross axis). The exam trap is assuming a phone screen 'naturally' centers a lone widget the way some other UI toolkits' default gravity might; Compose does not.
Fix: Wrap it in a full-size Box with contentAlignment:
Box(modifier = Modifier.fillMaxSize(), contentAlignment = Alignment.Center) { Text("Hello") }
✗ FALSEOption 5

It appears in the bottom-left corner of the screen by default.

Why: The default origin for root content is the top-start corner, not the bottom. Compose's coordinate system has its y-axis increasing downward from (0, 0) at the top-left, and the root container places its single child at that top origin. Nothing pushes the child toward the bottom unless you deliberately add vertical space (for example a Column filling the height with verticalArrangement = Arrangement.Bottom, or Alignment.BottomStart in a full-size Box). So 'bottom-left by default' is simply the wrong corner.
Fix: To deliberately pin to the bottom-left, fill the height and align to bottom-start:
Box(modifier = Modifier.fillMaxSize(), contentAlignment = Alignment.BottomStart) { Text("Hello") }
✗ FALSEOption 6

It automatically stretches to fill the entire width of the screen.

Why: A leaf composable never auto-stretches; it measures to wrap-content and chooses the minimum width it needs, as covered in option 3. Filling the available width is an explicit request you make with Modifier.fillMaxWidth() (or fillMaxSize(), or Modifier.width(...)), which sets the child's width to the parent's max width. Even when a Text fills its width, the glyphs still start at the start edge unless you also set textAlign = TextAlign.Center — filling the box and centering the text are two separate steps. Without any modifier, the node is exactly as wide as the word 'Hello'.
Fix: Make the text span and center its glyphs:
Text("Hello", modifier = Modifier.fillMaxWidth(), textAlign = TextAlign.Center)

⚠ Common traps

  • Assuming a phone 'naturally' centers a single widget. Compose never centers implicitly — the root places its child at the top-start origin. Centering requires a full-size Box with contentAlignment = Alignment.Center (or a Column/Row with matching arrangement and alignment).
  • Forgetting that edge-to-edge means no free inset padding. With enableEdgeToEdge() (or by default for raw content), your UI draws behind the status and navigation bars. Use Scaffold's innerPadding, or Modifier.statusBarsPadding() / Modifier.safeDrawingPadding(), or the clock will overlap your text.
  • Confusing fillMaxWidth() with centering the text. Modifier.fillMaxWidth() only stretches the node's width; the glyphs still hug the start edge unless you also pass textAlign = TextAlign.Center. They are two separate steps.
  • Thinking 'left/right' instead of 'start/end'. Compose resolves placement by layout direction. The default top-left is really top-start, so under an RTL locale the very same code renders top-right.
  • Mixing up arrangement and alignment. For a Row/Column, Arrangement positions children along the main axis (horizontal for Row, vertical for Column) while Alignment positions them on the cross axis — they are not interchangeable.

Key concept

Raw content in setContent gets zero free layout. The composition root behaves like a single-child container that places its child at the top-start origin (0,0) (top-left in LTR, top-right in RTL), applies no window insets (so content can sit under the status bar), and lets each leaf measure to wrap-content. Every desirable layout behavior is opt-in: Modifier.fillMaxWidth()/fillMaxSize() to grow, Box(..., contentAlignment = Alignment.Center) or a Column/Row (Arrangement = main axis, Alignment = cross axis) to position, textAlign to align glyphs, and Scaffold innerPadding or statusBarsPadding()/safeDrawingPadding() to respect insets. Memorize the default — top-start, no insets, wrap-content, no centering, no fill — and treat anything else as something you must explicitly request.

QUESTION 3
Which of the following statements about the Jetpack Compose Column composable are TRUE? Select ALL that apply.
At a glance Options 1, 2, and 3 are TRUE — because Column is Compose's vertical layout primitive whose main axis is vertical (controlled by verticalArrangement) and whose cross axis is horizontal (controlled by horizontalAlignment), stacking children top-to-bottom in declaration order.
✓ TRUE — correct answerOption 1

Column places its children vertically, one above the next, in the order they are declared (first child at the top).

Why: This is the defining behavior of Column. Its measure/layout policy treats the vertical axis as the main axis, so it measures each child and stacks them one beneath the previous, advancing a vertical cursor by each child's height. Because the layout walks the children in source/declaration order, the first composable you write lands at the top and the last at the bottom. The content lambda has the signature content: @Composable ColumnScope.() -> Unit, and that ordered emission is exactly what determines the visual stacking. This is the vertical counterpart to Row, which stacks horizontally.
✓ TRUE — correct answerOption 2

The verticalArrangement parameter controls how children are positioned and spaced along the vertical (main) axis.

Why: verticalArrangement: Arrangement.Vertical governs the main axis of a Column — both the block's overall position when there is leftover vertical space and the spacing between children. Values come from the Arrangement object: positional ones like Arrangement.Top (the default), Arrangement.Center, and Arrangement.Bottom; distributive ones like Arrangement.SpaceBetween, Arrangement.SpaceAround, and Arrangement.SpaceEvenly; and the gap helper Arrangement.spacedBy(8.dp). Note that distribution only has visible effect when the Column is taller than the sum of its children (for example with Modifier.fillMaxHeight()). This is the correct, idiomatic way to space a Column's children apart.
✓ TRUE — correct answerOption 3

The horizontalAlignment parameter controls how children line up along the horizontal (cross) axis, for example Alignment.CenterHorizontally.

Why: In a Column the cross axis is horizontal, and horizontalAlignment: Alignment.Horizontal decides where each child sits left-to-right within the Column's width. The legal values are Alignment.Start (the default), Alignment.CenterHorizontally, and Alignment.End — these are Alignment.Horizontal instances, distinct from the 2D Alignment values (such as Alignment.Center) used by Box. Alignment is a single value applied to every child, whereas spacing/distribution belongs to verticalArrangement; do not confuse the two. A child can also override the Column-wide setting per-item with Modifier.align(Alignment.End) inside ColumnScope.
✗ FALSEOption 4

Column places its children horizontally, side by side.

Why: This describes Row, not Column. A Column never lays children side by side — its main axis is vertical, so children always stack top-to-bottom. Placing items horizontally is a different layout primitive entirely; swapping the composable name changes the whole measure/layout policy, the arrangement type, and the alignment type. The mistake still compiles, but produces the wrong UI, so it is a semantic/layout error, not a build error.
Fix: Use Row when you want side-by-side children: Row(horizontalArrangement = Arrangement.spacedBy(8.dp)) { Text("A"); Text("B") }. Keep Column only when you genuinely want vertical stacking.
✗ FALSEOption 5

A Column scrolls vertically by default, with no extra modifier needed.

Why: Column is not scrollable by default. It measures and places all children eagerly and installs no scroll gesture, so content taller than the available height is simply cut off — there is no scrollbar and no automatic clip indicator (a plain Column does not even clip by default; overflowing children draw out of bounds until some ancestor, such as the window or a Scaffold, bounds them). To make a Column scroll you must opt in with Modifier.verticalScroll(rememberScrollState()), which still composes and measures every child up front. For long or unbounded lists prefer LazyColumn, which only composes the visible items and is the performant choice. Assuming automatic scrolling is a very common beginner bug that manifests as cut-off content.
Fix: Add the scroll modifier explicitly: Column(modifier = Modifier.verticalScroll(rememberScrollState())) { /* many items */ }. For large data sets use LazyColumn { items(list) { item -> Text(item) } } instead.
✗ FALSEOption 6

You space the children of a Column apart by setting a horizontalArrangement parameter.

Why: Column has no horizontalArrangement parameter — that parameter belongs to Row, where horizontal is the main axis. Spacing a Column's children apart is a main-axis (vertical) concern, so it is controlled by verticalArrangement. Passing horizontalArrangement = ... to a Column is a compile error: the Kotlin compiler reports Cannot find a parameter with this name: horizontalArrangement. The horizontal axis of a Column is its cross axis and is governed by horizontalAlignment (an alignment), not by an arrangement.
Fix: Use verticalArrangement for spacing: Column(verticalArrangement = Arrangement.spacedBy(12.dp)) { Text("A"); Text("B") }. Use horizontalAlignment only to position children left/center/right on the cross axis.

⚠ Common traps

  • Axis/parameter swap: for a Column, spacing lives on verticalArrangement and alignment on horizontalAlignment — for a Row it is the reverse. Writing horizontalArrangement on a Column is a compile error (Cannot find a parameter with this name: horizontalArrangement).
  • Assuming Columns scroll: a plain Column has no scroll gesture, so overflow is cut off (and not even clipped by the Column itself by default). You must add Modifier.verticalScroll(rememberScrollState()), or use LazyColumn for long/dynamic lists (it composes only visible items).
  • Arrangement vs Alignment confusion: verticalArrangement takes Arrangement.Vertical values (Top, SpaceBetween, spacedBy), while horizontalAlignment takes Alignment.Horizontal values (Start, CenterHorizontally, End) — they are not interchangeable types.
  • Distribution does nothing in a wrap-content Column: Arrangement.SpaceBetween/SpaceEvenly only show an effect when the Column is taller than its children (e.g. with Modifier.fillMaxHeight()); otherwise there is no extra space to distribute.

Key concept

A Column is Compose's vertical layout primitive: main axis = vertical, cross axis = horizontal. Children stack top-to-bottom in declaration order. Control the main (vertical) axis — both position and inter-child spacing — with verticalArrangement (an Arrangement.Vertical: Top, Center, Bottom, SpaceBetween, spacedBy(dp)). Control the cross (horizontal) axis with horizontalAlignment (an Alignment.Horizontal: Start, CenterHorizontally, End), overridable per child via Modifier.align(...). A Column has no horizontalArrangement (that belongs to Row) and is not scrollable by default — add Modifier.verticalScroll(rememberScrollState()) or reach for LazyColumn. Mnemonic: Column = down the column = vertical main axis; Row = across the row = horizontal main axis; arrangement = main axis, alignment = cross axis.

QUESTION 4
Which of the following statements about the Jetpack Compose Row composable are TRUE? Select ALL that apply.
At a glance Options 1, 2, and 3 are TRUE because Row is the horizontal layout primitive whose main axis is horizontal (controlled by horizontalArrangement) and whose cross axis is vertical (controlled by verticalAlignment); options 4, 5, and 6 are FALSE because they describe Column, assume scrolling that Row does not provide, and invent a horizontalAlignment parameter that does not exist on Row.
✓ TRUE — correct answerOption 1

Row places its children horizontally, side by side, in the order they are declared (first child at the leading edge).

Why: Row is the canonical horizontal layout container in Compose Foundation, with signature Row(modifier, horizontalArrangement, verticalAlignment, content: @Composable RowScope.() -> Unit). It measures and places each direct child one after another along the horizontal main axis, so children sit side by side rather than overlapping. Placement follows declaration order: the first composable in the lambda is positioned at the leading (start) edge, the next to its right, and so on. In a left-to-right (LTR) locale the leading edge is the left; in an RTL locale Compose automatically mirrors so the leading edge becomes the right. This declaration-order, leading-to-trailing packing is the defining behavior of Row.
✓ TRUE — correct answerOption 2

The horizontalArrangement parameter controls how children are positioned and spaced along the horizontal (main) axis, for example Arrangement.SpaceBetween.

Why: Because the main axis of a Row is horizontal, the parameter that distributes children along that axis is horizontalArrangement, and it takes an Arrangement.Horizontal. Its default is Arrangement.Start, which packs children at the leading edge with no extra gaps. You can change it to Arrangement.End, Arrangement.Center, or the spacing variants Arrangement.SpaceBetween (first/last flush to the edges, equal gaps between), Arrangement.SpaceAround (equal space around each child, so half-size gaps at the two ends), and Arrangement.SpaceEvenly (equal gaps everywhere, including the ends). You can also request a fixed gap with Arrangement.spacedBy(8.dp). Note the spreading arrangements (SpaceBetween, SpaceAround, SpaceEvenly, End, Center) only have a visible effect when the Row is wider than the sum of its children (e.g. it has Modifier.fillMaxWidth()).
✓ TRUE — correct answerOption 3

The verticalAlignment parameter controls how children line up along the vertical (cross) axis, for example Alignment.CenterVertically.

Why: The cross axis of a Row is vertical, so alignment perpendicular to the flow direction is set with verticalAlignment, which takes an Alignment.Vertical. Its default is Alignment.Top, meaning children of differing heights are pinned to the top edge of the row. The other built-in options are Alignment.CenterVertically (children centered within the row's height — extremely common when mixing an icon with text so they look balanced) and Alignment.Bottom. This row-wide default can be overridden per child inside RowScope with Modifier.align(Alignment.CenterVertically). The alignment only produces a visible difference when the children have different heights or the row is taller than its tallest child.
✗ FALSEOption 4

Row stacks its children vertically, top to bottom.

Why: This describes Column, not Row. Column is the vertical layout primitive whose main axis is vertical, stacking children top to bottom and using verticalArrangement + horizontalAlignment. Row does the opposite: it lays children out along a horizontal main axis (leading to trailing). Confusing the two is the single most common beginner mistake because their signatures are mirror images of each other. The failure here is purely conceptual (no compiler error) — using a Row expecting a vertical stack just renders everything side by side and may overflow off the trailing (right in LTR) edge.
Fix: To stack children vertically, use Column instead of Row:
Column { Text("Top"); Text("Bottom") }
✗ FALSEOption 5

A Row scrolls horizontally by default, with no extra modifier needed.

Why: Row is a plain, eager layout that measures and lays out all of its children at once and does not scroll by default. If the children are wider than the available space, the content simply clips/overflows rather than becoming scrollable. To make a finite set of children scroll horizontally you must add Modifier.horizontalScroll(rememberScrollState()) to the Row. For a long or unbounded list you should instead use LazyRow, which only composes and measures the items currently visible (lazy/virtualized), making it far more performant than a scrollable Row.
Fix: Add the scroll modifier (or switch to LazyRow for long lists):
Row(Modifier.horizontalScroll(rememberScrollState())) { /* children */ }
✗ FALSEOption 6

You align the children of a Row on the cross axis by setting a horizontalAlignment parameter.

Why: Row has no horizontalAlignment parameter — that parameter belongs to Column (whose cross axis is horizontal). For a Row the cross axis is vertical, so cross-axis alignment is done with verticalAlignment. Passing horizontalAlignment to Row is a compile error; the Kotlin compiler reports Cannot find a parameter with this name: horizontalAlignment. The horizontal direction in a Row is the main axis, which is governed by horizontalArrangement, not by an alignment parameter.
Fix: Use verticalAlignment for cross-axis alignment in a Row:
Row(verticalAlignment = Alignment.CenterVertically) { /* children */ }

⚠ Common traps

  • Swapping the parameter names. Row uses horizontalArrangement + verticalAlignment; Column uses verticalArrangement + horizontalAlignment. Writing horizontalAlignment on a Row is a compile error (Cannot find a parameter with this name: horizontalAlignment).
  • Expecting SpaceBetween/Center to do anything in a wrap-content Row. Spreading arrangements only show an effect when the Row is wider than its children — add Modifier.fillMaxWidth() or a fixed width, otherwise the Row shrinks to its content and there is no extra space to distribute.
  • Assuming a Row scrolls. A bare Row clips/overflows when content is too wide. Add Modifier.horizontalScroll(rememberScrollState()) for a few items, or use LazyRow for long/dynamic lists so items are composed lazily.
  • Forgetting alignment needs a height difference to be visible. verticalAlignment only changes the picture when children have different heights (or the Row is taller than its tallest child); with equal-height children Top/Center/Bottom look identical.

Key concept

Row = horizontal main axis, vertical cross axis. A Row lays its children out leading-to-trailing (left-to-right in LTR) in declaration order — the first child at the leading/start edge, mirrored automatically in RTL. Because the main axis runs horizontally, spacing along it is set with horizontalArrangement (an Arrangement.Horizontal like Start, SpaceBetween, or spacedBy), and because the cross axis runs vertically, alignment perpendicular to the flow is set with verticalAlignment (an Alignment.Vertical like Top, CenterVertically, Bottom). Column is the exact mirror image (vertical main axis -> verticalArrangement + horizontalAlignment). Memorize the rule: the parameter prefix matches the axis it acts on — arrangement always names the main axis, alignment always names the cross axis. Finally, a Row is eager and non-scrolling by default — reach for Modifier.horizontalScroll(...) or LazyRow when content exceeds the available width.

QUESTION 5
Which of the following statements about the Jetpack Compose Material 3 Card composable are TRUE? Select ALL that apply.
At a glance Options 1, 2, and 3 are TRUE. The Material 3 Card is a Surface-based container that applies a default rounded shape and themed colors, exposes a ColumnScope content lambda (so children stack vertically like a Column), and ships a clickable overload taking onClick and enabled.
✓ TRUE — correct answerOption 1

Card is a Material 3 container (built on top of Surface) used to group related content, and it applies a themed background color and rounded-corner shape by default.

Why: Internally androidx.compose.material3.Card delegates to a Surface, which is the Material 3 primitive responsible for drawing background color, shape clipping, elevation/shadow, and tonal overlays. The defaults are supplied by the CardDefaults object: shape = CardDefaults.shape (a medium rounded corner from the theme's Shapes), colors = CardDefaults.cardColors() (container color surfaceContainerLow and a matching contentColor from the ColorScheme), and elevation = CardDefaults.cardElevation(). Because these defaults are pre-filled parameters, calling Card { ... } with no styling still renders a themed, rounded, slightly raised panel. This is exactly its job: a self-contained visual grouping of related content.
✓ TRUE — correct answerOption 2

The content you pass to a Card is a ColumnScope lambda, so by default its children are stacked vertically, the same way a Column lays them out.

Why: The trailing lambda parameter of Card has the receiver type @Composable ColumnScope.() -> Unit. A receiver of ColumnScope means the content is composed inside a real Column, so children are placed one below another along the vertical main axis in source order. It also means ColumnScope-only modifiers such as Modifier.align(Alignment.CenterHorizontally) and Modifier.weight(1f) are in scope and compile. So a Card behaves like a styled, elevated Column for layout purposes.
✓ TRUE — correct answerOption 3

A Card can handle taps directly by using the Card overload that takes an onClick parameter (and an enabled flag).

Why: Material 3 ships two Card overloads. The display overload has no click handling, but the second overload's signature begins fun Card(onClick: () -> Unit, modifier: Modifier = Modifier, enabled: Boolean = true, ...). Passing onClick selects this overload, which internally routes through a clickable Surface so the entire card becomes one tappable target with a ripple indication and the correct accessibility/semantics (role, click action). The enabled flag, defaulting to true, gates both the click callback and the visual/semantic enabled state. No external Button or manual Modifier.clickable is required.
✗ FALSEOption 4

By default a Card lays out its children horizontally, the same way a Row does.

Why: This inverts the axis. The Card content lambda is a ColumnScope receiver, not a RowScope, so children stack vertically along the main vertical axis, exactly like a Column — never side by side. Assuming horizontal layout also leads to a compile error if you reach for a RowScope modifier: writing Modifier.align(Alignment.CenterVertically) inside a Card fails because CenterVertically is an Alignment.Vertical value, while ColumnScope.align only accepts an Alignment.Horizontal. To actually get a horizontal arrangement you must add your own Row.
Fix: Wrap the children in an explicit Row if you want horizontal layout:
Card { Row { Text("A"); Text("B") } } — only inside that Row can you use RowScope modifiers like Modifier.weight(1f) and Modifier.align(Alignment.CenterVertically).
✗ FALSEOption 5

A Card has no default shape or container color, so nothing is drawn until you set shape and colors explicitly.

Why: Every styling parameter of Card has a default, so omitting them does not produce a blank/invisible composable. shape defaults to CardDefaults.shape, colors to CardDefaults.cardColors() (a themed surfaceContainerLow container color), and elevation to CardDefaults.cardElevation(). Because the underlying Surface always paints its container color and clips to the shape, a bare Card { Text("Hi") } renders a visible rounded, tinted, slightly raised panel. You only override shape/colors to customize the look, not to make the card appear at all.
Fix: Customizing is optional, but if you want a specific look you override the defaults:
Card(shape = RoundedCornerShape(16.dp), colors = CardDefaults.cardColors(containerColor = MaterialTheme.colorScheme.primaryContainer)) { /* content */ }. With no arguments the card is still fully visible.
✗ FALSEOption 6

A Card cannot respond to clicks on its own; you must wrap it in a Button or attach a separate clickable to make it tappable.

Why: This denies the existence of the clickable Card overload. Supplying onClick selects fun Card(onClick: () -> Unit, ...), which makes the whole card tappable with a built-in ripple and proper click semantics — no wrapping Button needed. Wrapping a card in a Button is actually wrong practice: it nests a container inside an interactive component and produces conflicting padding, shape, and semantics. While you could hand-add Modifier.clickable, the idiomatic and accessibility-correct approach is the onClick overload.
Fix: Use the click overload directly:
Card(onClick = { /* handle tap */ }, enabled = true) { Text("Tap me") } — this gives ripple feedback and correct semantics without any external Button or manual clickable.

⚠ Common traps

  • Confusing the axis. Because cards often look like horizontal rows in designs, students assume the content lambda is a RowScope. It is a ColumnScope — children stack vertically by default, and horizontal layout requires an explicit nested Row.
  • Thinking a bare Card is invisible. shape, colors, and elevation all default through CardDefaults, so Card { Text("Hi") } already shows a rounded, tinted, raised panel. You override those parameters to restyle, not to make the card appear.
  • Wrapping a Card in a Button to make it clickable. Use the Card(onClick = ...) overload instead. Nesting a card in a Button creates conflicting shape, padding, and accessibility semantics; the dedicated overload gives the correct ripple and click role.
  • Using the wrong scope's alignment. Modifier.align(Alignment.CenterVertically) belongs to RowScope (it takes an Alignment.Vertical); inside a Card's ColumnScope you instead use Modifier.align(Alignment.CenterHorizontally) (an Alignment.Horizontal). Modifier.weight exists in both scopes but distributes the scope's main axis — vertical in a Column/Card, horizontal in a Row. Passing the wrong alignment type is a compile error.

Key concept

A Material 3 Card is a themed, elevated Column built on Surface. Three things follow from that and are worth memorizing:
1. Styling is automatic. shape, colors, and elevation default through CardDefaults (rounded corners, surfaceContainerLow background, slight shadow), so a card is always visible without explicit styling.
2. Layout is vertical. The content lambda is @Composable ColumnScope.() -> Unit, so children stack top-to-bottom along the vertical main axis and you get ColumnScope modifiers; use a nested Row for horizontal arrangement.
3. Clicks are first-class. The overload Card(onClick: () -> Unit, enabled: Boolean = true, ...) makes the whole card tappable with a built-in ripple and proper semantics — never wrap a card in a Button to make it clickable.

GLOSSARY
Key terms used throughout this answer key.
composable
A function annotated with @Composable that describes a piece of UI; it can call other composables and is re-invoked ("recomposed") when the state it reads changes.
setContent
The ComponentActivity extension that attaches a Compose UI tree to the activity's window. Whatever composable you place inside its { } lambda becomes the root and is laid out starting at the top-start of the available space, filling it; nothing centers or stacks it for you unless you add a layout like Column or a Scaffold.
Modifier
An ordered, chainable object passed to a composable to change its size, padding, background, alignment, click behavior, and more; order matters because each modifier wraps the result of the previous one.
ColumnScope / RowScope
The receiver scopes inside a Column or Row lambda. They unlock layout-only modifiers such as Modifier.weight() and Modifier.align(), which is why a Card's content (a ColumnScope) behaves like a vertical stack.
Arrangement
Controls spacing of children along the main axis (vertical for Column, horizontal for Row), e.g. Arrangement.spacedBy(8.dp), SpaceBetween, or Center.
Alignment
Controls positioning of children across the cross axis, e.g. Alignment.CenterHorizontally in a Column or Alignment.CenterVertically in a Row.
main axis vs cross axis
The main axis is the direction a container stacks its children (vertical in Column, horizontal in Row); the cross axis is perpendicular to it. Arrangement acts on the main axis, alignment on the cross axis.
wrap-content
The default sizing behavior where a composable is only as big as its content needs. Opposite of stretching with Modifier.fillMaxWidth() / fillMaxSize().
fillMaxWidth
A modifier (Modifier.fillMaxWidth()) that makes a composable take all available horizontal space from its parent; siblings are fillMaxHeight() and fillMaxSize().
sp / TextUnit
sp ("scale-independent pixels") is the TextUnit recommended for font sizes because it respects the user's system font-scale accessibility setting; written as 16.sp.
dp
"Density-independent pixels," the unit for sizes, padding, and spacing so layouts look consistent across screen densities; written as 16.dp. Use dp for dimensions and sp for text.
Scaffold
A Material composable that provides the standard screen skeleton (top bar, bottom bar, FAB, snackbars) and hands its body a PaddingValues you should apply so content is not hidden behind those bars or the system window insets.
Surface
A Material container that paints a background color, applies elevation/tonal tint, clips to a shape, and sets the default content color; Card is essentially a styled Surface.
TextStyle / typography
TextStyle bundles font attributes (size, weight, line height, color) into one object; MaterialTheme.typography exposes ready-made styles like titleLarge and bodyMedium for consistent text.
elevation
The simulated z-height of a surface, expressed in dp; in Material 3 it produces both a drop shadow and a subtle tonal color tint. CardDefaults.cardElevation() supplies a Card's default values.
ripple
The expanding circular touch feedback shown when an interactive element (such as a clickable Card or Button) is pressed; it is added automatically by clickable Material components.
