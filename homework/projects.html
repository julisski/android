Which layout container should I use?

Every Compose screen is built by nesting a handful of containers. This is the first decision you make on any screen — pick the one whose job matches what you're arranging.

Container Reach for it when… The props you'll set
Column Stacking a few children top → bottom — a form, a labelled value, a card's body. verticalArrangement · horizontalAlignment
Row Placing children side by side — icon + label, a toolbar. The mirror of a Column. horizontalArrangement · verticalAlignment
Box Overlapping children — a badge on an avatar, text over an image, a centered spinner. contentAlignment · Modifier.align
LazyColumn / LazyRow A long or unbounded scrolling list — it only builds the rows on screen. Don't put a big list in a scrolling Column. items(list) { … }
Scaffold The whole-screen skeleton — top bar, bottom bar, FAB and snackbar in their standard Material slots. topBar · bottomBar · floatingActionButton

Rule of thumb: Scaffold is the outer screen frame; inside it you arrange content with Column / Row, layer things with Box, and scroll long lists with LazyColumn. They nest freely — a Scaffold's content is usually a Column, whose rows may be a LazyColumn, and so on.

See it live: the Compose Playground (root-layout dropdown + the "Choosing a container" reference cards) and the Layouts tab in the ComposeParts app both let you flip between these and watch the result change.

The shared tech stack

Jetpack Compose + Material 3

Declarative UI — you describe what the screen should look like for the current state, and Compose redraws when that state changes.

Navigation 3

NavDisplay, rememberNavBackStack, entryProvider and @Serializable NavKeys. The back stack is just a list of keys; the top key is the screen on screen.

Kotlin 2.2 · single Activity

One Activity hosts everything via setContent { }. Screens are composables that get swapped — no more starting a new Activity per screen.

1 · Compose foundation

SingleActivity

The bare-bones starting point: one Activity hosting a single Compose 'Hello Android!' screen.

Project 1 of 25 ▫️ no data domain ⏱ 15-25 min Navigation track
The one big idea

A modern Android app is one Activity (MainActivity) that calls setContent to render a tree of Composable functions; the Activity is the bridge from the OS into the world of Compose.

What it does

When you launch the app you see a single screen with the text 'Hello Android!' near the top-left. There are no buttons, lists, or navigation; nothing is tappable. The whole screen is drawn edge-to-edge behind the system bars, with a Scaffold leaving padding so the text doesn't slide under the status bar. This is the absolute baseline Android Studio scaffold, trimmed down to show the minimum wiring needed to put Compose UI on screen.

Screens you'll see

  • MainActivity — The single Activity and launcher entry point; its onCreate calls setContent to host the entire Compose UI.
  • Greeting — A small @Composable that renders a Text reading 'Hello <name>!' using a Kotlin string template.

Key concepts

Single-Activity architecture

Instead of one Activity per screen, the whole app lives inside one Activity (MainActivity). All UI is built from Composable functions inside it, which is the modern Compose default.

ComponentActivity

MainActivity extends ComponentActivity, the lightweight Activity base class designed to host Jetpack Compose via setContent.

setContent

Called inside onCreate, setContent { ... } hands Android the Composable tree to draw. Everything inside that lambda becomes the screen.

@Composable function

A function annotated @Composable (like Greeting) describes a piece of UI. You call it from other composables; you do not 'return' a view, you emit UI.

Scaffold and innerPadding

Scaffold provides standard Material screen structure and hands you innerPadding (insets for system bars) which you apply with Modifier.padding(innerPadding) so content isn't hidden under the status bar.

enableEdgeToEdge

enableEdgeToEdge() lets the app draw behind the status and navigation bars for a modern full-bleed look; Scaffold's padding compensates so text stays visible.

Modifier

A Modifier is a chainable instruction list for layout and appearance. Here Modifier.fillMaxSize() makes Scaffold fill the screen and Modifier.padding(innerPadding) insets the Greeting text.

AndroidManifest LAUNCHER intent-filter

The intent-filter with action MAIN and category LAUNCHER marks MainActivity as the screen that opens when you tap the app icon. exported='true' is required for the launcher activity.

2 · Navigation 3 · two screens

Intent

Replace Activity-and-Intent navigation with one Activity that swaps Compose screens via a Navigation 3 back stack.

Project 2 of 25 ▫️ no data domain ⏱ 15-25 min Navigation track
The one big idea

A screen is just a 'key' on a list called the back stack: push a key to navigate forward, pop the top key to go back, and the arguments a screen needs ride along inside its key instead of inside an Intent.

What it does

When the app opens you see 'Hello Android!' and a button labeled 'Open Detail'. Tapping that button moves you to a second screen that displays the text 'Hello from Home!' along with a 'Back' button. Tapping 'Back' (or the system back gesture) returns you to the Home screen. Everything happens inside a single Activity; no new Activity is ever launched.

Screens you'll see

  • HomeScreen — Shows a greeting and an 'Open Detail' button whose tap pushes a DetailKey onto the back stack.
  • DetailScreen — Displays the message it received from DetailKey plus a 'Back' button that pops the back stack.
  • AppNavigation — Owns the back stack and maps each key type to the composable screen that should be drawn.

Key concepts

NavKey

A marker interface every navigation destination implements. Both HomeKey and DetailKey are declared 'as NavKey' so the back stack can hold them as the items that identify each screen.

Back stack of keys

'rememberNavBackStack(HomeKey)' creates a remembered list of keys that starts with HomeKey. The top key is the visible screen; 'backStack.add(...)' goes forward and 'backStack.removeLastOrNull()' goes back.

Arguments live in the key

DetailKey is a 'data class DetailKey(val message: String)', so the data a screen needs travels inside its key. The entry reads it with 'key.message' instead of the old 'intent.getStringExtra(...)'.

NavDisplay

The composable that watches the back stack and renders whatever screen matches the top key. It also wires the system back button through its 'onBack' lambda.

entryProvider DSL

Inside NavDisplay, 'entryProvider { entry<HomeKey> { ... }; entry<DetailKey> { key -> ... } }' is the lookup table that says which composable to show for each key type.

@Serializable keys

Each key is annotated '@Serializable' so Nav3 can save and restore the back stack across rotation and process death. This needs the kotlin.serialization plugin and kotlinx-serialization-json in build.gradle.kts.

data object vs data class

HomeKey is a 'data object' because Home takes no arguments, so one shared instance is enough. DetailKey is a 'data class' because each visit can carry a different 'message'.

Single Activity

MainActivity is the only Activity (the manifest declares just one). Navigation swaps composables inside it rather than starting new Activities, so there are no Intents and no extra <activity> entries.

What's new here

SingleActivity established one Activity hosting a Compose UI in a Scaffold. Intent adds real navigation on top of it: '@Serializable' NavKey types (HomeKey, DetailKey), a 'rememberNavBackStack' back stack, and a 'NavDisplay' + 'entryProvider' that swap between a Home and a Detail screen, with the screen's argument carried inside DetailKey instead of via an Intent extra.

  1. 1Add a third key, e.g. 'data class ProfileKey(val name: String) : NavKey', register an 'entry<ProfileKey>' in entryProvider, and push it from a new button on DetailScreen to see a three-deep back stack.
  2. 2Change the pushed value in 'backStack.add(DetailKey('Hello from Home!'))' to your own text and confirm DetailScreen shows the new message after rebuilding.
  3. 3Give DetailKey a second field (like 'val count: Int'), update the entry to display it, and watch the compiler force you to pass it everywhere the key is created.
  4. 4Watch Logcat for tag 'LT' (the 'In function AppNavigation' message), rotate the device, and observe whether the back stack survives the configuration change.
  5. 5Temporarily comment out the '@Serializable' annotation on DetailKey and rebuild to see why Nav3 needs it for saving and restoring keys.
Good to know
  • This is a deliberately minimal demo: there is no ViewModel, no StateFlow, and no real domain data. The 'message' is just a hard-coded string, so it teaches navigation mechanics, not data layering.
  • Despite the project name, there are no android.content.Intent objects in the navigation flow. The only intent in the codebase is the MAIN/LAUNCHER intent-filter in AndroidManifest.xml that makes the app launchable; the whole point is that screen-to-screen navigation no longer uses Intents.
  • The Log tag is inconsistent: there is a 'private const val tag = 'LT'' that is never used, while the actual Log.d call passes the literal string 'LT'. Both happen to be 'LT', so logs still work, but the named constant is dead code.
  • 'backStack.removeLastOrNull()' is intentionally null-safe: it returns null (doing nothing) only when the stack is already empty, so the back action can never crash. Note it does NOT block popping the last (Home) key — on a one-item stack it removes HomeKey, emptying the stack, which is why pressing Back on Home exits the app.

Run & check it

Build and install with './gradlew installDebug' (or press Run in Android Studio), then launch 'Intent' from the app drawer. You should see 'Hello Android!' and an 'Open Detail' button; tap it to reach the screen showing 'Hello from Home!' and 'Back', then tap Back to return Home. Confirm the system back gesture also returns you Home and that pressing back on Home exits the app. Filter Logcat by tag 'LT' to see the 'In function AppNavigation' message each time AppNavigation recomposes. This project defines no <data> deep-link intent-filter, so there is no adb deep-link command to run here.

NavDetailList

Tap a planet in a scrolling list to open its detail screen, then go back - all in-app with Navigation 3.

Project 3 of 25 🪐 planets domain ⏱ 15-25 min Navigation track
The one big idea

Navigation is just a back stack of typed keys you push and pop: tapping pushes a DetailKey carrying an id, going back pops it - and the same NavDisplay redraws whatever key is on top.

What it does

The app opens on a scrolling list of five planets, each row showing a title (like 'Mercury') and a one-line blurb. Tapping any row slides in a detail screen that shows that planet's title in a larger style, its full blurb, and a 'Back to list' button. Pressing the button (or the system back gesture) returns you to the list. Everything happens inside one Activity - no new screen is ever launched as a separate app window.

Screens you'll see

  • ListScreen — A LazyColumn of planet rows; tapping a row calls onOpen with that planet's id to navigate forward.
  • DetailScreen — Shows the single planet resolved from the id in the key, plus a 'Back to list' button that pops the stack.
  • AppNavigation — Owns the back stack and maps each key (ListKey, DetailKey) to the screen that should be shown.

Key concepts

NavKey

A small typed object that identifies a screen AND carries its arguments. Here 'ListKey' is a NavKey with no data and 'DetailKey(itemId)' is a NavKey that carries which planet's id to show.

Back stack

A list of keys where the top one is the visible screen. 'rememberNavBackStack(ListKey)' creates it starting on the list; 'backStack.add(...)' pushes forward and 'backStack.removeLastOrNull()' pops back.

NavDisplay

The single Composable that watches the back stack and draws whatever key is on top. It calls 'onBack' when the user goes back and uses 'entryProvider' to know how to render each key.

entryProvider / entry

A registry that says 'for this key type, build this screen'. 'entry<ListKey>' builds the list and 'entry<DetailKey> { key -> ... }' builds the detail, where 'key' gives you the itemId argument.

Passing an argument via the key

Instead of passing whole objects between screens, you put a small id inside the key: 'DetailKey(id)'. The detail screen reads 'key.itemId' and looks the full Item back up with 'itemById(...)'.

LazyColumn + items

A scrolling column that only builds rows that are on screen. 'items(items) { item -> ... }' loops the planet list and draws one Column per planet, with 'HorizontalDivider()' between them.

clickable for navigation

Modifier.clickable on a row turns the whole row into a tap target. Here 'clickable { onOpen(item.id) }' is what triggers the push to the detail screen.

@Serializable keys

Both keys are marked '@Serializable' so Navigation 3 can save and restore the back stack across process death or config changes. That is why kotlinx.serialization is a dependency.

What's new here

Intent moved between screens by launching a separate Activity and stuffing primitive extras into a Bundle. NavDetailList replaces that with Navigation 3: one single Activity, no Intents, and navigation done by pushing and popping typed @Serializable NavKeys on a back stack that NavDisplay renders. The argument (itemId) now rides inside a strongly-typed DetailKey instead of loose string-keyed Intent extras, and the whole app is Compose-first list-to-detail flow.

  2. 2Give 'DetailKey' a second argument such as 'val fromList: Boolean = true', pass it when you call 'backStack.add(DetailKey(id, true))', and show it on the detail screen to prove keys can carry more than one piece of data.
  3. 3In the DetailScreen detail 'entry', replace 'itemById(key.itemId)' with a hardcoded item to see that the key is just data - the screen draws whatever Item you hand it.

  5. 5Remove the '@Serializable' annotations and rebuild to see why Navigation 3 needs serializable keys for state restoration.
Good to know
  • The detail screen is NOT given the whole Item - it only gets 'key.itemId' and re-looks-it-up with 'itemById(...)'. Passing only a small id in the key is the intended pattern; do not try to cram big objects into a NavKey.
  • This is Navigation 3 (the 'androidx.navigation3' packages with NavDisplay/entryProvider/rememberNavBackStack), which is newer and different from the older Jetpack 'NavController'/'NavHost' API - the names will not match older tutorials.
  • 'sampleItems' is hardcoded in memory, so 'itemById' assumes the id always exists and uses '.first { ... }', which would crash if you ever pushed a DetailKey with an unknown id. There is no database or network here on purpose.
  • Both 'onBack' on NavDisplay and the 'Back to list' button call 'removeLastOrNull()' - the system back gesture and the on-screen button do the same pop, which is why both behave identically.

Press 'Back to list' and also try the system back gesture - both should return you to the list. There are no deep links in this project, so no adb deep-link command applies; verify navigation purely by tapping rows and going back. NavListDetail

A one-file Navigation 3 sample: tap a planet in a list to push a detail screen, then pop back.

Project 4 of 25 🪐 planets domain ⏱ 15-25 min Navigation track
The one big idea

In Navigation 3 a screen is identified by a serializable 'key' on a back stack; pushing a key navigates forward, popping it navigates back, and NavDisplay just renders whatever key is on top.

What it does

On launch you see a scrolling list of five planets (Mercury through Jupiter), each row showing a bold title and a one-line blurb, separated by thin divider lines. Tapping any row navigates forward to a detail screen showing that planet's title as a large headline, its blurb as body text, and a 'Back to list' button. Pressing that button (or the system back gesture) returns you to the list. The data is a hardcoded in-memory list, so nothing is saved or fetched from a network.

Screens you'll see

  • ListScreen — Shows all five planets in a LazyColumn; tapping a row reports that item's id via the onOpen callback.
  • DetailScreen — Shows one resolved planet's title, blurb, and a 'Back to list' button that pops the stack.
  • AppNavigation — Owns the back stack and maps each key type (ListKey, DetailKey) to the screen it renders.

Key concepts

NavKey

A marker interface every navigation destination implements. Here 'ListKey' and 'DetailKey' both implement NavKey so Nav3 can treat them as entries on the back stack.

Back stack

An ordered list of keys, bottom to top. 'rememberNavBackStack(ListKey)' seeds it with the list; 'backStack.add(...)' pushes forward and 'backStack.removeLastOrNull()' pops back.

Argument-carrying key

'DetailKey(val itemId: Int)' is a data class, so each instance carries which planet to show. Only the tiny id travels in the key, not the whole Item object.

data object vs data class key

'ListKey' is a 'data object' (one single list screen, so a shared singleton) while 'DetailKey' is a 'data class' (one instance per planet, distinguished by itemId).

@Serializable keys

Nav3 keys are annotated '@Serializable' (via the kotlinx-serialization plugin) so the back stack can be saved and restored across process death.

entryProvider DSL

A builder where each 'entry<KeyType> { ... }' block defines the screen for one key type. Nav3 picks the block whose type matches the current top key; the DetailKey block receives the key instance so it can read 'key.itemId'.

Stateless screen composables

'ListScreen' and 'DetailScreen' know nothing about navigation. They render given data and report events through callbacks ('onOpen', 'onBack'), which keeps them reusable and previewable.

id-then-resolve pattern

The detail key carries only the id; 'itemById(key.itemId)' looks the full Item up from 'sampleItems' inside AppNavigation. This keeps the navigation key small and serializable.

NavThreeScreen

A three-level drill-down (Categories -> Items -> Detail) built on Navigation 3, where the back stack can grow three keys deep.

Project 5 of 25 🪐 planets domain ⏱ 15-25 min Navigation track
The one big idea

Navigation is just a list (the back stack) of small @Serializable keys; adding a key pushes a screen forward and removing the last key pops back, and the stack here can hold three keys at once.

What it does

You first see a scrolling list of two planet categories: 'Rocky Planets' and 'Gas Giants'. Tapping a category navigates forward to a second screen listing the planets in that group (Mercury, Venus, Earth... or Jupiter, Saturn), with a header and a 'Back to categories' button. Tapping a planet drills down a third level to a detail screen showing that one planet's title, blurb, and a 'Back to list' button. Both the on-screen buttons and the system Back button pop one level at a time, walking you back up the chain.

Screens you'll see

  • CategoriesScreen — Level 1 - a LazyColumn of the two planet categories; tapping a row reports its id via onOpen.
  • ItemsScreen — Level 2 - a fixed header naming the chosen category plus a scrolling list of that category's planets, with a 'Back to categories' button.
  • DetailScreen — Level 3 - shows a single planet's title and blurb resolved from the key's itemId, plus a 'Back to list' button.
  • AppNavigation — Owns the rememberNavBackStack and wires every key type to its screen through NavDisplay + entryProvider.
  • MainActivity — The single Activity; calls setContent and hosts the whole Compose + Nav3 tree inside a Scaffold.

Key concepts

Back stack as a list

The back stack created by rememberNavBackStack(CategoriesKey) is literally a mutable list of keys. backStack.add(...) pushes a new screen, backStack.removeLastOrNull() pops one. Here it can hold up to three keys: [CategoriesKey, ItemsKey, DetailKey].

NavKey

A marker interface every navigation destination implements. CategoriesKey, ItemsKey, and DetailKey all implement NavKey so Nav3 can track them on the stack.

Keys carry their arguments

A key both names a screen AND holds its data. ItemsKey(val categoryId: Int) carries which category to list, and DetailKey(val itemId: Int) carries which planet to show. CategoriesKey is a 'data object' because there is only one categories screen and it needs no arguments.

data object vs data class key

CategoriesKey is a 'data object' (a singleton) since there is only ever one categories screen. ItemsKey and DetailKey are 'data class' because each instance differs by the id it carries.

entryProvider DSL

entryProvider { entry<KeyType> { ... } } maps each key type to a Composable. Nav3 looks at the top key and runs the matching entry<> block; the entry<ItemsKey> { key -> } lambda even hands you the key instance so you can read key.categoryId.

Pass the id, not the object

Keys carry only a small Int id, and helper functions categoryById, itemById, and itemsInCategory resolve the full Category/Item objects at render time. This keeps each key tiny and @Serializable so it can survive process death.

Dumb (stateless) screens

Key concepts

Deep link

A URI fired from OUTSIDE the app (browser, notification, adb) that jumps straight to a specific screen. Here navdemo://planet/3 opens Earth's detail screen directly, no tapping required.

intent-filter (VIEW)

The AndroidManifest block declaring scheme navdemo and host planet plus action VIEW and categories DEFAULT + BROWSABLE. This is what tells Android to route navdemo://planet/... URIs to MainActivity.

Synthetic back stack

buildInitialBackStack returns a hand-built list [CategoriesKey, ItemsKey(category), DetailKey(itemId)] for a valid deep link, so Back has somewhere to go instead of exiting the app.

Defensive URI parsing

NavKey + entryProvider

Each screen is a @Serializable NavKey (CategoriesKey, ItemsKey, DetailKey); entryProvider's entry<KeyType> blocks map the top key to the right composable, and NavDisplay renders it.

id-only keys

Keys carry just an Int id (categoryId / itemId), not whole objects; screens resolve the full Category/Item via categoryById / itemById, keeping each key tiny and serializable.

Multiple back stacks

A bottom-nav app where each tab owns a separate navigation history. Here that is modeled by calling 'rememberNavBackStack' three times (planetsBackStack, searchBackStack, aboutBackStack) instead of using one global stack.

Per-tab System Back

'onBack = { activeBackStack.removeLastOrNull() }' pops only the selected tab's stack, so the hardware/gesture Back walks back through the current tab's history and never leaks into another tab.

NavigationBar / NavigationBarItem

Material 3's bottom bar. 'NavigationBar' is the container; one 'NavigationBarItem' per tab carries 'selected', an 'onClick' that just sets 'selectedTab = tab.index', an emoji 'icon', and a text 'label'.

Every destination is a key implementing 'NavKey' and marked '@Serializable' - 'data object' for single screens (CategoriesKey, SearchKey, AboutKey) and 'data class' when arguments differ (ItemsKey(categoryId), DetailKey(itemId)).

Key concepts

Nested graph / sub-flow

A self-contained multi-step flow (here onboarding) that runs as one cohesive unit and then hands off to the main app. The defining trait is that completing it pops the ENTIRE flow off the stack, not one screen at a time.

Modeling a graph as a key Set

Since Nav3's back stack is a flat list, the 'graph membership' is just 'onboardingKeys = setOf(WelcomeKey, ChooseFavoriteKey, DoneKey)'. That Set IS the definition of which destinations belong to the onboarding nested graph.

Back-stack surgery

Completing the sub-flow is done by directly editing the list: 'backStack.removeAll { it in onboardingKeys }' deletes every onboarding key, then 'backStack.add(CategoriesKey)' enters the main app. This is the Nav3 stand-in for Nav2's 'popUpTo(graph) { inclusive = true }'.

NavKey + data object vs data class

Every destination implements NavKey. Argument-free steps (WelcomeKey, CategoriesKey) are singleton 'data object' keys; steps that carry data (ItemsKey(categoryId), DetailKey(itemId)) are 'data class' keys that bundle the destination AND its arguments together.

rememberNavBackStack seeded in Graph A

'val backStack = rememberNavBackStack(WelcomeKey)' creates and remembers the stack seeded with a Welcome key, so the app deliberately STARTS inside the onboarding sub-flow rather than the main app.

entryProvider grouped by graph

The 'entryProvider { entry<KeyType> { ... } }' DSL maps each key type to a screen. Here the entry blocks are split into two commented sections, Graph A and Graph B, so the 'two graphs sharing one flat stack' idea is visible in code.

onBack as a flat pop

NavDisplay's 'onBack = { backStack.removeLastOrNull() }' just removes the top key. Inside onboarding it walks Done -> Choose -> Welcome; in the main app it walks Detail -> Items -> Categories; because finishOnboarding already deleted Graph A, Back can never cross from Graph B into Graph A.

Dumb screens, id-only keys

Screens are 'dumb' composables that only render and report taps via callbacks; nav keys carry only an id (e.g. DetailKey(itemId)) and the screen resolves the full object via 'itemById(id)', keeping each key tiny and serializable.

NavViewModelState

The one big idea

Screen state and the logic that changes it belong in a ViewModel that all screens share, not inside composables: the UI only reads state flowing down and sends events up. On Detail you tap a star to favorite the planet; that star then shows next to it back in the list. A ViewModel holds the set of favorites (and the screens also keep a small local tap counter), so favorites survive rotation and navigating away and back — unlike plain 'remember' state, which would reset.

Screens you'll see

  • CategoriesScreen — Level 1, stateless: a LazyColumn of categories (Rocky Planets, Gas Giants); tapping a row calls onOpen with its id to navigate forward.
  • ItemsScreen — Level 2, stateless: a header for the chosen category plus a list of its planets, drawing a star next to any planet whose id is in the favoriteIds set.
  • DetailScreen — Level 3, stateless: one planet's title and blurb, a star/unstar toggle button wired to the ViewModel, and a rememberSaveable 'Local taps' counter for contrast.
  • ItemsRoute / DetailRoute — Thin stateful wrappers that collect the ViewModel's favorites StateFlow and forward plain values plus event lambdas down to the stateless screens.
  • AppNavigation — Owns the Nav3 back stack, obtains the one shared FavoritesViewModel, and maps each NavKey to its screen via NavDisplay + entryProvider.

Key concepts

ViewModel

'FavoritesViewModel : ViewModel()' is a state holder that lives outside any composable. Because it is scoped to the Activity (a ViewModelStoreOwner), the framework re-attaches the SAME instance after a rotation instead of rebuilding it, so its favorites set survives.

StateFlow + MutableStateFlow

Inside the ViewModel, a private '_favorites = MutableStateFlow<Set<Int>>(emptySet())' is the writable source of truth, and the public 'val favorites: StateFlow<Set<Int>>' (via 'asStateFlow()') is the read-only view the UI observes. Encapsulation means only 'toggleFavorite' can change the set.

collectAsStateWithLifecycle

'favoritesViewModel.favorites.collectAsStateWithLifecycle()' subscribes the composable to the StateFlow and turns its latest value into Compose State, so the screen recomposes when favorites change. Unlike plain collectAsState, it pauses collection when the UI drops below STARTED (e.g. backgrounded) and resumes on return.

Unidirectional Data Flow (UDF)

State flows DOWN (the ViewModel exposes 'favorites' which the UI reads) and events flow UP (the button calls 'onToggleFavorite', which the Route forwards to 'favoritesViewModel.toggleFavorite(item.id)'). The ViewModel updates its StateFlow, which re-emits and recomposes the UI, closing the loop.

Stateful Route vs stateless Screen

Each level splits into a 'Route' composable that talks to the ViewModel (e.g. 'ItemsRoute', 'DetailRoute') and a dumb 'Screen' composable that takes plain values and lambdas (e.g. 'ItemsScreen', 'DetailScreen'). The stateless screens never touch a ViewModel, which is why @Preview can render them with hand-supplied values.

viewModel() scoping

'val favoritesViewModel: FavoritesViewModel = viewModel()' in AppNavigation grabs ONE instance scoped to the nearest ViewModelStoreOwner (the Activity) and passes it to all three screens, so every screen reads and writes the same single source of truth.

remember vs rememberSaveable vs ViewModel

remember survives only recomposition; rememberSaveable (used for the Detail 'count') also survives rotation and process death via the saved-instance Bundle but is local and Bundle-limited; a ViewModel survives rotation and navigation and is shared and holds logic, but loses state on process death unless backed by a SavedStateHandle.

Nav3 keys and back stack

Each screen is named by a @Serializable NavKey: 'CategoriesKey' (a data object, no args), 'ItemsKey(categoryId)' and 'DetailKey(itemId)' (data classes carrying tiny ids). 'rememberNavBackStack(CategoriesKey)' holds the stack; 'backStack.add(...)' pushes forward and 'removeLastOrNull()' pops back.

Screens you'll see

  • CategoriesScreen — LEVEL 1: a LazyColumn of the two planet categories; tapping a row pushes an ItemsKey (forward transition).
  • ItemsScreen — LEVEL 2: a header for the chosen category plus a scrolling list of its planets; rows push a DetailKey, the header button pops back.

Key concepts

ContentTransform

One transition bundled as an EnterTransition (how the incoming screen appears) plus an ExitTransition (how the outgoing screen leaves). Built here with the 'togetherWith' infix, e.g. slideInHorizontally(...) togetherWith slideOutHorizontally(...).

Transition spec lambda

Each NavDisplay spec is a lambda with receiver 'AnimatedContentTransitionScope<Scene<NavKey>>', which is why inside it you can call bare builders like slideInHorizontally and fadeIn directly without a qualifier.

Forward vs pop vs predictive pop

NavDisplay diffs the back stack and picks one of three specs: transitionSpec on push (forwardSpec), popTransitionSpec on back (popSpec, the mirror image), and predictivePopTransitionSpec for the Android 14+ back-swipe peek (predictivePopSpec, a scale+fade).

tween + easing

Every slide/fade takes an animationSpec of tween(TRANSITION_MS, easing = TRANSITION_EASING). TRANSITION_MS is 400ms and TRANSITION_EASING is FastOutSlowInEasing, the Material standard accelerate-then-decelerate curve.

offset lambdas

slideInHorizontally/slideOutHorizontally take a lambda over the screen size, e.g. initialOffsetX = { fullWidth -> fullWidth } to start one screen-width to the right, or targetOffsetX = { fullWidth -> -fullWidth / 3 } to only slide a third of the way off.

Per-entry metadata override

An individual entry can carry its own specs via 'metadata = NavDisplay.transitionSpec { ... } + NavDisplay.popTransitionSpec { ... }'. Each helper returns a Map you merge with '+'; entry metadata WINS over the NavDisplay-wide defaults. The FactKey entry uses this to slide vertically.

NavKey + @Serializable keys

Each screen is a key implementing NavKey: CategoriesKey is a 'data object' (a singleton, no args), while ItemsKey/DetailKey/FactKey are 'data class' types carrying ids. All are @Serializable so the stack survives process death.

rememberNavBackStack / NavDisplay

rememberNavBackStack(CategoriesKey) holds the list of keys; backStack.add pushes (forward), backStack.removeLastOrNull pops (back). NavDisplay renders whichever key is on top and animates the swap.

overlaidEntries = entries.dropLast(1)

Inside DialogSceneStrategy, the DialogScene's overlaidEntries is everything under the dialog — so the detail screen stays composed and on screen behind it. That single fact is why an overlay looks different from a replace.

Key concepts

Composable function

A function marked '@Composable' (like 'NoteCard' or 'SectionHeader') that DESCRIBES UI instead of mutating widgets; Compose re-runs it ('recomposes') when the data it reads changes.

State hoisting

A child takes its current value as a parameter (state DOWN) and reports changes through a callback (event UP); the parent holds the real value. 'FavoriteToggle' takes 'isFavorite: Boolean' and 'onToggle: (Boolean) -> Unit' and stores nothing.

Stateless vs stateful overload

'NotesScreen' exists twice: the stateless one is pure parameters plus callbacks, and the stateful one is the only place that calls 'rememberSaveable' and then forwards state into the stateless version. This is the 'state holder vs stateless content' split.

rememberSaveable + mutableStateOf

'var notes by rememberSaveable { mutableStateOf(seedNotes) }' creates observable state that triggers recomposition on write AND survives screen rotation or low-memory recreation, unlike plain 'remember'.

Hoisted text input

'OutlinedTextField' has no internal text: its 'value' is the hoisted 'newTitle' and 'onValueChange' sends every keystroke up to the owner, who is the single source of truth for the field.

Material 3 theming

Components read 'MaterialTheme.typography' (e.g. titleSmall, bodyMedium) and 'MaterialTheme.colorScheme' (e.g. primary, onSurfaceVariant) instead of hardcoded sizes or hex, so light/dark and dynamic color work automatically.

Layout primitives

'Column' stacks children vertically, 'Row' horizontally, 'Box' overlaps them, and spacing comes from 'Spacer', 'padding', and 'Arrangement.spacedBy' rather than manual gaps.

LazyColumn with keys

'LazyColumn' only composes the rows on screen, and 'items(notes, key = { it.id })' gives each note a stable id so Compose can track items efficiently as the list grows.

@Preview

'@Preview' renders a composable in Android Studio's design pane without an emulator. The previews call the STATELESS 'NotesScreen' with fake data and no-op callbacks, which is exactly why hoisting matters.

Composition: nesting & reusable composables

A Compose UI is a TREE: containers like 'Column' and 'LazyColumn' take their children in a trailing { } lambda, so 'NotesScreen' nests a header, input, button, toggle, and a list. Repeated UI is EXTRACTED into its own @Composable and reused by CALLING it: 'SectionHeader' is called three times and 'NoteCard' once per note. Key concepts

The Modifier chain is ordered

'Modifier.a().b().c()' returns a NEW Modifier at each step that WRAPS the previous; Compose applies them outside-in (first = outermost layer). That is why order matters.

Order matters: background vs padding

'Modifier.background(Blue).padding(16.dp)' fills blue first then insets the text (blue shows AROUND the text); 'Modifier.padding(16.dp).background(Blue)' leaves a transparent margin first then fills (blue HUGS the text). Same two modifiers, different picture.

Sizing

'size(w, h)' fixes both dimensions; 'fillMaxWidth()' takes the full width and 'fillMaxWidth(0.5f)' takes half; 'wrapContentWidth' shrinks back to the content. 'fillMaxHeight' / 'fillMaxSize' are the height / both variants.

Arrangement vs Alignment

'Arrangement' spreads children along the layout's MAIN axis (vertical for Column, horizontal for Row): 'spacedBy(8.dp)', 'Center', 'SpaceBetween', 'SpaceAround', 'SpaceEvenly'. 'Alignment' lines them up on the CROSS axis: 'horizontalAlignment' in a Column, 'verticalAlignment' in a Row.

weight

Inside a Row/Column, 'Modifier.weight(n)' divides the LEFTOVER main-axis space by ratio. A 'weight(2f)' child is twice as wide as a 'weight(1f)' child; fixed-size children are measured first and weights split what remains. 'weight' only exists inside a RowScope / ColumnScope.

Box & Modifier.align

'Box' stacks its children on top of each other; each child calls 'Modifier.align(Alignment.TopStart / Center / BottomEnd / …)' to position itself within the Box. 'align' only exists inside a BoxScope.

Decoration & transforms

'clip(CircleShape)' rounds, 'border(width, color, shape)' outlines, 'shadow(elevation, shape)' lifts off the surface; 'offset' nudges, 'alpha' fades, 'rotate' spins, and 'aspectRatio(16f/9f)' locks a width:height ratio.

The one big idea

  • NotesContent — Stateless UI shell: a Scaffold with the search field, the optional selection bar, and the switch between column and grid (or the empty state).

  • NotesGrid — Renders the SAME notes as a two-column LazyVerticalGrid (GridCells.Fixed(2)) using the same stable keys.
  • NoteCard — One clickable note item (used by both layouts) whose background highlights when selected.

Key concepts

LazyColumn / LazyVerticalGrid

Compose's scrolling list and grid that only compose the rows currently on screen. 'LazyColumn' stacks one column; 'LazyVerticalGrid' with 'GridCells.Fixed(2)' lays the same data in two columns. They replace RecyclerView with almost no boilerplate.

Stable item keys

Passing 'key = { it.id }' to 'items(...)' tells Compose to track each row by the note's unique 'id' instead of its position. That preserves scroll position, selection highlight, and animations across filtering and reordering.

State hoisting

All mutable state (query, selectedIds, layout) lives in 'NotesViewModel', not inside the composables. The UI receives state as parameters and reports events via lambdas, so the UI is a predictable function of state.

Derived state

'visibleNotes' is a computed property (a 'get()') that re-filters 'allNotes' by 'query' on every read. The filtered list is never stored, so there are no stale duplicates to keep in sync.

Immutable updates

'toggleSelected' builds a brand-new 'Set<Int>' ('selectedIds - id' or '+ id') rather than mutating the old set. Replacing the reference is what lets Compose detect the change and recompose.

Stateful vs stateless split

'NotesScreen' is the only composable that touches a ViewModel; 'NotesContent' and everything below it take state as parameters. This split is exactly what lets the '@Preview' functions render the UI with hand-supplied data and no ViewModel.

Empty state

When 'notes.isEmpty()' the UI shows the dedicated 'EmptyState' composable instead of a blank screen, echoing the query so the user knows their search matched nothing rather than thinking the app broke.

ViewModel survives config changes

'NotesViewModel' extends 'androidx.lifecycle.ViewModel' and is obtained via 'viewModel()'. Because the ViewModel outlives Activity recreation, the search text, selection, and layout are kept across rotation.

The one big idea

A WebSocket is a STREAM, not a single call: unlike HTTP request/response, one connection stays open and either side can push messages at any time - so model the connection as a Flow of events and fold that stream into one immutable UI state.

What it does

The app opens on a single chat-style screen that immediately connects. A status line shows the live connection state - Disconnected, Connecting, Connected (green), or Failed (red). Below it a scrolling transcript fills the screen: messages you send are right-aligned and prefixed 'you:', messages from the server are left-aligned and prefixed 'server:'. At the bottom a text box plus a Send button let you type a message; the echo server sends the exact same text straight back, so each line you send appears twice - once as yours, once echoed. By default it runs offline against a fake echo client, so it connects and echoes with no network.

Screens you'll see

  • LiveScreen — Stateful entry composable: grabs the LiveViewModel, collects its uiState StateFlow, opens the connection once via LaunchedEffect, and owns the text-field input.
  • LiveContent — Stateless renderer: draws the status chip, the scrolling transcript (a LazyColumn), and the input row, reporting edits and sends through callbacks.
  • MessageRow — One transcript line - right-aligned and primary-coloured when it is from you, left-aligned and default when it came from the server.
  • LiveContentConnectedPreview / LiveContentConnectingPreview — Two @Preview functions that drive the stateless content with hand-made LiveUiState values, no real socket.

Key concepts

WebSocket vs HTTP

HTTP (the NetworkParsing demo) is one request then one response. A WebSocket keeps a single connection open so the server can push messages whenever it likes - chat, live scores, presence. The screen models that as a sequence of events over time, not a single answer.

Events as a sealed stream

ConnectionEvent is a sealed interface listing everything the wire can report: Connecting, Open, MessageReceived(text), Closed(reason), Failure(error). The connection exposes them as a Flow<ConnectionEvent> the ViewModel collects.

OkHttp WebSocketListener

RealWebSocketClient calls httpClient.newWebSocket(request, listener); the listener's onOpen / onMessage / onClosing / onFailure callbacks each tryEmit a ConnectionEvent into a MutableSharedFlow. tryEmit never suspends, so the OkHttp callback thread is never blocked.

A pure reducer

reduceLiveState(state, event) is a plain function with no coroutines that folds one event into the next LiveUiState (status + transcript). Pulling the fold out as a pure function makes the whole state machine unit-testable without a socket or a dispatcher.

Optimistic UI

When you send, the ViewModel appends your line to the transcript immediately (fromMe = true) rather than waiting for the network. The server's echo arrives later as a MessageReceived and is appended by the reducer (fromMe = false).

Real vs Fake switch

LiveConnection is an interface with connect/send/close plus events(). RealWebSocketClient opens a live wss:// socket; FakeWebSocketClient scripts a connect and echoes sends with no network. provideLiveConnection(useFake) is the single switch - the default fake keeps the app and tests offline.

LaunchedEffect for one-time work

LiveScreen opens the socket with LaunchedEffect(Unit) { viewModel.connect() }. Keying on Unit means it runs exactly once when the screen first appears, not on every recomposition - the right place for a side effect like opening a connection.

Lifecycle-aware collection

collectAsStateWithLifecycle() reads the StateFlow but pauses while the screen is not visible, the safe way to observe a flow from Compose - the same pattern used across the other ViewModel demos.

Two-step runtime permission

RequestMultiplePermissions

rememberLauncherForActivityResult with the RequestMultiplePermissions() contract requests fine and coarse together and hands back a Map<permission, granted>. The app treats ANY granted permission as enough, because the user may grant only the coarse (approximate) level.

FusedLocationProviderClient

The modern, battery-efficient device-location API. RealLocationRepository calls client.getCurrentLocation(PRIORITY_HIGH_ACCURACY, token) to ask the hardware for a single fresh fix, then maps the framework Location into the app's own LocationData.

Callback to suspend bridge

Play services exposes getCurrentLocation as a callback-based Task. suspendCancellableCoroutine wraps its success/failure listeners so callers just write 'val fix = repository.currentLocation()' - and cancelling the coroutine cancels the underlying request via a CancellationTokenSource.

DTO vs domain model

LocationData is a plain data class with latitude and longitude and no Android imports. Keeping the framework's android.location.Location out of the ViewModel and UI means they never depend on Play services types and are trivial to fake in tests.

UiState sealed interface

LocationUiState has exactly four cases: Idle (nothing requested yet), Loading (acquiring a fix), Success(location), and Error(message). Being sealed, a 'when' over it is exhaustive, so the UI is a pure function of state.

No fetch in init

Unlike the networking demos, the ViewModel does NOT fetch in its init block. A location read must wait until permission is granted, so the screen calls fetchLocation() only after the launcher reports a grant (or when a permission is already held).

Real vs Fake switch

LocationRepository is an interface with one method currentLocation(). Key concepts

ViewModel

NotesViewModel extends androidx.lifecycle.ViewModel, a holder that the framework keeps alive across configuration changes like rotation, so the state it owns is not destroyed and recreated with the Activity.

Immutable UiState

NotesUiState is a data class whose every property (notes, isLoading, newNoteTitle) is a 'val'. It is never edited in place; the ViewModel always builds a brand-new instance with .copy(...) so the screen always renders one consistent snapshot.

StateFlow exposure

The ViewModel keeps a private writable MutableStateFlow named _uiState and exposes it publicly only as a read-only StateFlow named uiState via asStateFlow(). The UI can read but never write the state, which enforces one-way data flow.

collectAsStateWithLifecycle

In NotesScreen, 'val uiState by viewModel.uiState.collectAsStateWithLifecycle()' subscribes to the StateFlow, turns each emission into Compose State so the screen recomposes, and pauses collecting while the screen is not visible.

Sealed event interface

NotesEvent is a sealed interface listing every user intention (NewTitleChanged, AddNote, ToggleDone, DeleteNote). Because it is sealed, the 'when (event)' in onEvent is exhaustive at compile time and needs no 'else'.

Unidirectional data flow (UDF)

State flows DOWN (uiState -> composables) and events flow UP (callbacks -> onEvent). The UI never mutates state; it only sends a NotesEvent, the ViewModel computes a new state, and Compose redraws.

viewModelScope + suspend repository

Async work runs in viewModelScope.launch and calls suspend functions on NotesRepository (loadNotes, addNote, toggleDone, deleteNote). The scope is cancelled automatically when the ViewModel is cleared, so coroutines never leak.

Stateless and previewable UI

NotesScreenContent holds no ViewModel and no remembered state, so the @Preview functions can hand it a fabricated NotesUiState and a no-op onEvent ({}) without constructing a ViewModel.

Key concepts

Room @Entity

The data class 'Note' is annotated '@Entity(tableName = notes)', which tells Room to generate a real SQLite table named 'notes'. Each property (id, title, body, done) becomes a column and each Note instance becomes a row.

@PrimaryKey autoGenerate

'@PrimaryKey(autoGenerate = true) val id: Long = 0' makes id the unique row id; passing 0 on insert tells SQLite to assign the next id automatically.

@Dao with reactive Flow

'NoteDao' is an interface where 'observeNotes(): Flow<List<Note>>' returns a live stream. Room re-runs the SELECT and emits a fresh list every time the notes table changes, so the UI updates with no manual refresh.

suspend DAO writes

insert/update/delete are marked 'suspend' because they touch disk and must run off the main thread; the ViewModel calls them inside 'viewModelScope.launch { }'.

Room database singleton

'NoteDatabase.getInstance()' uses '@Volatile' plus double-checked locking around 'Room.databaseBuilder(..., notes.db)' to build exactly one database instance per process, avoiding wasted connections and state corruption.

DataStore Preferences

'SettingsRepository' uses 'preferencesDataStore(name = settings)' to store simple key/value settings. It reads via 'dataStore.data.map { }' (a reactive Flow) and writes via the suspend, transactional 'dataStore.edit { }' block; it is the modern replacement for SharedPreferences.

stateIn (cold Flow to hot StateFlow)

The ViewModel converts each persistence Flow into a StateFlow with 'stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), initial)', so Compose always has a current value and collection stays tied to the ViewModel lifetime.

flatMapLatest switching queries

'settings.sortOrder.flatMapLatest { order -> ... }' makes the notes Flow re-subscribe to either observeNotes() or observeNotesByTitle() whenever the persisted sort setting changes, so DataStore and Room cooperate reactively.

AppReleaseBasics

Take a notes app from 'runs on my emulator' toward 'shippable': runtime permissions plus release build config.

Project 23 of 25 🗒️ notes domain ⏱ 15-25 min Concepts track
The one big idea

Shipping an app is two jobs the emulator hides from you: ask the user for dangerous permissions at run time (declare in the manifest AND request in code), and configure how the build is versioned, typed (debug vs release), and signed.

What it does

You see a single screen titled 'My Notes - Reminders' that explains the app wants permission to post notification reminders. A status line shows 'Notifications: GRANTED' (green) or 'Notifications: DENIED' (red) reflecting the live permission state. A button reads 'Request notification permission' when denied; tapping it pops the real Android system permission dialog. Once you allow it, the status flips to GRANTED and the button becomes disabled, reading 'Permission already granted'. There is nothing to navigate to - the whole lesson is this one flow plus the build files behind it.

Screens you'll see

  • NotificationPermissionScreen — Stateful screen that reads the current POST_NOTIFICATIONS status, builds the permission launcher, and fires the system dialog on tap.
  • NotificationPermissionContent — Stateless UI that draws the title, explanation, GRANTED/DENIED status text, and the request Button from plain inputs.
  • NotificationPermissionDeniedPreview / NotificationPermissionGrantedPreview — Two @Preview functions that render the stateless content with hand-supplied isGranted values, no real Context.

Key concepts

Two-step permission model

A dangerous permission needs BOTH a static <uses-permission> line in AndroidManifest.xml AND a run-time request in code. Declaring 'android.permission.POST_NOTIFICATIONS' in the manifest is not enough on API 33+; the user still has to approve it via a dialog.

ActivityResult permission launcher

rememberLauncherForActivityResult with the ActivityResultContracts.RequestPermission() contract gives you a 'launcher' object. Calling launcher.launch(permission) shows the OS dialog, and the lambda receives a Boolean 'granted' when the user answers.

checkSelfPermission

ContextCompat.checkSelfPermission(context, permission) returns PackageManager.PERMISSION_GRANTED or PERMISSION_DENIED. It is how the screen knows the real status on first launch (and after the user returns from settings) instead of guessing.

SDK version guard

POST_NOTIFICATIONS only exists on Android 13 (API 33, Build.VERSION_CODES.TIRAMISU) and up. The code checks Build.VERSION.SDK_INT >= TIRAMISU and treats older devices as already granted, so it never requests a permission the platform does not recognize.

versionCode vs versionName

versionCode is an integer (here 1) that Google Play uses ONLY to decide which build is newer - bump it every published release; users never see it. versionName is a human string (here '1.0') shown in the store and Settings; it is not used for ordering.

Build types: debug vs release

buildTypes { debug { ... } release { ... } } define two flavors. debug is auto-signed with a throwaway key, debuggable, no shrinking - never shippable. release is what you ship: it would be signed with your keystore and usually minified with R8 (isMinifyEnabled / isShrinkResources).

Release signing (signingConfigs)

Every APK/AAB must be cryptographically signed. The signingConfigs { create('release') { ... } } block is left COMMENTED so assembleDebug works with no keystore; it shows where a release.jks store file, alias, and passwords (pulled from env vars, never hardcoded) would be wired in.

Stateful vs stateless split

NotificationPermissionScreen owns the Context, state, and launcher; NotificationPermissionContent takes only isGranted plus an onRequestPermission callback. That separation is what lets the @Preview functions render the UI without building a real launcher or Context.

The one big idea

What it does

MatchMania is a build-it-yourself starter (the assignment for Homework 7), not a finished demo. Key concepts

Pure logic, separate from the UI

MatchLogic.kt has no Android or Compose imports, so the rules can be exercised by plain JVM unit tests in milliseconds. The Compose layer just renders the board and forwards taps.

Immutable state updates

Functions never edit a board in place; they build and return a NEW Board (via list.map / copy()). Reassigning that new value into state is exactly what makes Compose redraw, and what makes the rules easy to test.

State hoisting with remember + mutableStateOf

The screen owns the board as 'var board by remember { mutableStateOf(newBoard(...)) }'. State lives at the top and flows down; a tap flows up and becomes a new board stored back in that same state.

A three-state tile (a small state machine)

Each tile is FaceDown, FaceUp, or Matched. The flip rules decide the next board from how many tiles are currently face-up (0, 1, or 2) — that branching is the heart of the game.

Seeded Random for testability

newBoard takes a Random parameter (defaulting to the system RNG). Tests pass Random(seed) so the shuffle is repeatable and assertions are stable — dependency injection in its simplest form.

A 4-column grid

The 16 tiles render as a fixed 4-column grid of square, tappable cells whose look depends on the tile's state — the layout you build in task C2.

JVM unit testing with JUnit

Because the rules are pure, MatchLogicTest.kt verifies generation, flipping, matching, mismatch-then-clear, win, and reset with assertEquals / assertTrue — no emulator needed.
