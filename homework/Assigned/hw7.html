Homework 7 — Match Mania: A Memory-Match Grid Game

Build a complete single-screen Android game in Jetpack Compose: a memory / pairs-matching grid. You'll design a clean three-layer app — pure game rules, a Compose UI that draws them, and unit tests that prove the rules are correct — and implement four actions: start a new game, reset the board, flip a tile, and detect a win. A runnable starter project is provided — it already compiles and launches; you fill in the numbered TODOs to bring the game to life.

▶ Projects, Explained ← Homework 6

Every game you build with Compose has the same skeleton: some state (what the board looks like right now), a pure rules layer that decides how that state changes, a UI that draws the state and forwards taps, and tests that prove the rules are right. In this assignment you'll build all four for a memory game — and because the rules live apart from the UI, you can test them on a plain JVM in milliseconds, then drop them into a thin Compose screen.

The game. A grid of face-down tiles. Each symbol (emoji, flag, fruit — your choice of theme) is hidden on exactly two tiles. Tap a tile to flip it face-up; flip a second; if the two faces match, they lock as a found pair; if they don't, you see them, then they flip back. Find all the pairs to win. (This is the classic "Concentration" / memory card game.)

The exact rules your game must follow

These are the spec — your logic and tests must match them precisely.

  1. The board is 16 tiles in a 4×4 grid. There are 8 distinct symbols; each appears on exactly two tiles; the 16 tiles are shuffled into random positions.
  2. Every tile is in one of three states: FaceDown, FaceUp, or Matched. A new board has all 16 tiles FaceDown.
  3. Tapping a tile resolves as follows, based on how many tiles are currently face-up (not counting matched ones):
    • 0 face-up: the tapped FaceDown tile flips to FaceUp.
    • 1 face-up: the tapped FaceDown tile flips to FaceUp, making two. Immediately compare them: if the two faces are equal → set both to Matched; if not → leave both FaceUp so the player sees the miss.
    • 2 face-up and they did NOT match: the next tap first flips those two back to FaceDown and reveals nothing else (that tap just clears the miss). The following tap starts a fresh pair.
  4. Tapping a Matched tile, or the same tile that is already FaceUp, does nothing.
  5. Win = all 16 tiles are Matched.
  6. Track a moves counter: increment it by one each time a second tile is revealed (i.e., once per pair attempt, match or miss).
Why "clear on the next tap" instead of a timer? It keeps the rules pure and synchronous: every action is "old board in → new board out," with no coroutines or delays to design tests around. (Want the polished auto-flip-after-a-moment behavior? That's the optional stretch goal in Part E, using LaunchedEffect + delay.)

How to work this assignment

1

Sketch the data first

Before writing any UI, decide your types on paper: a tile has a face and a state; the board is a list of tiles. Then walk through rule 3 by hand for a few taps so the three branches of the flip logic are clear in your head.

2

Build logic → tests → UI, in that order

Do Part B (logic) and Part D (tests) together — write a function, write its test, see green — before you touch the UI. A correct, tested rules layer makes the Compose part almost mechanical.

3

Run on the emulator early and often

Get a board drawing as soon as possible, then wire one button at a time. The emulator (and the @Preview) is the source of truth for the UI.

Part B. The game logic — MatchLogic.kt Android Studio

Pure Kotlin, no Android / no Compose imports — that's what lets Part D test it instantly. The constants and the data model (TileState, Tile, and Board with its faceUpIndices() / isSolved() / pairsFound() helpers) are provided in the starter — read them. Your job is the three stubbed functions below. Each must be pure: take a board, return a new board, never mutate in place. Implement one, test it (Part D), then move to the next.

Implement newBoard (B2), Board.flip (B3), and Board.reset (B4)
B2 Build a fresh, shuffled board — newBoard(symbols, random) MatchLogic.kt

Return a brand-new board that satisfies the spec: the first PAIR_COUNT symbols, each appearing exactly twice, shuffled into the 16 positions, every tile FaceDown, and moves at 0. The function already takes a Random (defaulting to the system RNG) — use it for the shuffle.

Think about: how do you turn 8 symbols into 16 shuffled tiles? And why does accepting the Random as an argument — rather than calling it inside — make this function testable?

B3 Flip a tile — Board.flip(index) (the heart of the game) MatchLogic.kt

Apply the tap rules (see "The exact rules your game must follow", rule 3) and return a new board. The outcome depends on how many tiles are currently face-up — that's what faceUpIndices() is for.

Think about: what are the distinct situations a tap can land in, and what does each one produce? How do you change only the one or two tiles involved while leaving every other tile — and the original board object — untouched? Exactly when does moves go up, and which taps must do nothing at all?

B4 Reset the board — Board.reset() MatchLogic.kt

Return a board with the same faces in the same positions, but every tile FaceDown and moves back to 0. (Reset = same layout, cleared; New Game = a fresh shuffle — they're different actions.)

Part C. The Compose UI — MainActivity.kt Android Studio

The starter already wires the screen frame: the Activity, the Scaffold + top bar, the hoisted board state, the stats line, and the three buttons (New Game / Reset / Check). Your work is the board itself (C2) and the win banner (C3). Keep the shape in mind: state flows down into the UI, and a tap flows up to become a new board you store back in state.

Build the board (C2) and the win banner (C3)
C2 The board grid MainActivity.kt

Replace the BoardPlaceholder with the real board: the 16 tiles laid out as a 4-column grid of square cells. Each cell must (1) be tappable and drive the game — a tap on tile i produces a new board — and (2) look clearly different in each state: hidden while FaceDown, showing its symbol while FaceUp, and visibly "done" once Matched.

Think about: which Compose container lays items out in a fixed number of columns? How do you keep each cell square? How does a cell get both the tile and its index, so a tap knows which one it is? And where must the new board be stored for the screen to redraw?

C3 Win banner MainActivity.kt

When the board is solved, show a clear, celebratory message that includes the move count (e.g. "🎉 You win in N moves!"). The stats line (pairs found / moves) is already on screen; you add the banner.

Think about: how do you derive "did the player win?" — is it new state you have to track, or does it fall straight out of the board you already hold?

Part D. Unit tests — MatchLogicTest.kt Android Studio

Prove the rules in MatchLogic.kt are correct on the plain JVM — no emulator. The starter gives you two example tests and a checklist; write the rest yourself (at least eight tests total).

Verify each behavior (JUnit 4)

Your tests must cover, at minimum:

  • Generation — a fresh board has 16 tiles, exactly PAIR_COUNT distinct faces with each used twice, all FaceDown, moves == 0, and is not solved.
  • Determinism — the same seed produces the same board (so the rest of your tests are repeatable).
  • Flip · reveal — tapping a face-down tile on an empty board reveals exactly that tile.
  • Flip · match — revealing a tile and then its twin locks both as Matched and counts one move.
  • Flip · mismatch — revealing two different tiles leaves both FaceUp (one move); the next tap clears them back to FaceDown without adding a move.
  • Immutability — flip() returns a new board and does not change the board it was called on.
  • Win & reset — matching every pair makes the board solved; reset() returns all tiles to FaceDown with moves == 0 and the same faces in the same order.

You'll need to work out: how to seed the RNG for a repeatable board, and how to find a tile's twin (the other tile sharing its face) so you can force a guaranteed match.

Run them: right-click MatchLogicTest.kt ▸ Run, or ./gradlew testDebugUnitTest. All tests must pass.
Deliver: a green test run (JUnit panel screenshot or the BUILD SUCCESSFUL terminal output).

Part E. Stretch goals optional

Worth bonus credit. Each builds on a real Compose concept beyond the required app — attempt them only after the core game is solid and tested.

E1Auto-flip mismatches after a beatLaunchedEffect + delay

Instead of "clears on next tap," make a mismatched pair flip back down on its own after ~800ms. Use LaunchedEffect(board) + kotlinx.coroutines.delay(800) when board.faceUpIndices().size == 2 and they don't match. Keep the synchronous rule available so your existing tests still pass — gate the auto-flip in the UI only.

E2Choose a difficultystate + grid sizing

Add buttons for 4×4 / 4×5 / 6×6 (keep an even tile count so pairs work out). Drive GridCells.Fixed(columns) and the symbol count from state.

E3Accessibilitysemantics / stateDescription

Tiles convey state by color alone, which a screen reader can't perceive. Add Modifier.semantics { stateDescription = … } so it announces "face down / showing 🐶 / matched."

E4Flip animationanimateColorAsState / AnimatedContent

Animate the reveal — a color cross-fade with animateColorAsState, or swap the face/back with AnimatedContent.
