Part A. Storage — the Room query Android Studio

Make the database hand back tasks in a sensible order. First, read the three data/ files — they're your Room reference.

Read data/Task.kt + data/TaskDatabase.kt, then complete TODO 1 in data/TaskDao.kt

What Room gives you here. Task is a @Entity — one annotated Kotlin class that becomes a SQLite table named tasks, one row per task (fields: id, title, notes, done, priority 0–2, createdAt). TaskDatabase ties the entity to its DAO and builds the on-disk file. The DAO's observeTasks() returns a reactive Flow<List<Task>> — Room re-runs the query and re-emits a fresh list every time the table changes, so the UI never has to manually reload.

TODO 1 Sort the task query data/TaskDao.kt builds on RoomAndPreferences — @Query

The shipped query is SELECT * FROM tasks with no order, so rows come back arbitrarily. Add an ORDER BY so the list reads well: unfinished tasks first, then higher priority first, then newest first.

Hint (API only): edit just the SQL string — @Query("SELECT * FROM tasks ORDER BY done ASC, priority DESC, createdAt DESC"). Room checks this SQL at compile time against the Task columns, so a typo (it's createdAt, not created_at) fails the build with a clear message — that's Room protecting you.

TODO 3 Save a task (insert OR update) TaskViewModel.kt builds on RoomAndPreferences — @Insert/@Update

Implement save(task). A new task has id == 0L (the entity default) → insert; an existing task has a real id → update. Both DAO calls are suspend, so they must run inside a coroutine.

Hint (API only): viewModelScope.launch { if (task.id == 0L) taskDao.insert(task) else taskDao.update(task) }.
TODO 4 Delete + toggle done TaskViewModel.kt builds on NotesViewModel.toggleDone

Implement delete(task) and toggleDone(task), both as suspend Room writes launched on viewModelScope. Toggling persists the same task with its done flag flipped.

Hint (API only): viewModelScope.launch { taskDao.delete(task) } and viewModelScope.launch { taskDao.update(task.copy(done = !task.done)) } — copy() comes free with the data class.

Part D. Navigation — connect the two screens Android Studio

Make the buttons actually move between screens. In MainActivity.kt's AppNavigation — compare to NavListDetail.

Complete TODO 8 in MainActivity.kt (the entryProvider)

How Nav3 works here. The single Activity holds a back stack of keys (rememberNavBackStack(TaskListKey)). NavDisplay draws whichever key is on top. You move by changing the stack: backStack.add(key) pushes a screen on, backStack.removeLastOrNull() pops back. The two @Serializable keys are TaskListKey (no args) and TaskEditKey(taskId) — where a null taskId means "new task" and a non-null id means "edit that task".

TODO 8 Wire the three nav actions MainActivity.kt → AppNavigation builds on NavListDetail — add / removeLastOrNull

The add button, row taps, and Save/Cancel currently do nothing. In the entryProvider, make each one navigate:

  • 8a — Add: onAddTask → backStack.add(TaskEditKey(taskId = null)) (open the form for a brand-new task).
  • 8b — Open: onOpenTask → backStack.add(TaskEditKey(taskId = task.id)) (open the form to edit that task).
  • 8c — Back: in onSave (after viewModel.save(task)) and in onCancel → backStack.removeLastOrNull() (pop back to the list).
Hint: these are one-liners that go where each // TODO 8a/8b/8c comment already sits. System-back already pops (see NavDisplay(onBack = …)), so once 8c is in, all three ways out of the form work.
