Part B. Modifier order is a feature ★

The most important idea here. A modifier chain applies outside-in: the first modifier wraps everything after it. Reordering the same modifiers changes the UI.

Refresher — what is a Modifier?

A Modifier is Compose's standard, reusable way to decorate and configure a composable from the outside — its size, position, padding, background, border, shape, and even behavior like click handling — without editing the composable itself. That is why nearly every built-in composable (Text, Column, Button, Box …) accepts an optional modifier: parameter: it's the one knob you turn to change how that composable looks and lays out.

You build a modifier by chaining calls onto the Modifier companion object. Each call returns a new modifier with one more instruction added, so a chain is really an ordered list of instructions that Compose applies top-to-bottom (outside-in). Read this one line by line:

Text(
    "Hi",
    modifier = Modifier                         // start from the Modifier companion object
        .fillMaxWidth()                      // 1) be as wide as the parent allows   (size / layout)
        .padding(16.dp)                       // 2) inset everything after this by 16dp (size / layout)
        .clip(RoundedCornerShape(8.dp))      // 3) round the corners FIRST            (drawing)
        .background(Color(0xFF3DDC84))      // 4) fill the area — respects the clip  (drawing)
)

Three things to keep in mind:

  • It's ordered and immutable. Each .something() returns a brand-new modifier; nothing is mutated in place. The chain is data — a recipe Compose reads in sequence.
  • Order is meaningful. Because the chain is applied outside-in, the first modifier wraps the rest. .padding().background() is not the same as .background().padding() — proving that to yourself is the whole point of Part B.
  • There are two broad kinds. Some modifiers change the composable's measured size / position (size, padding, fillMaxWidth, weight, offset); others only change how it is drawn or behaves (background, border, clip, shadow, alpha, rotate, and — beyond the playground — clickable). Task T6 makes you label which is which.
The convention. When you write your own composable, give it a modifier: Modifier = Modifier parameter (default to the empty Modifier), make it the first optional parameter, and apply it to your outermost element. That hands the caller the same positioning control the built-in composables give you — exactly how every screen in the teaching projects is written.
The mental model. Each modifier is a wrapper box around the next. .background().padding() paints first, then insets — so paint covers the padding. .padding().background() insets first, then paints — so paint stops at the inset. The playground renders a chain as exactly this stack of nested boxes; real Compose measures it the same way.

Part C. State & interactivity — real Compose

Now beyond the playground. These need recomposition, so they run in Android Studio only. Add each as a new @Composable (with a @Preview) in ComposeCatalog, and run on an emulator/device.

The core idea. Compose UI is a function of state. You hold state with var x by remember { mutableStateOf(...) }; when you change x, Compose recomposes — re-runs the affected composables to redraw with the new value. The playground can't do this; the emulator can.
T7 — a counter (state & recomposition) Android Studio

Build a counter: a Text showing a number and a Button that increments it. Tapping the button must update the displayed count live.

@Composable
fun Counter() {
    // remember survives recomposition; mutableStateOf makes changes trigger one
    var count by remember { mutableStateOf(0) }       // the single source of truth
    Column(modifier = Modifier.padding(16.dp),
        horizontalAlignment = Alignment.CenterHorizontally) {
        Text("Count: $count", style = MaterialTheme.typography.headlineSmall)
        Spacer(Modifier.height(8.dp))
        Button(onClick = { count++ }) { Text("Add one") }  // changing state -> recompose
    }
}

Then, in a comment, answer: change remember { mutableStateOf(0) } to a plain var count = 0 (no remember, no state). Run it and tap. What happens to the number, and why?

Deliver: working counter code, an emulator screenshot, and your explanation of the plain-var version.
T8 — a real toggle that drives the UI (state hoisting) Android Studio

Build a Switch whose state actually does something. Hold a Boolean in state and have the switch's onCheckedChange update it; when on, change something visible — e.g. show an extra Card, or swap a Text between “Dark” and “Light”.

Split it into a stateless composable that takes checked: Boolean + onCheckedChange: (Boolean) -> Unit, and a stateful caller that owns the remembered state — the state-hoisting pattern.

@Composable
fun ToggleDemo() {                                  // stateful: owns the state
    var on by remember { mutableStateOf(false) }
    ToggleRow(checked = on, onCheckedChange = { on = it }) // state down, event up
}

@Composable
fun ToggleRow(checked: Boolean, onCheckedChange: (Boolean) -> Unit) { // stateless: reusable
    Row(verticalAlignment = Alignment.CenterVertically, modifier = Modifier.padding(16.dp)) {
        Text(if (checked) "Notifications: ON" else "Notifications: OFF",
            modifier = Modifier.weight(1f))
        Switch(checked = checked, onCheckedChange = onCheckedChange)
    }
}
Why hoist? The stateless ToggleRow is now trivial to @Preview and reuse — you pass it any value. This is the same Route/Screen split you saw in NavViewModelState.
Deliver: both composables + an emulator screenshot in each state (on and off).
T9 — a bound TextField Android Studio

Build a text field bound to state: hold the text in remember { mutableStateOf("") }, feed it to the field's value, update it in onValueChange, and below it show a live Text that echoes what's typed plus a character count.

@Composable
fun NameField() {
    var name by remember { mutableStateOf("") }
    Column(modifier = Modifier.padding(16.dp)) {
        OutlinedTextField(
            value = name,
            onValueChange = { name = it },                // every keystroke updates state -> recompose
            label = { Text("Your name") },
            singleLine = true
        )
        Spacer(Modifier.height(8.dp))
        Text("Hello, ${'$'}{name.ifBlank { "stranger" }} (${'$'}{name.length} chars)")
    }
}

In a comment, explain the loop: typing → onValueChange → state changes → recomposition → field and the echo Text both show the new value. (This is unidirectional data flow.)

Deliver: code + an emulator screenshot mid-typing + your data-flow comment.
