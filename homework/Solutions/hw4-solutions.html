One mental model carries the whole assignment A modifier chain is an ordered, immutable list of instructions applied outside-in: the first modifier wraps everything after it. Layout modifiers (size, padding, fillMaxWidth, weight) change the measured box; draw modifiers (background, border, clip, shadow, alpha) only change how that box is painted. Parts A–B are about order; Parts C–D add state (remember { mutableStateOf(...) } → recomposition).

Part A. The three containers & the layout tree

Column stacks vertically, Row horizontally, Box layers on the z-axis. Arrangement spaces children on the main axis; Alignment positions them on the cross axis.

T1 — a Column with arrangement & alignmentBoth

A full-width Column with 16.dp padding, 8.dp between children, centered horizontally, holding three Texts and a Button. Then switch verticalArrangement to Arrangement.spacedBy(24.dp) and note what visibly changes.

Where: drop into ComposeCatalog as a new @Composable with a @Preview, then run.

@Composable
fun T1PlanetColumn() {                                   // T1 — a vertical stack
    Column(
        // fillMaxWidth = take ALL horizontal space the parent allows (a layout/size modifier).
        // padding(16.dp) = inset the whole column's content by 16dp on every edge.
        modifier = Modifier
            .fillMaxWidth()
            .padding(16.dp),
        // verticalArrangement controls spacing on the MAIN axis (vertical, for a Column).
        // spacedBy(8.dp) puts an 8dp gap BETWEEN each child (not before the first / after the last).
        verticalArrangement = Arrangement.spacedBy(8.dp),
        // horizontalAlignment controls the CROSS axis (horizontal, for a Column).
        // CenterHorizontally centers each child left-to-right inside the column's width.
        horizontalAlignment = Alignment.CenterHorizontally
    ) {
        Text("Mercury", style = MaterialTheme.typography.titleLarge) // child 1
        Text("Closest to the Sun")                                   // child 2
        Text("88-day year")                                          // child 3
        Button(onClick = { /* no-op for the layout demo */ }) {      // child 4
            Text("Open")
        }
    }
}

Required comment — what changes with spacedBy(24.dp): only the gaps grow. The three Texts and the Button do not resize; Compose inserts 24dp of empty space between each pair of neighbours instead of 8dp, so the children spread further apart and the whole group becomes taller. verticalArrangement distributes space on the column's main axis; it never touches the children's own sizes (that is what a Modifier.size on a child would do).

Why centering needs fillMaxWidth() horizontalAlignment = CenterHorizontally centers children inside the column's width. If the column only wrapped its content, there'd be no extra width to center within — so the fillMaxWidth() is what makes the centering visible.
T2 — a Row that shares width with weightBoth

A Row of three Buttons in a 1 : 2 : 1 width ratio, 8dp apart. Then remove .weight() from the middle button only — predict the widths first.

Where: ComposeCatalog (or ComposeModifiers), run on the emulator.

@Composable
fun T2WeightedRow() {
    Row(
        modifier = Modifier
            .fillMaxWidth()                              // the row spans the full width...
            .padding(16.dp),
        horizontalArrangement = Arrangement.spacedBy(8.dp) // ...minus 8dp gaps between buttons
    ) {
        // weight(n) divides the LEFTOVER horizontal space in proportion to n.
        // 1 : 2 : 1  ->  A gets 1/4, B gets 2/4, C gets 1/4 of the space left after gaps.
        Button(onClick = {}, modifier = Modifier.weight(1f)) { Text("A") }
        Button(onClick = {}, modifier = Modifier.weight(2f)) { Text("B (wide)") }
        Button(onClick = {}, modifier = Modifier.weight(1f)) { Text("C") }
    }
}
Prediction (write this BEFORE running the no-weight version) Remove .weight(2f) from the middle button. Now B has no weight, so Compose measures it at its natural content width (just wide enough for the text "B (wide)" plus the button's minimum size). That width is reserved first; whatever horizontal space is left is then split between the two weight(1f) buttons — equally. So A and C become equal and noticeably wider than before, and B becomes a narrow, text-hugging button in the middle.

Confirm-comment (what actually happens): exactly as predicted — un-weighted children are measured first at their intrinsic size, then weighted children share the remainder. Going from 1:2:1 (A=25%, B=50%, C=25%) to "A & C weighted, B not" makes A and C each grab roughly half of (total − B's content width), so A == C and both grow, while B shrinks to fit its label.

T3 — a Box that layers & alignsBoth

A 140dp-tall full-width Box with a tinted background, a centered title, and a "NEW" badge in the top-end corner. Then move the badge to BottomEnd, add a second in TopStart, and say why Box (not Row/Column) is right for a badge.

Where: ComposeCatalog, run on the emulator.

@Composable
fun T3Banner() {
    Box(
        modifier = Modifier
            .fillMaxWidth()
            .height(140.dp)                              // fixed banner height
            .background(Color(0xFF22314A)),              // tinted backdrop (draw-only)
        contentAlignment = Alignment.Center              // DEFAULT placement for children w/o their own align
    ) {
        // This child has no .align(), so it inherits contentAlignment = Center.
        Text("Featured planet", color = Color.White)

        // .align(...) on a child OVERRIDES the default for THAT child only.
        // Each corner badge is pinned independently because Box children stack on the z-axis.
        Text(
            "NEW",
            color = Color.White,
            modifier = Modifier
                .align(Alignment.BottomEnd)              // moved from TopEnd -> BottomEnd
                .padding(8.dp)
        )
        Text(
            "NEW",
            color = Color.White,
            modifier = Modifier
                .align(Alignment.TopStart)               // second badge, opposite corner
                .padding(8.dp)
        )
    }
}

Required comment — why Box, not Row/Column: a badge has to overlap the banner content and sit in a corner. Row and Column lay their children out sequentially along one axis — children never share the same space — so a badge placed in a Row/Column would push the title aside instead of floating over it. Box stacks children on the z-axis (later children paint on top) and lets each one be aligned independently with Modifier.align(...). That overlap-plus-corner-pinning is exactly what a badge needs.

Paint order: children are drawn in source order, so the last Text in the Box renders on top. If a badge ever disappears behind the title, move it lower in the block (or give it a higher Modifier.zIndex()).

Part B. Modifier order is a feature ★

The same modifiers in a different order produce a different UI, because the chain is applied outside-in: the first modifier wraps the rest.

T4 — padding vs background (predict, then run)Both

Two Texts: one is .background().padding(), the other .padding().background(). Predict which one's green includes the padding and which has a transparent gap to the edge.

Where: prototype in the Playground, reproduce in ComposeModifiers, run.

// A — background FIRST, then padding
Text(
    "Hello",
    modifier = Modifier
        .background(Color(0xFF3DDC84))  // 1) paint the WHOLE area green (outermost)
        .padding(16.dp)                 // 2) THEN inset the text 16dp INSIDE that green
)

// B — padding FIRST, then background
Text(
    "Hello",
    modifier = Modifier
        .padding(16.dp)                 // 1) inset 16dp first -> leaves a transparent margin
        .background(Color(0xFF3DDC84))  // 2) paint green only on the SMALLER inner area
)
Prediction (BEFORE running) A (background → padding): the green is the outer box, and the 16dp of padding lives inside the green — so the green includes the padding (a thick green border around the text). B (padding → background): the 16dp inset is applied first as a transparent margin, then green paints only the inner area hugging the text — so B has a transparent gap to the edge and the green is small.

T5 — clip, border & background togetherBoth

A rounded "avatar": fixed-size Box, clipped to a rounded shape, filled, and outlined so the border follows the rounded corners. Then move .background() before .clip() and describe what changes.

Where: ComposeModifiers, run on the emulator.

@Composable
fun T5Avatar() {
    Box(
        modifier = Modifier
            .size(72.dp)                                       // fixed square (layout)
            .clip(RoundedCornerShape(16.dp))                   // 1) round the corners FIRST...
            .background(Color(0xFFB3E5FC))                     // 2) ...so the fill respects the rounded clip
            .border(2.dp, Color(0xFF0288D1), RoundedCornerShape(16.dp)), // 3) border on the same rounded shape
        contentAlignment = Alignment.Center
    ) {
        Text("JP")
    }
}

Required comment — reorder observation (.background() BEFORE .clip()): clip only affects what is drawn after it in the chain. So with .background().clip(), the fill is painted on the full square before the clip takes effect — the color now reaches into the square corners and you lose the rounded fill (the rounding only clips later content). Keep .clip() first for a clean rounded avatar: clip, then fill, then a border that traces the rounded outline.

T6 — size-affecting vs draw-only modifiersBoth

One Card using at least one size-affecting and one draw-only modifier; label each, and explain why fading with alpha does not move the cards below it, but adding padding does.

Where: ComposeModifiers, run on the emulator.

@Composable
fun T6Card() {
    Card(
        modifier = Modifier
            .fillMaxWidth()                                // size  — changes the MEASURED width
            .padding(16.dp)                                // size  — adds measured space around the card
            .shadow(8.dp, RoundedCornerShape(12.dp))       // draw  — paints a shadow, no size change
            .alpha(0.9f)                                   // draw  — changes opacity only
    ) {
        Column(modifier = Modifier.padding(16.dp)) {       // size  — inner content padding
            Text("Olympus Mons", style = MaterialTheme.typography.titleMedium)
            Text("Tallest volcano in the solar system")
        }
    }
}

Required explanation — alpha vs padding: the two modifiers act in different phases of Compose's frame. alpha is a draw-only modifier: it runs in the draw phase, after measurement is already fixed, so the card keeps the exact same measured box — its neighbours below see no change and don't move; it just paints more transparently. padding is a layout modifier: it runs in the measure phase and adds space to the card's measured size, so the card now occupies more room and everything laid out after it is pushed down. Rule of thumb: if a modifier changes how big the node is, siblings move; if it only changes how it's painted, they don't.

Part C. State & interactivity — real Compose

These need recomposition, so they run in Android Studio only. UI is a function of state: change the state, Compose re-runs the affected composables to redraw.

T7 — a counter (state & recomposition)Android Studio

A Text showing a number and a Button that increments it live. Then explain what happens if you replace the state with a plain var count = 0.

Where: ComposeCatalog, new @Composable + @Preview, run on the emulator.

@Composable
fun Counter() {
    // remember { } caches a value ACROSS recompositions (so it survives re-runs of this function).
    // mutableStateOf(0) makes it OBSERVABLE: reading `count` subscribes this composable, and
    // writing `count` schedules a recomposition. `by` lets us read/write it like a normal var.
    var count by remember { mutableStateOf(0) }            // the single source of truth

    Column(
        modifier = Modifier.padding(16.dp),
        horizontalAlignment = Alignment.CenterHorizontally
    ) {
        Text("Count: $count", style = MaterialTheme.typography.headlineSmall) // re-reads on each recompose
        Spacer(Modifier.height(8.dp))
        Button(onClick = { count++ }) {                    // mutate state -> Compose recomposes -> Text updates
            Text("Add one")
        }
    }
}

Required explanation — the plain var count = 0 version: the number stays at 0 no matter how many times you tap. Two reasons, both fatal: (1) a plain var is not a snapshot State, so mutating it does not notify Compose — no recomposition is ever scheduled, and the Text never re-reads the value. (2) even if something else forced a recomposition, var count = 0 runs again from the top and re-initialises to 0 every time, because there is no remember to preserve it. You need both pieces: remember to survive recomposition, and mutableStateOf to trigger it.

T8 — a real toggle that drives the UI (state hoisting)Android Studio

A Switch whose state does something visible, split into a stateless composable (takes checked + onCheckedChange) and a stateful caller that owns the remembered state.

Where: ComposeCatalog, run on the emulator in both states (on / off).

@Composable
fun ToggleDemo() {                                          // STATEFUL: owns the state
    var on by remember { mutableStateOf(false) }           // the source of truth lives here

    Column(modifier = Modifier.padding(16.dp)) {
        // "state down, event up": pass the value DOWN, pass the change-handler UP.
        ToggleRow(checked = on, onCheckedChange = { on = it })

        // Something VISIBLE driven by the same state: an extra card appears only when ON.
        if (on) {
            Spacer(Modifier.height(12.dp))
            Card(modifier = Modifier.fillMaxWidth()) {
                Text(
                    "Notifications are enabled — you'll get alerts.",
                    modifier = Modifier.padding(16.dp)
                )
            }
        }
    }
}

@Composable
fun ToggleRow(                                              // STATELESS: owns NOTHING, fully reusable
    checked: Boolean,                                      // value handed in from above
    onCheckedChange: (Boolean) -> Unit,                   // event handed back up
    modifier: Modifier = Modifier                          // convention: hoisted modifier, first optional param
) {
    Row(
        modifier = modifier.padding(16.dp),
        verticalAlignment = Alignment.CenterVertically
    ) {
        Text(
            if (checked) "Notifications: ON" else "Notifications: OFF",
            modifier = Modifier.weight(1f)                 // label takes the leftover width...
        )
        Switch(checked = checked, onCheckedChange = onCheckedChange) // ...switch sits at the end
    }
}
Why hoist? ToggleRow holds no state, so it is trivial to @Preview in either position (just pass checked = true or false) and reusable anywhere. The stateful ToggleDemo is the only place that decides where the truth lives — the same Screen-owns-state / Row-is-pure split you saw in NavViewModelState.
T9 — a bound TextFieldAndroid Studio

A text field bound to state, with a live Text below echoing what's typed plus a character count. Explain the typing → recompose loop.

Where: ComposeCatalog, run on the emulator (screenshot mid-typing).

@Composable
fun NameField() {
    var name by remember { mutableStateOf("") }            // holds the current text (source of truth)

    Column(modifier = Modifier.padding(16.dp)) {
        OutlinedTextField(
            value = name,                                  // the field DISPLAYS state (state flows down)
            onValueChange = { name = it },                 // every keystroke writes state (event flows up)
            label = { Text("Your name") },
            singleLine = true
        )
        Spacer(Modifier.height(8.dp))
        // This Text re-reads `name`, so it recomposes on every keystroke alongside the field.
        Text("Hello, ${name.ifBlank { "stranger" }} (${name.length} chars)")
    }
}

Required comment — the data-flow loop: this is unidirectional data flow. You type a key → the field calls onValueChange → that writes the new string into name (state) → because name changed, Compose recomposes everything that reads it → the OutlinedTextField redraws with the new value and the echo Text redraws with the new text and length. The field never stores its own text; it is just a view of name, and the only way the text changes is by sending an event up and letting state flow back down. (Skip the remember and the field appears frozen — your keystrokes have nowhere to be stored.)

// A tiny immutable model. `id` is the stable identity we key favorites by.
data class Planet(val id: Int, val name: String, val blurb: String)

private val planets = listOf(
    Planet(1, "Mercury", "Closest to the Sun · 88-day year"),
    Planet(2, "Venus",   "Hottest planet · thick CO₂ clouds"),
    Planet(3, "Earth",   "The only known living world"),
    Planet(4, "Mars",    "The red planet · home of Olympus Mons"),
    Planet(5, "Jupiter", "Largest planet · the Great Red Spot"),
    Planet(6, "Saturn",  "Ringed gas giant"),

The screen (owns the state)

@Composable
fun PlanetScreen() {
    // STATE THAT CHANGES THE UI: the set of favorited planet ids.
    // We store a Set<Int> in state and replace it with a NEW set on every toggle
    // (assigning a new value is what tells Compose to recompose).
    var favorites by remember { mutableStateOf(setOf<Int>()) }

    Column(
        modifier = Modifier
            .fillMaxWidth()        // outer column spans the width  (size)
            .padding(16.dp)        // ...with a 16dp inset          (size)
    ) {
        // --- Banner Box (from T3): centered title + corner badge showing the live count ---
        Box(
            modifier = Modifier
                .fillMaxWidth()
                .height(120.dp)
                .clip(RoundedCornerShape(16.dp))     // round corners FIRST (so the fill is rounded)
                .background(Color(0xFF22314A)),      // draw-only fill behind the content
            contentAlignment = Alignment.Center      // default placement = centered title
        ) {
            Text(
                "Solar System",
                color = Color.White,
                style = MaterialTheme.typography.headlineSmall
            )
            // Corner badge — pinned top-end, and it RE-READS favorites.size so it updates live.
            // Note the modifier ORDER (Part B): position -> clip -> fill -> inner text padding.
            Text(
                "★ ${favorites.size}",
                color = Color(0xFF0F172A),
                modifier = Modifier
                    .align(Alignment.TopEnd)
                    .padding(8.dp)                          // gap from the corner
                    .clip(RoundedCornerShape(50))           // pill shape
                    .background(Color(0xFFFFD54F))          // gold fill, respects the clip
                    .padding(horizontal = 8.dp, vertical = 4.dp) // text breathing room INSIDE the pill
            )
        }

        Spacer(Modifier.height(12.dp))                      // required Spacer
        HorizontalDivider()                                 // required HorizontalDivider
        Spacer(Modifier.height(4.dp))

        // --- The list: LazyColumn only composes the rows that are visible ---
        LazyColumn(verticalArrangement = Arrangement.spacedBy(8.dp)) {
            items(planets, key = { it.id }) { planet ->     // stable keys help recomposition/scrolling
                PlanetRow(
                    planet = planet,
                    isFavorite = planet.id in favorites,    // value DOWN
                    onToggleFavorite = {                    // event UP
                        // produce a NEW set (don't mutate in place) so Compose sees the change
                        favorites = if (planet.id in favorites)
                            favorites - planet.id
                        else
                            favorites + planet.id
                    }
                )
            }
        }
    }
}

The extracted row (stateless & reusable)

@Composable
fun PlanetRow(
    planet: Planet,
    isFavorite: Boolean,                  // state in (does NOT own it — hoisted to PlanetScreen)
    onToggleFavorite: () -> Unit,         // event out
    modifier: Modifier = Modifier         // convention: hoisted modifier, first optional param
) {
    Card(
        modifier = modifier
            .fillMaxWidth()                              // size-affecting modifier (T6)
            .shadow(4.dp, RoundedCornerShape(12.dp))     // draw-only modifier      (T6)
    ) {
        Row(
            modifier = Modifier
                .fillMaxWidth()
                .padding(16.dp),
            verticalAlignment = Alignment.CenterVertically
        ) {
            // Title + blurb take the leftover width via weight, pushing the star to the end.
            Column(modifier = Modifier.weight(1f)) {
                Text(
                    planet.name,
                    style = MaterialTheme.typography.titleMedium,
                    fontWeight = FontWeight.Bold
                )
                Text(planet.blurb, style = MaterialTheme.typography.bodyMedium)
            }
            // The star is the interactive bit: tapping it sends the event up,
            // which rewrites `favorites`, which recomposes the badge AND this row.
            Text(
                text = if (isFavorite) "★" else "☆",
                style = MaterialTheme.typography.headlineSmall,
                modifier = Modifier
                    .clickable { onToggleFavorite() }     // real click handling (emulator only)
                    .padding(8.dp)                        // larger, comfier tap target
            )
        }
    }
}
