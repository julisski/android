TODO 1 — Sort the reactive query with ORDER BY (unfinished, then high priority, then newest)Android Studio

Add an ORDER BY to the reactive query so the list reads well: unfinished tasks first, then higher priority first, then newest first.

Where: data/TaskDao.kt -> observeTasks() builds on RoomAndPreferences - NoteDao.observeNotes (@Query returning Flow)

    // =========================================================================
    // ★ TODO 1 — Room query (builds on RoomAndPreferences: NoteDao.observeNotes) ★
    //
    // Observe ALL tasks as a reactive Flow. THE KEY ROOM IDEA: a query that returns
    // Flow is "live" — Room watches the "tasks" table and, on every insert/update/
    // delete, re-runs this SELECT and pushes a brand-new List down the Flow, so the
    // UI refreshes with zero extra work.
    //
    // We add an ORDER BY so the rows arrive in a sensible, stable order instead of
    // SQLite's arbitrary insertion order. ORDER BY sorts ROWS, and you may sort by
    // SEVERAL columns separated by commas — SQLite applies them left to right: it
    // sorts by the first column, and only uses the next column to break TIES.
    // =========================================================================
    @Query(
        // SELECT * FROM tasks  -> read every column of every row from the "tasks" table.
        "SELECT * FROM tasks " +
        // ORDER BY (multi-key sort, applied left-to-right; later keys break ties):
        //   done ASC      -> 'done' is stored as 0/1; ASC puts 0 (unfinished) BEFORE
        //                    1 (finished), so open tasks float to the top of the list.
        //   priority DESC -> within the same done-state, sort 2,1,0 (High before
        //                    Normal before Low) — DESC = descending = biggest first.
        //   createdAt DESC-> final tie-breaker: 'createdAt' is epoch millis, so the
        //                    LARGEST value is the most recent => newest task first.
        "ORDER BY done ASC, priority DESC, createdAt DESC"
        // Room CHECKS this SQL at COMPILE time against the Task entity: every name
        // here (tasks, done, priority, createdAt) must really exist as a table/column,
        // or the build fails with a clear Room error pointing at this line. That is
        // Room protecting you — note it's `createdAt`, not `created_at`.
    )
    fun observeTasks(): Flow<List<Task>>   // returns Flow => Room re-emits a fresh, sorted list on every table change

Why it works: Returning Flow<List<Task>> makes the query reactive: Room registers an observer on the "tasks" table and re-runs this exact SELECT (re-applying the ORDER BY) every time a row is inserted, updated, or deleted, so the collected list stays correctly sorted with no manual refresh. ORDER BY is a multi-key sort applied left to right — done ASC floats unfinished (0) above finished (1), priority DESC ranks High(2) over Normal(1) over Low(0) within the same done-state, and createdAt DESC breaks remaining ties by putting the largest epoch-millis (newest) first. The column names (done, priority, createdAt) and table name (tasks) match the Task @Entity exactly, so Room's compile-time SQL verification passes; the imports Query, Flow, and Task are all already present in the starter, so nothing extra is needed.

Common mistake: Writing created_at instead of createdAt (Room verifies column names at compile time — a typo fails the build), or dropping done ASC so finished tasks intermix with open ones.

TODO 2 — Expose Room's Flow as UI stateAndroid Studio

Replace the disconnected MutableStateFlow placeholder so tasks reflects the live Room table — convert the cold Flow into a hot StateFlow the UI can read at any time.

Where: TaskViewModel.kt -> val tasks property builds on RoomAndPreferences - NotesViewModel.notes

// =========================================================================
// ★ TODO 2 — expose Room's Flow as UI state (builds on NotesViewModel.notes) ★
//
// taskDao.observeTasks() is a COLD Flow<List<Task>>: it does no work until
// someone collects it, and it has no "current value" to read synchronously.
// Compose needs a value it can read RIGHT NOW on every recomposition, so we
// convert the cold Flow into a HOT StateFlow with stateIn(...).
// =========================================================================
val tasks: StateFlow<List<Task>> =                  // the screen collects this; it ALWAYS has a value
    taskDao.observeTasks()                          // COLD Flow<List<Task>>; Room re-emits on every table change
        .stateIn(                                   // turn that cold Flow into a hot, cached StateFlow
            scope = viewModelScope,                 // collection lives exactly as long as this ViewModel (survives rotation)
            started = SharingStarted.WhileSubscribed(5_000),  // stay subscribed to Room 5s after the last UI collector leaves (survives a rotation gap without re-querying)
            initialValue = emptyList(),             // value to expose BEFORE Room's first emit, so the UI never sees null
        )

// NOTE: now that `tasks` is wired to observeTasks(), the MutableStateFlow
// import at the top of the file is unused. Remove it (Alt+Enter / Optimize
// Imports) so the finished file compiles with no leftover warnings.

Why it works: stateIn collects the cold Flow inside viewModelScope and republishes its latest emission as a StateFlow, which always holds a readable current value for Compose. Because Room's observeTasks() re-emits the whole list on every insert/update/delete, the UI refreshes automatically with no manual reload. WhileSubscribed(5_000) keeps the collection alive for 5 seconds after the last collector unsubscribes, so a configuration change (rotation) reuses the cached list instead of re-running the query, and the initialValue of emptyList() means the list renders immediately rather than waiting for the first DB read.

Common mistake: Leaving the placeholder MutableStateFlow(emptyList()) in place — it compiles and shows an empty list but is never connected to Room, so nothing you add ever appears. Also: forgetting to delete the now-unused MutableStateFlow import.

TODO 3 — Save a task (insert OR update)Android Studio

Implement save(task) so a new task (id == 0L) is inserted and an existing one is updated — both as suspend Room writes on a coroutine.

Where: TaskViewModel.kt -> fun save(task: Task) builds on RoomAndPreferences - @Insert/@Update

// =========================================================================
// ★ TODO 3 — save a task (insert when new, update when editing) ★
//
// The edit screen hands us a Task. A NEW task still has its default id == 0L;
// an EXISTING (already-saved) task carries a real, non-zero primary key. That
// single fact tells us which DAO write to call. Both writes are `suspend`, so
// they MUST run inside a coroutine — viewModelScope.launch starts one.
// =========================================================================
fun save(task: Task) {
    viewModelScope.launch {                         // launch a coroutine tied to the ViewModel's lifetime (can call suspend fns; auto-cancels on clear)
        if (task.id == 0L) taskDao.insert(task)     // id still 0 => brand-new row; @Insert lets SQLite autoGenerate the id
        else               taskDao.update(task)     // id non-zero => existing row; @Update overwrites it by primary key
    }                                               // no manual UI refresh: the write changes the table, so observeTasks() re-emits and `tasks` updates itself
}

Why it works: taskDao.insert and taskDao.update are suspend functions (Room runs disk I/O off the main thread), so they can only be called from a coroutine — viewModelScope.launch provides one whose lifetime is bound to the ViewModel, so the work is cancelled automatically if the ViewModel is cleared. Checking task.id == 0L distinguishes a never-saved task (the @PrimaryKey(autoGenerate=true) default) from an edit of an existing row, routing to INSERT vs UPDATE. After either write the tasks table changes, which makes the reactive observeTasks() Flow re-emit, so the StateFlow from TODO 2 updates the UI with no extra code.

Common mistake: Always calling insert — then editing a task creates a duplicate row instead of updating it. Or calling the suspend DAO method directly (outside viewModelScope.launch), which won't compile.

TODO 4 — Delete + toggle doneAndroid Studio

Implement delete(task) and toggleDone(task) as suspend Room writes launched on viewModelScope; toggling persists the same task with its done flag flipped.

Where: TaskViewModel.kt -> fun delete(task) and fun toggleDone(task) builds on RoomAndPreferences - NotesViewModel.toggleDone

// =========================================================================
// ★ TODO 4 — delete a task, and toggle its "done" flag ★
//
// Both are `suspend` Room writes, so each one is wrapped in its own
// viewModelScope.launch { } coroutine. toggleDone doesn't need a new query:
// because Task is a data class, copy() gives us an identical row with just the
// `done` field flipped, which @Update then persists by primary key.
// =========================================================================
fun delete(task: Task) {
    viewModelScope.launch {                         // suspend write => must run in a coroutine
        taskDao.delete(task)                        // @Delete removes the row matched by its @PrimaryKey id
    }                                               // table changed => observeTasks() re-emits => the row disappears from the list automatically
}

fun toggleDone(task: Task) {
    viewModelScope.launch {                         // suspend write => must run in a coroutine
        taskDao.update(task.copy(done = !task.done))// copy() clones the task changing ONLY done (flip it); @Update saves it by primary key
    }                                               // same id => Room updates the existing row in place, and the reactive Flow refreshes the checkbox
}

Why it works: delete and update are suspend Room writes, so each is launched on viewModelScope to run off the main thread within the ViewModel's lifecycle. toggleDone relies on Task being a data class: copy(done = !task.done) produces a new instance identical to the original except for the flipped flag, and because it keeps the same primary key id, @Update overwrites that exact row rather than creating a new one. Both operations mutate the tasks table, so the reactive observeTasks() Flow re-emits and the UI (the list and the row's checkbox) updates itself without any manual refresh.

Common mistake: Flipping task.done in memory and expecting it to stick — Task is immutable, and nothing is persisted until you call update(...). Use task.copy(done = !task.done) inside viewModelScope.launch.

TODO 5 — Build the task list + an "add" controlAndroid Studio

Build TaskListScreen: an add control that calls onAddTask(), the list — a LazyColumn of TaskRows — and a friendly empty state when there are no tasks.

Where: MainActivity.kt -> TaskListScreen() builds on RoomAndPreferences list + hw5 Box/align (FAB)

@Composable
fun TaskListScreen(
    tasks: List<Task>,
    onAddTask: () -> Unit,
    onOpenTask: (Task) -> Unit,
    onToggleDone: (Task) -> Unit,
    onDelete: (Task) -> Unit,
    modifier: Modifier = Modifier,
) {
    // Box lets two children OVERLAP in the same space (z-order), so the scrolling
    // list and the "add" button can share the screen — the FAB floats ON TOP of
    // the list. This is the same hw5 "align a small thing in a corner" trick.
    // (Plain alternative: a Column with Button("New task") on top of the LazyColumn.)
    Box(modifier = modifier.fillMaxSize()) {                    // fill the whole content area
        if (tasks.isEmpty()) {
            // EMPTY STATE: with no rows a LazyColumn would render a blank screen,
            // which looks broken. Show a friendly message centered in the Box instead.
            Text(
                text = "No tasks yet.\nTap ＋ to add your first task.", // gentle hint, not a dead-end
                style = MaterialTheme.typography.bodyLarge,         // readable body text from the theme
                modifier = Modifier.align(Alignment.Center),       // center it inside the Box
            )
        } else {
            // THE LIST: LazyColumn only composes the rows currently on screen, so a
            // 10,000-task list stays fast (unlike a plain Column, which builds them all).
            LazyColumn(
                modifier = Modifier.fillMaxSize(),                 // take all the room so it scrolls
                // contentPadding reserves blank space at the very bottom of the SCROLL
                // content so the floating button can't sit on top of the last row.
                // NOTE: PaddingValues is the one extra import this FAB layout needs:
                //   import androidx.compose.foundation.layout.PaddingValues
                contentPadding = PaddingValues(bottom = 88.dp),    // ~FAB height + margin
            ) {
                // items() emits one composable per element. key = { it.id } gives each
                // row a STABLE identity, so when the list reorders/changes (reactive Room
                // Flow) Compose moves/keeps the right rows instead of rebuilding everything.
                items(tasks, key = { it.id }) { task ->
                    // TaskRow is stateless: we pass it the data plus lambdas that close
                    // over THIS task, translating its no-arg callbacks back to (Task) ones.
                    TaskRow(
                        task = task,
                        onToggleDone = { onToggleDone(task) },     // checkbox flipped -> Room UPDATE
                        onOpen = { onOpenTask(task) },             // row tapped -> navigate to edit
                        onDelete = { onDelete(task) },             // delete tapped -> Room DELETE
                    )
                    HorizontalDivider()                            // thin line between rows for readability
                }
            }
        }
        // THE ADD CONTROL: a round Material "＋" button pinned bottom-right.
        // align() places it in the BottomEnd corner of the Box; padding keeps it
        // off the screen edge. Tapping it asks the host to navigate (TODO 8a).
        FloatingActionButton(
            onClick = onAddTask,                                   // report intent; the host navigates
            modifier = Modifier
                .align(Alignment.BottomEnd)                        // corner placement (the hw5 dot trick)
                .padding(16.dp),                                   // breathing room from the edges
        ) {
            Text("＋")                                             // the universal "add" glyph
        }
    }
}

Why it works: A LazyColumn renders only the visible rows, and items(tasks, key = { it.id }) gives each row a stable identity so the screen diffs correctly when Room's reactive Flow emits a new ordering. The screen is stateless — it just draws the tasks it's given and reports intents (onAddTask/onOpenTask/etc.) up to the navigation host. Box + Modifier.align(Alignment.BottomEnd) overlays the FAB on the list, and contentPadding(bottom = 88.dp) keeps the floating button from covering the last row. The only symbol beyond the starter imports is PaddingValues (androidx.compose.foundation.layout.PaddingValues).

Common mistake: Using a plain Column for the list (it builds every row, no recycling) instead of LazyColumn; omitting key = { it.id } (rows lose identity on re-sort); or a FAB that covers the last row (needs contentPadding).

TODO 6 — Build one task rowAndroid Studio

Build TaskRow: a Checkbox, a weighted title + notes column that opens the editor when tapped, a small priority label, and a delete button.

Where: MainActivity.kt -> TaskRow() builds on hw5 settings row — Row + Modifier.weight(1f)

@Composable
fun TaskRow(
    task: Task,
    onToggleDone: () -> Unit,
    onOpen: () -> Unit,
    onDelete: () -> Unit,
) {
    // A Row lays its children out left-to-right. CenterVertically lines the checkbox,
    // text, and buttons up on a single baseline; spacedBy(8.dp) puts an even gap
    // between every child so we don't pepper each one with manual padding.
    Row(
        modifier = Modifier
            .fillMaxWidth()                                        // span the list's full width
            .padding(horizontal = 16.dp, vertical = 8.dp),        // inset the row from the edges
        verticalAlignment = Alignment.CenterVertically,           // vertically center all children
        horizontalArrangement = Arrangement.spacedBy(8.dp),       // even 8.dp gaps between children
    ) {
        // The "done" control. checked reflects the CURRENT task.done; onCheckedChange
        // fires when tapped. We ignore the new boolean and just call onToggleDone(),
        // because the real source of truth is Room — the ViewModel flips it and the
        // updated list flows back down (we don't keep local checkbox state here).
        Checkbox(
            checked = task.done,                                  // mirror the persisted value
            onCheckedChange = { onToggleDone() },                 // tap -> ask the host to UPDATE Room
        )
        // The weighted Column is the KEY to this layout: weight(1f) tells it to take
        // ALL the leftover horizontal space, which shoves the priority label and the
        // delete button to the far right. clickable makes tapping the text open the
        // editor (the checkbox and buttons keep their own taps). weight is a RowScope
        // extension, so it's available here without an extra import.
        Column(
            modifier = Modifier
                .weight(1f)                                       // eat the remaining width -> pushes siblings right
                .clickable { onOpen() },                          // tap the text body -> navigate to edit
        ) {
            Text(
                text = task.title,                                // the task's headline
                style = MaterialTheme.typography.titleMedium,     // a slightly larger "title" style
            )
            // Only show the notes line when there is something to show — a blank
            // notes string would just add an empty gap under the title.
            if (task.notes.isNotBlank()) {                        // skip the line entirely if empty
                Text(
                    text = task.notes,                            // the longer detail text
                    style = MaterialTheme.typography.bodyMedium,  // smaller "body" style for secondary text
                )
            }
        }
        // Turn the stored priority Int into human text. else covers 0 (Low) AND any
        // unexpected value, so the when is exhaustive without a separate branch.
        Text(
            text = when (task.priority) {                         // map 0/1/2 -> words
                2 -> "High"                                       // 2 = High
                1 -> "Normal"                                     // 1 = Normal
                else -> "Low"                                     // 0 (or anything) = Low
            },
            style = MaterialTheme.typography.labelMedium,         // small label-sized text
        )
        // A bordered (secondary) button to remove the task. onClick reports the intent;
        // the ViewModel performs the Room DELETE and the list flows back without it.
        OutlinedButton(onClick = onDelete) {                      // bordered = less prominent than the FAB
            Text("Delete")                                        // the button's label
        }
    }
}

Why it works: Modifier.weight(1f) on the middle Column makes it absorb all leftover horizontal space, which pushes the priority label and Delete button to the right edge — the same weight technique from the hw5 settings row, and weight is a RowScope member so no extra import is needed. The row is stateless: the Checkbox shows task.done and just forwards the tap to onToggleDone() because Room (via the ViewModel) is the single source of truth, so the new state flows back down rather than being held locally. The when on task.priority uses else to cover 0 and any stray value, making it exhaustive.

Common mistake: Forgetting Modifier.weight(1f) on the middle Column, so the Delete button never reaches the right edge; or making the whole Row clickable so tapping the checkbox or Delete also opens the editor.

TODO 7 — Build the add/edit formAndroid Studio

Build TaskEditScreen: local form state seeded from existing, OutlinedTextFields for title + notes, a priority control, a done checkbox, and Save / Cancel.

Where: MainActivity.kt -> TaskEditScreen() builds on RoomAndPreferences inputs + hw5 remembered state

@Composable
fun TaskEditScreen(
    existing: Task?,
    onSave: (Task) -> Unit,
    onCancel: () -> Unit,
    modifier: Modifier = Modifier,
) {
    // LOCAL FORM STATE — this is the ONE screen that owns editable state. Each field
    // is a remembered MutableState seeded from `existing` (or a blank default for a
    // new task). KEYING each remember on `existing` is important: if the task arrives
    // a moment later (e.g. the app was killed and reopened straight on this screen),
    // the key change re-runs the lambda and RE-SEEDS the field — without the key,
    // remember would cache its first (blank) value and the form would stay empty.
    var title by remember(existing) { mutableStateOf(existing?.title ?: "") }      // text field state
    var notes by remember(existing) { mutableStateOf(existing?.notes ?: "") }      // text field state
    var done by remember(existing) { mutableStateOf(existing?.done ?: false) }     // checkbox state
    var priority by remember(existing) { mutableStateOf(existing?.priority ?: 1) } // priority state (default Normal)

    Column(modifier = modifier.fillMaxSize().padding(16.dp)) {  // vertical form, inset from edges
        Text(
            text = if (existing == null) "New task" else "Edit task", // header reflects add-vs-edit
            style = MaterialTheme.typography.headlineSmall,    // prominent screen title
        )
        Spacer(Modifier.height(12.dp))                         // gap below the header

        // TITLE input. value is the current state; onValueChange writes every keystroke
        // back into `title`, which recomposes the field to show what the user typed.
        OutlinedTextField(
            value = title,                                     // show current state
            onValueChange = { title = it },                    // each keystroke updates state
            label = { Text("Title") },                         // floating field label
            modifier = Modifier.fillMaxWidth(),                // span the form width
        )
        Spacer(Modifier.height(8.dp))                          // gap between fields

        // NOTES input — same pattern, bound to `notes`.
        OutlinedTextField(
            value = notes,                                     // show current state
            onValueChange = { notes = it },                    // each keystroke updates state
            label = { Text("Notes") },                         // floating field label
            modifier = Modifier.fillMaxWidth(),                // span the form width
        )
        Spacer(Modifier.height(8.dp))                          // gap before the priority control

        // PRIORITY control — ONE self-labelling button that both SHOWS the value and
        // CYCLES it. (priority + 1) % 3 walks 0 -> 1 -> 2 -> 0... and the same when()
        // mapping as TaskRow turns the number into the word on the button.
        Button(onClick = { priority = (priority + 1) % 3 }) {  // tap cycles Low->Normal->High->Low
            Text(
                "Priority: " + when (priority) {               // label reflects the current value
                    2 -> "High"                                // 2 = High
                    1 -> "Normal"                              // 1 = Normal
                    else -> "Low"                              // 0 = Low
                }
            )
        }
        Spacer(Modifier.height(8.dp))                          // gap before the done row

        // DONE control — a checkbox plus a label, lined up in a Row.
        Row(verticalAlignment = Alignment.CenterVertically) {  // center the box and its label
            Checkbox(checked = done, onCheckedChange = { done = it }) // bind to local `done` state
            Text("Done")                                       // label for the checkbox
        }
        Spacer(Modifier.height(20.dp))                         // gap before the action buttons

        // ACTION buttons in a Row with even spacing.
        Row(horizontalArrangement = Arrangement.spacedBy(8.dp)) { // 8.dp gap between buttons
            // SAVE — build a Task from the form and hand it up to be persisted.
            // PRESERVING id and createdAt is what makes Room UPDATE the SAME row when
            // editing: a non-zero id means "update this row"; id = 0 (new task) means
            // "INSERT and assign me an id". trim() drops accidental leading/trailing spaces.
            Button(
                onClick = {
                    onSave(
                        Task(
                            id = existing?.id ?: 0L,                        // 0 -> INSERT, non-zero -> UPDATE
                            title = title.trim(),                          // clean up whitespace
                            notes = notes.trim(),                          // clean up whitespace
                            done = done,                                   // current checkbox value
                            priority = priority,                           // current priority value
                            createdAt = existing?.createdAt                 // keep original timestamp when editing
                                ?: System.currentTimeMillis(),             // ...or stamp now for a new task
                        )
                    )
                }
            ) {
                Text("Save")                                   // primary action label
            }
            // CANCEL — leave without saving. The host pops the back stack (TODO 8c).
            OutlinedButton(onClick = onCancel) {               // bordered = secondary action
                Text("Cancel")                                 // label
            }
        }
    }
}

Why it works: This is the one screen that owns editable state, so each field is a remembered MutableState; keying remember(existing) re-seeds the fields if the task loads after first composition (e.g. process death + reopen straight on this screen), which a plain remember would cache as blank. Save preserves existing?.id and existing?.createdAt so Room performs an UPDATE on the same row when editing (id != 0) but an INSERT for a new task (id = 0), avoiding duplicate rows and keeping the original creation time. The single cycling Button uses (priority + 1) % 3 to walk 0->1->2->0 and reuses the same when() mapping as TaskRow. Every symbol used (OutlinedTextField, Button, Checkbox, Spacer, remember, mutableStateOf, System.currentTimeMillis) is already imported in the starter.

Common mistake: A plain remember { … } (not keyed on existing) leaves the form blank if the task loads a moment later; and dropping id/createdAt on Save makes an edit insert a duplicate instead of updating the same row.

                )
            }

            // When a TaskEditKey is on top, show the add/edit form for that key.
            entry<TaskEditKey> { key ->
                // Resolve WHICH task we're editing. A null taskId means "new task", so
                // `existing` stays null and the form starts blank. Otherwise we find the
                // task in the already-loaded list (no extra DB round-trip needed).
                val existing: Task? = key.taskId?.let { id -> tasks.find { it.id == id } }
                TaskEditScreen(
                    existing = existing,
                    onSave = { task ->
                        viewModel.save(task)                    // persist via Room FIRST (TODO 3)
                        // TODO 8c — BACK: pop the edit screen off the stack AFTER saving so
                        // the top key becomes TaskListKey again. removeLastOrNull() is the
                        // safe pop (no crash if the stack is somehow empty). The list updates
                        // itself because Room's reactive Flow re-emits — no manual refresh.
                        backStack.removeLastOrNull()
                    },
                    onCancel = {
                        // TODO 8c — BACK (cancel path): same pop, but WITHOUT saving, so
                        // the form's edits are simply discarded and we return to the list.
                        backStack.removeLastOrNull()
                    },
                )
            }
        }

Why it works: In Nav3 the back stack is the single source of navigation truth: backStack.add(key) pushes a new top key (navigating forward) and backStack.removeLastOrNull() pops it (navigating back, safely no-op if empty), and NavDisplay re-renders whenever the top key changes. Passing TaskEditKey(taskId = null) vs TaskEditKey(taskId = task.id) is how the SAME edit screen serves both add and edit — the key carries the argument that selects blank-vs-prefilled. onSave calls viewModel.save(task) BEFORE popping so the write is issued first; the list then refreshes on its own because Room emits a new list through its Flow, so popping back to TaskListKey shows the change with no manual refresh.

Common mistake: Popping before saving (or forgetting to pop, so the form stays up); pushing a fresh TaskListKey to "go back" instead of removeLastOrNull(); or stuffing the whole Task into the key instead of just its id.
