▸ How to use this each class

  • Duplicate LAST week's filled-in copy and rename it (e.g. “Logbook — Class 08”). You fill this by DIFFING against last week, so keep the old one open beside it.
  • Fill top-to-bottom in 10–15 minutes. Phrases, checkmarks, and a sketch — not paragraphs. If a section takes more than ~2 minutes, write less.
  • Each section has ONE job, so you only type things once: the Diff is WHAT changed · the Design Log is WHY · the Plan is NEXT. Carried-over items copy verbatim — don't reword them.
  • Paste a real screenshot / emulator capture where you can; a quick boxed wireframe is fine otherwise. Always note light/dark and which UI state is showing.
  • First class, or missed a week? Skip Burndown and the Diff's ~/- lines — list what exists now as all “+” lines and just fill the Plan. You'll diff for real next week.
  • Write in Compose vocabulary on purpose (NavKey, ViewModel, StateFlow, sealed UI state, modifier order) — filling this page is your weekly review of the course.

🚀 The Diff — what shipped since last class

One headline plus the delta only, in git-diff style. The markers structurally stop you from re-listing old work.

  • Top line — “Shipped this week”: in ONE sentence, what a classmate reopening your app would notice is new.
  • Then 3–6 delta lines, each prefixed: + added · ~ changed existing · − removed/redid (just like a git diff).
  • The WHAT only — save every “why” for the Design Log so you don't type it twice.
Example Shipped: the ride-detail screen is live. + Added RideDetailScreen (NavKey RideDetail(rideId)). + Added RideRepository emitting a sealed RideUiState. ~ RideListViewModel now exposes StateFlow<RideUiState> instead of a raw List. − Removed the hard-coded sampleRides list from the Composable.
 

🧠 Design decision log

The design steps and the WHY behind your choices — the trade-offs you made on purpose. At least one UI/layout decision AND one architecture/state decision each week.

  • Log each real decision. The Why must name a concrete force — rotation, screen size, scannability, testability, back-stack depth, recomposition cost. “It's cleaner” is not a Why.
  • If you can't name what you gave up in the Trade-off column, you didn't make a decision — you just coded. Write “see Design Log” elsewhere instead of repeating it.
Example UI — Gave RideDetail its own Nav3 screen | Why: a map + notes + driver card is too much for a list row | Rejected: an inline-expanding Card (it pushed siblings off-screen in the Playground) | Trade-off: one extra back-stack entry, but the list stays scannable.
