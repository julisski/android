Storage, one element at a time

How an item reaches storage

Most storage demos show you a finished app and leave you to guess how the data got onto disk. Storage Journey does the opposite: it follows a single element — one line of text you type — through every layer it passes on the way into the database, and back to the screen. This page is that trip as blocks of real code, in order, from the build wiring that switches storage on to the moment the app is uninstalled.

Room · SQLite Jetpack Compose ViewModel + Flow Kotlin 2.2.10 heavily commented
The app mirrors this page The running app draws a Journey panel: each time you save an element it lists the numbered, color-coded steps the data just took (UI → ViewModel → Repository → Room → Flow), with the time since the trip began. The colors on this page match that panel, so the lesson and the app read as one thing. The trace itself lives in JourneyLog.kt.

The one sentence to remember

You never update the list by hand. You write once (an insert), and because the read side is a reactive Flow, Room re-runs the query and pushes the new data back to the screen for you. The whole project exists to make that single round trip impossible to miss.

Initialization · before any data moves

Step 0 — turning storage on, and launching

Two things must be true before an element can travel anywhere: the build must wire up Room (so the SQLite code is generated), and the app must start and build the database once. This is the initialization end of the journey.

Build wiring: KSP generates the SQLite code

Room is annotations only; a code generator turns your @Entity/@Dao/@Database into real SQLite classes at build time. That generator runs through KSP, so the plugin and the ksp(...) dependency are not optional — without them the app compiles but crashes at first use.

build.gradle.kts
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.compose)
    // KSP (Kotlin Symbol Processing) — REQUIRED so Room's annotation processor runs during
    // assembleDebug and GENERATES the concrete ItemDatabase_Impl / ItemDao_Impl classes.
    alias(libs.plugins.ksp)
}

dependencies {
    // --- THE STORAGE COMPONENT (the whole point of this project) ---
    // Room: the SQLite-backed store our one StoredItem row travels into. This single
    // app deliberately uses ONLY Room (no DataStore/files).
    implementation(libs.androidx.room.runtime)
    implementation(libs.androidx.room.ktx)
    // The Room compiler is wired through KSP (NOT implementation). This is the line that
    // makes KSP generate the SQLite-backed implementation of @Dao / @Database at build time.
    ksp(libs.androidx.room.compiler)
}
One more required flag AGP 9's built-in Kotlin normally forbids adding sources via the kotlin.sourceSets DSL — which is exactly how KSP registers its generated output. gradle.properties sets android.disallowKotlinSourceSets=false so the generated Room code is allowed in.

Launch: build the screen, lazily build the store

When the launcher opens MainActivity, onCreate sets the Compose content and asks for the ViewModel. The database itself is not built here — it is created on first use inside the ViewModel (next), so startup stays cheap.

MainActivity.kt · onCreate
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)                     // always call through to the framework first
    enableEdgeToEdge()                                     // draw under the system bars for a modern look
    setContent {
        // Obtain the ViewModel (constructed via its Application factory). viewModel()
        // remembers it across recompositions and scopes it to this Activity.
        val viewModel: StorageJourneyViewModel = viewModel(
            factory = StorageJourneyViewModel.factory(application)
        )
        StorageJourneyTheme(dynamicColor = false) {
            Scaffold(modifier = Modifier.fillMaxSize()) { innerPadding ->
                JourneyRoute(
                    viewModel = viewModel,
                    modifier = Modifier.padding(innerPadding),
                )
            }
        }
    }
}

Step 1 · the thing that travels

The element — one row, described as a class

The "element" is a StoredItem. A Room @Entity is a Kotlin class that Room turns into a SQLite table at compile time; each property becomes a column, and each instance becomes a row. This is the payload that will travel all the way to disk.

data/StoredItem.kt
@Entity(tableName = "items")                 // <-- THE concept: this class IS a table named "items"
data class StoredItem(
    @PrimaryKey(autoGenerate = true)         // <-- unique row id; SQLite auto-increments it
    val id: Long = 0,                        // default 0 => "let Room/SQLite pick the id on insert"
    val label: String,                       // becomes a TEXT column "label" (the user's text)
    val createdAt: Long,                     // becomes an INTEGER column "createdAt" (epoch millis)
)
  • id = 0 + autoGenerate = true means "SQLite, assign the next id on insert." After the row lands, the real id comes back (you will see it as #7 in the saved list).
  • label is the text you typed — the actual cargo of the journey.
  • createdAt is stamped by the ViewModel, so the row carries its own "saved at" moment.

Step 2 · the destination

The storage component — the Room database

This abstract class is the single connection to the SQLite file on disk (journey.db). It ties the entity to the DAO, and Room.databaseBuilder(...) brings it to life. It is abstract because Room generates the real subclass (ItemDatabase_Impl) at build time — that is the KSP step from before paying off.

data/ItemDatabase.kt
@Database(
    entities = [StoredItem::class],         // <-- every table in this DB; here just StoredItem
    version = 1,                            // schema version; increment when columns change
    exportSchema = false,
)
abstract class ItemDatabase : RoomDatabase() {

    abstract fun itemDao(): ItemDao         // <-- Room fills this in with ItemDao_Impl

    companion object {
        @Volatile private var INSTANCE: ItemDatabase? = null

        // Return the shared database, building it once on first use (double-checked locking
        // so two threads racing on startup can't each build their own database object for the same file).
        fun getInstance(context: Context): ItemDatabase =
            INSTANCE ?: synchronized(this) {
                INSTANCE ?: Room.databaseBuilder(
                    context.applicationContext,             // applicationContext => no Activity leak
                    ItemDatabase::class.java,               // the database class to implement
                    "journey.db",                           // the SQLite FILE name on disk
                ).build().also { INSTANCE = it }            // cache the instance for next time
            }
    }
}
Why a singleton Opening two databases over the same file is wasteful and can corrupt state. The companion object caches exactly one instance for the whole process — the ViewModel just asks ItemDatabase.getInstance(app) and always gets the same store.

Step 3 · the door

The DAO — the read/write surface

A DAO ("Data Access Object") is the set of methods that read and write the table. You describe it as an interface; Room generates the implementation. Two methods matter for the journey: the write (insert) and the reactive read (observeItems).

data/ItemDao.kt
@Dao
interface ItemDao {

    // THE RETURN TRIP: a query that returns Flow is LIVE. Room watches the "items" table and
    // re-runs this SELECT on every change, pushing a fresh List down the stream.
    @Query("SELECT * FROM items ORDER BY id DESC")        // <-- reactive read; re-emits on every change
    fun observeItems(): Flow<List<StoredItem>>

    // THE WRITE: suspend because it touches disk. Returns the new row's auto-generated id.
    @Insert
    suspend fun insert(item: StoredItem): Long            // <-- disk write => suspend; returns the new id

    @Delete
    suspend fun delete(item: StoredItem)                  // <-- disk write => suspend

    @Query("DELETE FROM items")
    suspend fun clearAll()                                // <-- disk write => suspend
}
⬇ The write

insert is suspend: Room runs the disk write on a background thread for you, so you call it from a coroutine and never freeze the UI. It returns the new id straight from SQLite.

⬆ The reactive read

observeItems() returns Flow<List<StoredItem>> — a live stream. You subscribe once; Room re-emits the whole list every time the table changes. No manual "reload" call exists anywhere in this app.

Step 4 · the boundary

The repository — the storage component's front door

The repository is the one named place the rest of the app talks to for persistence. The ViewModel never touches the DAO directly. It is also the honest vantage point to narrate the write: right here we watch the call go into Room and the new id come back, so this is where the Repository and Room legs of the journey are recorded.

data/ItemRepository.kt
// The live list of saved items, straight from Room's reactive query.
val items: Flow<List<StoredItem>> = dao.observeItems()

suspend fun save(item: StoredItem): Long {
    journey.record(
        JourneyStage.REPOSITORY,
        "Repository.save(...) handed the element to the engine: dao.insert(item) — a " +
            "suspend write Room runs off the main thread.",
    )
    val newId = dao.insert(item)                       // <-- THE write; suspends until SQLite commits
    journey.record(
        JourneyStage.ROOM,
        "Room executed INSERT INTO items(...) and SQLite committed the row to " +
            "journey.db as id #$newId. The element is now on disk.",
    )
    return newId
}
Why suspend ripples up Because dao.insert is suspend, so is save, so the ViewModel must call it inside a coroutine. That is how the disk write stays off the main thread without anyone naming a background thread — the type system enforces it.

Step 5 · the conductor

The ViewModel — wiring, and the write path as prose

The ViewModel turns Room's Flow into UI state and owns the write action. The items property below is the return trip; addItem() is the write path — read it top-to-bottom and you are reading steps 1→4 in order.

The return trip: Flow → StateFlow (and record the Flow leg)

StorageJourneyViewModel.kt · items
val items: StateFlow<List<StoredItem>> = repository.items
    .onEach { list ->
        journeyLog.record(
            JourneyStage.FLOW,
            "Room re-ran SELECT * FROM items, found ${list.size} row(s), and pushed a " +
                "new List down the Flow. collectAsStateWithLifecycle turns that into a recomposition.",
        )
    }
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())
  • .onEach { ... } runs once when the screen first subscribes (Room reads what is already on disk) and again after every write — that is exactly when we record the green Flow leg.
  • .stateIn(...) converts the cold Flow into a hot StateFlow that always has a value and is shared across collectors. WhileSubscribed(5_000) keeps it alive 5s after the last collector leaves, so a quick rotation does not re-query.

The write path: addItem() is the journey

StorageJourneyViewModel.kt · addItem
fun addItem(rawLabel: String) {
    val label = rawLabel.trim()
    if (label.isEmpty()) return                            // ignore empty submissions

    journeyLog.reset()                                     // 1. fresh trace for this trip
    journeyLog.record(
        JourneyStage.UI,
        "The Save button captured \"$label\" from the OutlinedTextField and called " +
            "viewModel.addItem(\"$label\").",
    )

    // 3. Stamp metadata in the ViewModel. id stays 0 so SQLite assigns the real key on insert.
    val element = StoredItem(label = label, createdAt = System.currentTimeMillis())
    journeyLog.record(
        JourneyStage.VIEWMODEL,
        "ViewModel built StoredItem(label=\"$label\", createdAt=…) and launched a " +
            "coroutine on viewModelScope so the suspend write never blocks the UI thread.",
    )

    viewModelScope.launch {                                // coroutine -> safe to call suspend save
        repository.save(element)                           // records Repository + Room; then Flow re-emits
    }
}
Steps 1–4 are literally here The UI and ViewModel legs are recorded inline; the coroutine then calls repository.save, which records the Repository and Room legs; and the items Flow above records the Flow leg when Room re-emits. Five files, one continuous trip.

Step 6 · where it starts and ends on screen

The screen — capture the text, render the result

The journey begins with a tap and ends with a recomposition, both here. The stateful wrapper collects the two flows; the Save button captures the text and kicks off addItem.

Collect the state (the return trip arrives here)

MainActivity.kt · JourneyRoute
val items by viewModel.items.collectAsStateWithLifecycle()       // live rows from Room
val journey by viewModel.journey.collectAsStateWithLifecycle()   // the recorded trip

collectAsStateWithLifecycle() subscribes to a StateFlow, wraps its latest value in Compose State, and recomposes whatever reads it — pausing while the screen is off-screen. A new emission flips the state, and only the parts that read it redraw.

Capture the text and launch the write

MainActivity.kt · the input + Save button
OutlinedTextField(
    value = label,
    onValueChange = { label = it },                    // local UI state only
    label = { Text("Text to store") },
    singleLine = true,
    modifier = Modifier.fillMaxWidth(),
)
Spacer(Modifier.height(8.dp))
Button(
    onClick = {
        onSave(label)                                  // -> ViewModel.addItem -> Room insert
        label = ""                                     // clear the local input field
    },
    modifier = Modifier.fillMaxWidth(),
) {
    Text("Save to storage")
}

Render the trip (the Journey panel)

The recorded steps are drawn with forEachIndexed, so the list position gives the step number — the same numbers you see on this page.

MainActivity.kt · JourneyPanel
journey.forEachIndexed { index, step ->
    JourneyStepRow(number = index + 1, step = step)
    if (index != journey.lastIndex) Spacer(Modifier.height(8.dp))
}

All together

The full trip, one tap at a time

Step through the exact chain the code follows when you save the word "milk". This is the same sequence the in-app Journey panel prints, plus the final recomposition.

Scenario: you type “milk” and tap Save to storage
1 / 6
  1. 1
    UI. The Button's onClick reads the text field and calls onSave("milk"), which is viewModel::addItem.MainActivity.kt · Button(onClick = { onSave(label) })
  2. 2
    ViewModel. addItem trims the text, resets the trace, builds StoredItem(label="milk", createdAt=now), and launches a coroutine on viewModelScope.StorageJourneyViewModel.kt · addItem()
  3. 3
    Repository. Inside the coroutine, repository.save(item) calls the suspend dao.insert(item) — the storage boundary handing the element to the engine.ItemRepository.kt · save()
  4. 4
    Room · disk. Room runs INSERT INTO items(...), SQLite assigns id #1 and commits the row to journey.db. insert returns that id.ItemDao.kt · @Insert suspend fun insert(): Long
  5. 5
    Flow. The write changes the table, so Room re-runs observeItems() and pushes a new List down the Flow. The ViewModel's .onEach records this leg; stateIn caches it as the new items value.StorageJourneyViewModel.kt · items (onEach + stateIn)
  6. 6
    Recompose. collectAsStateWithLifecycle() sees the new value, flips Compose State, and the saved-items list redraws with #1 milk — no manual refresh anywhere.MainActivity.kt · JourneyRoute + ItemRow
The point You wrote once (step 4). Steps 5–6 happened on their own because the read side is a reactive Flow. That is the whole reason to prefer Room's Flow queries over a one-shot read: the UI can never drift out of sync with the database.

The end of the journey

What survives — and when the element is finally gone

Because the row lives in journey.db on disk, it outlives the things that wipe in-memory state. The journey only truly ends when the user clears the app's data or uninstalls.

EventDoes the saved element survive?Why
Recompose / rotate the screen✓The row is on disk; the ViewModel and its StateFlow also survive rotation.
Kill the process / swipe the app away✓SQLite already committed the file. On relaunch, observeItems() reads it back on first subscription.
Reboot the device✓The database file persists across reboots.
Tap “Clear storage” in the app✕That calls DELETE FROM items — you deleted it on purpose.
Settings → Clear data / Clear storage✕Android deletes the app's private files, including journey.db.
Uninstall the app✕Private app storage is removed with the app. This is the true end of the journey.
Re-open = the read path, alone When you relaunch, no write happens — yet the list fills in. That is the return trip running by itself: the screen subscribes, observeItems() emits the persisted rows, and the Journey panel's first entry is a lone green Flow step (“Room … found N row(s)”). Storage with no save in sight.
