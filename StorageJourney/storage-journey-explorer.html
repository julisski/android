Storage Journey, hands-on

Storage Journey is a small Android app with one job: make it impossible to miss how a piece of data actually gets into storage. Type a line of text, tap Save, and the app follows that single element through every layer on the way to disk — and back to the screen. This page lets you run that trip yourself, click through the architecture and source, and poke at the two ideas that make it work: a reactive Flow and on-disk persistence.

@Entity(tableName = "items")                 // this class IS a table named "items"
data class StoredItem(
    @PrimaryKey(autoGenerate = true)         // unique row id; SQLite auto-increments it
    val id: Long = 0,                        // 0 => "let Room/SQLite pick the id on insert"
    val label: String,                       // TEXT column "label" (the user's text)
    val createdAt: Long,                     // INTEGER column "createdAt" (epoch millis)
)

data/ItemDao.kt — the read/write door
@Dao
interface ItemDao {

    // Reactive READ: a Flow that re-emits the whole list on every table change.
    @Query("SELECT * FROM items ORDER BY id DESC")
    fun observeItems(): Flow<List<StoredItem>>

    // The WRITE: suspend (touches disk); returns the new row's auto-generated id.
    @Insert
    suspend fun insert(item: StoredItem): Long

    @Delete
    suspend fun delete(item: StoredItem)

    @Query("DELETE FROM items")
    suspend fun clearAll()
}
@DaoFlow querysuspend
data/ItemDatabase.kt — the store (singleton)
@Database(entities = [StoredItem::class], version = 1, exportSchema = false)
abstract class ItemDatabase : RoomDatabase() {

    abstract fun itemDao(): ItemDao         // Room generates ItemDao_Impl

    companion object {
        @Volatile private var INSTANCE: ItemDatabase? = null

        fun getInstance(context: Context): ItemDatabase =
            INSTANCE ?: synchronized(this) {
                INSTANCE ?: Room.databaseBuilder(
                    context.applicationContext,
                    ItemDatabase::class.java,
                    "journey.db",                       // the SQLite FILE on disk
                ).build().also { INSTANCE = it }
            }
    }
}
@Databasesingleton@Volatile
data/ItemRepository.kt — the storage boundary
class ItemRepository(
    private val dao: ItemDao,
    private val journey: JourneyLog,
) {
    val items: Flow<List<StoredItem>> = dao.observeItems()

    suspend fun save(item: StoredItem): Long {
        journey.record(JourneyStage.REPOSITORY, "… dao.insert(item) — a suspend write …")
        val newId = dao.insert(item)            // suspends until SQLite commits
        journey.record(JourneyStage.ROOM, "… SQLite committed the row as id #$newId …")
        return newId
    }

    suspend fun delete(item: StoredItem) = dao.delete(item)
    suspend fun clear() = dao.clearAll()
}

journey/JourneyLog.kt — the observable trace
enum class JourneyStage(val label: String) {
    UI("UI"), VIEWMODEL("ViewModel"), REPOSITORY("Repository"),
    ROOM("Room · SQLite"), FLOW("Flow · UI"),
}

data class JourneyStep(val stage: JourneyStage, val message: String, val sinceStartMillis: Long)

class JourneyLog(private val now: () -> Long = { System.currentTimeMillis() }) {
    @Volatile private var startMillis: Long = now()
    private val _steps = MutableStateFlow<List<JourneyStep>>(emptyList())
    val steps: StateFlow<List<JourneyStep>> = _steps.asStateFlow()

    fun reset() { startMillis = now(); _steps.value = emptyList() }

    fun record(stage: JourneyStage, message: String) {
        val elapsed = now() - startMillis
        _steps.update { old -> old + JourneyStep(stage, message, elapsed) }
    }
}
enumMutableStateFlow.updateAndroid-free (testable)
StorageJourneyViewModel.kt — items + addItem
val items: StateFlow<List<StoredItem>> = repository.items
    .onEach { list -> journeyLog.record(JourneyStage.FLOW, "Room re-ran … ${list.size} row(s) …") }
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())

fun addItem(rawLabel: String) {
    val label = rawLabel.trim()
    if (label.isEmpty()) return

    journeyLog.reset()
    journeyLog.record(JourneyStage.UI, "The Save button captured \"$label\" …")

    val element = StoredItem(label = label, createdAt = System.currentTimeMillis())
    journeyLog.record(JourneyStage.VIEWMODEL, "ViewModel built StoredItem(…) and launched a coroutine …")

    viewModelScope.launch { repository.save(element) }   // records Repository + Room; then Flow re-emits
}

The key idea #1

Why a reactive Flow (and not a one-shot read)

The whole reason the screen stays correct without any “reload the list” code is that the read side is a live Flow. Change the database below and watch what happens with the Flow on versus off.

The screen reads via:
The database (disk)
items table — 0 row(s)

With a Flow, every write re-emits automatically.

The screen (what the user sees)
UI shows: 0 row(s)

⚠ Stale! The DB changed but the screen didn’t. Someone has to remember to reload().

✓ In sync — the screen tracks the database automatically.

That’s the lesson With the reactive Flow (how this app really works), the screen can never drift out of sync. With a one-shot read, every writer must remember to refresh every reader — exactly the bug Room’s Flow queries delete for you.

The key idea #2

What survives — the end of the journey

The row lives in journey.db on disk, so it outlives in-memory state. Run an event against the demo row and see whether it survives — and why. The journey only truly ends at clear-data or uninstall.
