★ Why design before coding?

A design document is a plan. It is far cheaper to change a diagram than to rewrite working code — and a plan keeps you (and your teammates) pointed at the same goal.

  • It surfaces decisions early. "How many screens? What data does each need? Where does state live?" are answered on paper, before they cost you a rewrite.
  • It is a shared language. A screen map and a data model let two people build different parts that still fit together.
  • It exposes the hard parts. Writing "the cloud sync can fail and be slow" forces you to plan a loading spinner and an error message before you hit them.
  • It is a checklist. When the design lists every screen, state, and edge case, "done" becomes measurable.
The rule of thumbYou should be able to explain your whole app — screens, data, navigation, state — using only this document, without opening the code. If you can't, the design isn't finished yet.

1 Problem & goals

Problem. Travelers keep a mental "someday" list of places, but it lives in their heads — easy to forget, impossible to see progress on, and never with them when they need it.

Goal. A small app to collect places you want to visit, track which you've been to, and play with the list — that also serves as a teaching reference for Compose + Navigation.

Goals (what success looks like)

  • Browse a list of places and open one for details.
  • Add and remove places; mark them visited.
  • See progress (how many visited) at a glance.
  • Two light "games" built from the same data.
  • Save the list so it survives closing the app.

Non-goals (deliberately out of scope)

  • User accounts / login.
  • Maps, real photos, or GPS.
  • Sharing or social features.
  • A production backend (cloud is simulated).
Why write non-goals?Saying what you won't build is as important as what you will. It stops scope creep and keeps the first version finishable.

4 Information architecture — the screen map

Before any UI, decide what screens exist and how you move between them. Four bottom tabs; two of them drill one level deeper.

                         Wanderlist (one Activity)
                                  │
        ┌──────────────┬──────────┴───────┬───────────────┐
   [ 🧭 Explore ]   [ 🎮 Play ]        [ ➕ Add ]       [ 📊 Stats ]
        │                │                  │                │
   Explore list     Play menu            Add form        Stats + Storage
        │              ┌─┴───┐             (leaf)          (leaf)
   tap a place    Flashcards  Guess
        │           (leaf)     (leaf)
   Detail (leaf)

   Each tab keeps its OWN history (back stack). Switching tabs preserves
   where you were; re-tapping the current tab returns it to its root.
Reading itA "leaf" is a screen with nowhere deeper to go. Only Explore and Play have a second level, so only their back stacks ever hold two keys.

6 Data model

One entity drives the whole app: a Destination. Design the data before the screens, because every screen is just a different view of this list.

FieldTypePurpose / used by
idIntStable identity; travels in the nav key; list keys.
nameStringList, Detail, Flashcard answer, Guess prompt.
country / continentStringDetail; Stats groups by continent.
emojiStringStand-in for a photo (no image dependency).
blurb / notesStringShort (list) and long (detail) descriptions.
bestSeasonStringDetail.
priorityInt (1–5)"Excitement" stars; set by the Add slider.
visitedBooleanToggled on list + detail; drives all progress numbers.
Design choice: immutable data.Every field is a val. To "change" a place we don't edit it in place — we replace it with a .copy(visited = …). Immutable data + an observable list is the idiomatic Compose way, and it makes "what changed?" obvious.

Storage shape. The whole list is serialized to one JSON string (via @Serializable) and written to local storage or "the cloud":

List<Destination>  ──encodeToString()──▶  "[ {id:1,name:\"Kyoto\",…}, … ]"  ──▶  device / cloud
List<Destination>  ◀──decodeFromString()──  "[ … ]"                       ◀──  device / cloud

7 Navigation design

Navigation 3 models navigation as a back stack of keys. Each screen has a key; pushing a key goes forward, popping goes back.

The keys (one per screen)

KeyKindScreen
ExploreKeydata objectExplore list (Tab 0 root)
DetailKey(destinationId)data classDetail — carries which place
PlayKeydata objectPlay menu (Tab 1 root)
FlashcardKey / GuessKeydata objectThe two games
AddKeydata objectAdd form (Tab 2 root)
StatsKeydata objectStats (Tab 3 root)
object vs class.A screen that needs an argument (which place?) is a data class that carries it. A screen with one instance and no arguments is a data object singleton.

Rules

  • One back stack per tab. All four exist at once, so each tab remembers its place.
  • Back pops the active tab's stack. Re-tapping the current tab pops it to its root.
  • The top app bar's title and Back arrow, and the FAB, are derived from the top key — no separate "current screen" variable to keep in sync.

8 State management plan

Decide what state exists and where it lives before coding — this is where most beginner bugs come from.

StateLives inHowSurvives rotation?
The list of places (app data)WanderlistAppremember { …toMutableStateList() }No — by design (see note)
Selected tabWanderlistApprememberSaveableYes
Dark themeMainActivity (above the theme)rememberSaveableYes
Each tab's back stackWanderlistApprememberNavBackStackYes
Dialog open? / card flipped? / form fieldsThe individual screenremember (local UI state)Mostly no — momentary

Unidirectional data flow

   WanderlistApp  (owns the list — the single source of truth)
        │  data flows DOWN (places, callbacks)
        ▼
   Screens  (stateless: render data, report taps)
        │  events flow UP (onToggleVisited, onAdd, onDelete)
        ▼
   Edit helpers  (the only code that changes the list)
        └──────────────▶ list changes ▶ screens recompose
Design decision (and its honest trade-off).The list uses plain remember, so it resets on rotation. That's intentional: it sets up the next topics — persistence (the Storage card) and a ViewModel — which is exactly the kind of trade-off a design doc should record on purpose, not stumble into.

10 Storage design next topic

The key idea: depend on an interface, not a storage technology. The UI calls save()/load() and doesn't care where the bytes go.

                DestinationStore  (interface)
                 suspend save(list)
                 suspend load(): list?
                  ┌──────────┴───────────┐
        LocalDestinationStore     CloudDestinationStore
        (this device,              (a remote server —
         SharedPreferences          SIMULATED here with a
         + JSON)                    delay; real = Firebase
                                    or a REST API)

Local vs cloud

LocalCloud
WhereOn the device onlyA server (shared across devices)
SpeedFast (disk)Slow (network) — needs a spinner
Can fail?RarelyYes (offline, auth) — needs error handling
In this appReal (SharedPreferences + JSON)Simulated (in-memory + delay)
Real-world toolDataStore / RoomFirebase Firestore / Retrofit
Why both are suspend.Saving/loading can be slow, so it must run off the main thread or the UI freezes. Marking the functions suspend forces callers to run them in a coroutine — and lets the cloud version show a spinner and handle errors, exactly like real life. Push then Pull behaves like syncing two devices through one cloud document.
Constraint recorded in the design.We can't ship real Firebase keys in a teaching repo, so cloud is simulated. The code's shape is identical to the real thing — the comments in CloudDestinationStore.kt show the exact Firestore/Retrofit calls that would replace the fake one.

12 Edge cases & states

Real apps spend most of their code on the "unhappy paths". Plan them now.

  • Empty list. Explore shows "No places yet — tap ＋". The games show "add some places first".
  • Deleting the open place. Detail pops first, then removes — so it never renders a place that's gone. Resolution uses firstOrNull + a null guard.
  • Invalid form. The Add button stays disabled until name + country are filled.
  • Slow / failed cloud. A spinner shows and buttons disable while syncing; failures surface as a Snackbar (try/catch/finally).
  • Nothing saved yet. Load reports "nothing saved" rather than crashing or clearing the list.
  • Corrupt saved data. Local load is wrapped in runCatching so a bad value can't crash the app.

13 Quality goals (non-functional)

Accessibility & theming

  • Material 3 colors → readable contrast in light & dark.
  • Dark mode is a real toggle; the theme is just state.
  • Text uses the type scale (titleMedium, …), so it respects the user's font size.

Performance & build

  • LazyColumn with stable key = { it.id } — only visible rows compose.
  • Disk/network work runs off the main thread (Dispatchers.IO, suspend).
  • Zero image/icon dependencies (emoji) — keeps the build simple & green.

16 Open questions & future work

  • Persistence done right. Move the list into a ViewModel + Room/DataStore so it survives rotation and process death automatically (today it needs an explicit Save/Load).
  • Real cloud. Replace CloudDestinationStore with Firebase Firestore or a REST API; add accounts so each user has their own list.
  • Images. Swap emoji for real photos (Image + a loader like Coil); the Guess game's hit-testing already works on coordinates, so it ports to real images directly.
  • Conflict handling. What if local and cloud disagree? Define a sync/merge rule (last-write-wins? newest timestamp?).
Remember:a design document is a living thing. Update it when the design changes — a stale design doc is worse than none.
