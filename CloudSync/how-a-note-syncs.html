Cloud storage, in conjunction with Room

How a note syncs

CloudSync shows the pattern every real app eventually needs: a local database (Room) kept in sync with a cloud backend, that keeps working when the device is offline. The trick is one rule — Room is the single source of truth — plus a background SyncWorker that reconciles it with the cloud. This page is that design as blocks of real code, from the entity's sync fields to last-write-wins conflict resolution.

Room (source of truth) WorkManager sync optimistic writes last-write-wins offline-first
The one sentence to hold onto The UI never touches the network. It reads Room (so it's instant and works offline) and writes to Room (optimistically, marked Pending); a background worker pushes those changes and pulls remote ones when the network allows. Offline isn't an error state — it's just "the sync queue hasn't drained yet."

The shape of it

Room is the source of truth; the cloud is a sync target

The UI ⇄ Room loop is the same reactive pattern as the local-storage demos. The new part is the repository + worker that sync Room with the cloud — invisible to the UI.

Initialization · before any sync

Step 0 — wiring the offline-first stack

Four pieces: Room (+ KSP) for the local cache, Retrofit + serialization for the cloud transport, and WorkManager for background sync.

build.gradle.kts · the offline-first stack
// Room: the LOCAL single source of truth. The UI only ever reads this.
implementation(libs.androidx.room.runtime)
implementation(libs.androidx.room.ktx)
ksp(libs.androidx.room.compiler)                    // KSP generates the SQLite-backed impls
// Cloud transport: Retrofit + OkHttp + JSON for the Real sync backend.
implementation(libs.retrofit)
implementation(libs.retrofit.converter.kotlinx.serialization)
implementation(libs.okhttp)
implementation(libs.kotlinx.serialization.json)
// WorkManager: deferrable, retryable background sync (network-constrained, survives death).
implementation(libs.androidx.work.runtime.ktx)
Two permissions The manifest declares INTERNET (for the real cloud) and ACCESS_NETWORK_STATE (so WorkManager's network constraint and the online/offline banner can read connectivity). Both are normal install-time permissions — harmless in the default offline Fake mode.

Step 1 · the row, plus sync bookkeeping

The entity — a normal note + the offline-first fields

A Note is a normal Room @Entity with four extra fields that make sync possible. The cloud's wire shape is a separate @Serializable NoteDto, converted by mappers.

data/Note.kt
enum class SyncState { SYNCED, PENDING, FAILED }   // the outbox marker

@Entity(tableName = "notes")
data class Note(
    @PrimaryKey val id: String,            // CLIENT-generated UUID — an offline row has an id before the server sees it
    val title: String,
    val body: String,
    val updatedAt: Long,                   // the clock for last-write-wins conflict resolution
    val syncState: SyncState = SyncState.PENDING,
    val deleted: Boolean = false,          // soft-delete tombstone — a delete you can still PUSH while offline
)

@Serializable
data class NoteDto(val id: String, val title: String, val body: String, val updatedAt: Long, val deleted: Boolean = false)

fun Note.toDto() = NoteDto(id, title, body, updatedAt, deleted)
fun NoteDto.toEntity(state: SyncState = SyncState.SYNCED) = Note(id, title, body, updatedAt, state, deleted)
  • Client-generated id — a UUID assigned on creation, so a note made offline already has a stable identity to push later (you can't let the server assign it).
  • updatedAt — the timestamp two devices compare to decide whose edit wins.
  • syncState — has this row reached the cloud yet? Drives both the sync engine and the UI badge.
  • deleted tombstone — a delete is a soft-delete first, so it survives offline and can still be pushed; the row is hard-deleted only after the cloud confirms.

Step 2 · reads for the UI, queries for the engine

The DAO

The DAO has two audiences: reactive reads the UI observes (visible notes only — tombstones hidden), and queries the sync engine uses (the outbox of unsynced rows, and the pull cursor).

data/NoteDao.kt
@Dao
interface NoteDao {
    // --- reactive reads for the UI ---
    @Query("SELECT * FROM notes WHERE deleted = 0 ORDER BY updatedAt DESC")
    fun observeVisibleNotes(): Flow<List<Note>>            // tombstones hidden

    @Query("SELECT COUNT(*) FROM notes WHERE syncState != 'SYNCED'")
    fun observePendingCount(): Flow<Int>                    // drives "N pending"

    // --- sync-engine queries ---
    @Query("SELECT MAX(updatedAt) FROM notes")
    suspend fun maxUpdatedAt(): Long?                       // the pull cursor

    @Query("SELECT * FROM notes WHERE syncState != 'SYNCED'")
    suspend fun notesToPush(): List<Note>                   // the OUTBOX

    @Query("SELECT * FROM notes WHERE id = :id")
    suspend fun findById(id: String): Note?

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun upsert(note: Note)

    @Query("DELETE FROM notes WHERE id = :id")
    suspend fun hardDelete(id: String)                     // only after the cloud confirms a tombstone
}

Step 3 · the cloud, Real and Fake

The cloud transport — one interface, two implementations

The sync engine talks to a tiny CloudApi with two operations: pullSince (remote rows after a cursor) and push (upsert one row). Behind it sits a Retrofit-backed real cloud or an in-memory fake — swapped by one flag, exactly like the NetworkParsing/WebSocketLive demos.

data/CloudApi.kt · the contract + the switch
interface CloudApi {
    suspend fun pullSince(since: Long): List<NoteDto>   // remote rows newer than the cursor
    suspend fun push(note: NoteDto): NoteDto            // upsert; returns the server's copy
}

// THE SWITCH — fake (default, offline) is a process singleton so its in-memory server persists.
fun provideCloudApi(useFake: Boolean = true): CloudApi =
    if (useFake) fakeCloud else RetrofitCloudApi()
data/CloudApi.kt · FakeCloudApi (offline server)
class FakeCloudApi : CloudApi {
    private val server = linkedMapOf<String, NoteDto>()   // the pretend server's store

    init {
        // Seed one row so a brand-new install pulls a "Welcome to CloudSync" note on first sync.
        val seededAt = System.currentTimeMillis() - 1_000
        server["seed-welcome"] = NoteDto("seed-welcome", "Welcome to CloudSync", "Came FROM the cloud.", seededAt)
    }

    override suspend fun pullSince(since: Long): List<NoteDto> {
        delay(700)                                         // imitate latency
        return server.values.filter { it.updatedAt > since }.sortedBy { it.updatedAt }
    }
    override suspend fun push(note: NoteDto): NoteDto {
        delay(500)
        val existing = server[note.id]
        if (existing == null || note.updatedAt >= existing.updatedAt) server[note.id] = note  // server LWW too
        return server.getValue(note.id)
    }
    fun simulateRemoteEdit() { /* add a row as if another device created it */ }
}

Step 4 · write locally, instantly

Optimistic writes — Room first, marked Pending

Every create/edit/delete writes to Room immediately with syncState = PENDING. The user sees the change at once — online or offline — and the row joins the outbox to be pushed later. A delete is a soft-delete so it can still be pushed.

data/NoteRepository.kt · the write methods
val notes: Flow<List<Note>> = dao.observeVisibleNotes()    // what the UI observes

suspend fun addNote(title: String, body: String) {
    dao.upsert(Note(
        id = UUID.randomUUID().toString(),                 // identity before the server sees it
        title = title.trim(), body = body.trim(),
        updatedAt = System.currentTimeMillis(),
        syncState = SyncState.PENDING,                      // joins the outbox
    ))
}

suspend fun deleteNote(id: String) {
    val existing = dao.findById(id) ?: return
    dao.upsert(existing.copy(deleted = true,               // SOFT delete (tombstone) — still pushable offline
        updatedAt = System.currentTimeMillis(), syncState = SyncState.PENDING))
}

Step 5 · reconcile with the cloud

sync() — push the outbox, then pull remote changes

The repository's sync() is the reconciler. It pushes every unsynced row, then pulls rows newer than the local cursor. It may throw (no network) — the worker catches that and retries; the local data is safe in Room the whole time.

data/NoteRepository.kt · sync()
suspend fun sync() {
    // 1) PUSH — every row created/edited/deleted offline.
    for (local in dao.notesToPush()) {
        try {
            val serverCopy = cloud.push(local.toDto())     // may throw -> retried by the worker
            if (local.deleted) dao.hardDelete(local.id)    // tombstone confirmed -> remove locally
            else dao.upsert(serverCopy.toEntity(SyncState.SYNCED))
        } catch (e: Exception) {
            dao.upsert(local.copy(syncState = SyncState.FAILED))   // surface it in the UI
            throw e                                                 // let the worker retry
        }
    }
    // 2) PULL — remote rows newer than the newest we already have.
    val since = dao.maxUpdatedAt() ?: 0L
    for (remote in cloud.pullSince(since)) {
        val local = dao.findById(remote.id)
        if (shouldAcceptRemote(local, remote)) {           // last-write-wins
            if (remote.deleted) dao.hardDelete(remote.id)
            else dao.upsert(remote.toEntity(SyncState.SYNCED))
        }
    }
}
Push before pull Pushing first means your local edits reach the server before you ask it for the latest — so the pull sees the merged result and your work isn't accidentally overwritten by a slightly-older server copy.

Step 6 · who wins when both changed?

Conflict resolution — last-write-wins, as a pure function

When the local and remote copies disagree, something must decide. CloudSync uses last-write-wins by updatedAt, extracted into a pure function so it's trivially unit-testable (no Room, no coroutines).

data/NoteRepository.kt · shouldAcceptRemote
/**
 * Accept the incoming remote row only when we have no local copy, OR our local copy is
 * already SYNCED and not newer. A still-PENDING local edit wins here — it carries unpushed
 * work that will be pushed next sync, where the server makes the final call.
 */
fun shouldAcceptRemote(local: Note?, remote: NoteDto): Boolean =
    local == null || (local.syncState == SyncState.SYNCED && remote.updatedAt >= local.updatedAt)
StrategyHowUsed here?
Last-write-winsCompare updatedAt; newer overwrites✓ simple, fine for personal data
Version / optimistic lockServer rejects a stale version; client re-pulls— for shared/multi-user data
Field-level merge / CRDTMerge non-conflicting fields— collaborative editing

Step 7 · where offline "just queues"

The background worker

A ViewModel coroutine dies with the screen, so sync runs in a WorkManager job that survives process death, waits for the network, and retries. The NetworkType.CONNECTED constraint is exactly why offline writes simply wait instead of failing.

sync/SyncWorker.kt
class SyncWorker(appContext: Context, params: WorkerParameters) : CoroutineWorker(appContext, params) {
    override suspend fun doWork(): Result {
        val repo = NoteRepository.get(applicationContext)   // the SAME repo the UI uses
        return try { repo.sync(); Result.success() }
        catch (e: Exception) { Result.retry() }             // backoff; try again later
    }
    companion object { const val NAME = "cloud-sync" }
}
sync/SyncScheduler.kt · the constraint that makes offline queue
fun requestSync(context: Context) {
    val request = OneTimeWorkRequestBuilder<SyncWorker>()
        .setConstraints(Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED)  // run when online, DEFER when offline
            .build())
        .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
        .build()
    WorkManager.getInstance(context)
        .enqueueUniqueWork(SyncWorker.NAME, ExistingWorkPolicy.APPEND_OR_REPLACE, request)
}
This one constraint is the whole offline story With NetworkType.CONNECTED, an enqueued sync simply waits while the device is offline and fires the moment connectivity returns — no polling, no lost work, no special offline code path.

Step 8 · ViewModel & screen

Local state in, intents out

The ViewModel exposes Room as StateFlows (so the UI is offline-first by construction) plus the online signal; every intent does the same two steps — write to Room, request a sync.

CloudSyncViewModel.kt
val notes: StateFlow<List<Note>> = repo.notes.stateIn(viewModelScope, started, emptyList())
val pendingCount: StateFlow<Int> = repo.pendingCount.stateIn(viewModelScope, started, 0)
val isOnline: StateFlow<Boolean> = connectivity.isOnline.stateIn(viewModelScope, started, true)

fun addNote(title: String, body: String) = writeThenSync { repo.addNote(title, body) }

private fun writeThenSync(block: suspend () -> Unit) {
    viewModelScope.launch {
        block()                                     // 1) optimistic local write (Room updates the UI)
        SyncScheduler.requestSync(getApplication()) // 2) push/pull when the network allows
    }
}

The screen reads those flows with collectAsStateWithLifecycle(), draws each note with a color-coded sync badge (✓ Synced / ⏳ Pending / ⚠ Failed), and shows an online/offline + "N pending" banner.

The crux

What happens when the device is offline

Offline is the default case the architecture is built around, not an error to handle.

ActionOnlineOffline
App launchReads Room → instant; sync refreshes in the backgroundReads Room → instant. Sync silently deferred. No spinner, no crash.
Read / scrollFrom RoomFrom Room — identical
Create / editRoom write → pushed → SyncedRoom write → stays Pending in the outbox
DeleteTombstone → pushed → hard-deleted locallyTombstone stays queued (can't push a row already gone)
Reconnect—WorkManager fires, drains the outbox (→ Synced), pulls remote changes
Why the UI never breaks offline Because the UI only ever reads Room and writes Room, it literally cannot tell whether the network is up. The gap between the device and the cloud is just a queue waiting to drain.

The full trip — add offline, then reconnect

Step through what happens when you add a note with no connection, then come back online.

Scenario: offline, you add "Buy milk", then reconnect
1 / 6
  1. 1
    Write. addNote("Buy milk", …) inserts a row into Room with a fresh UUID and syncState = PENDING. The list recomposes instantly — even though you're offline.NoteRepository.addNote() → Room
  2. 2
    Queue. The ViewModel calls SyncScheduler.requestSync(). WorkManager accepts the job but its NetworkType.CONNECTED constraint is unmet, so it waits.SyncScheduler.requestSync()
  3. 3
    Offline UI. The note shows a ⏳ Pending badge; the banner reads "🔴 Offline — changes are queued · ⏳ 1 pending". Nothing is lost; it's safely in Room.MainActivity · SyncBadge + banner
  4. 4
    Reconnect. Connectivity returns; WorkManager's constraint is satisfied, so it runs SyncWorker.doWork() → repository.sync().SyncWorker → repository.sync()
  5. 5
    Push. notesToPush() returns the pending note; cloud.push(it) uploads it; the row is re-saved as SYNCED with the server's copy.sync() · push loop
  6. 6
    Pull + recompose. pullSince(cursor) brings any remote changes; upserts hit Room; the Flow re-emits and the badge flips to ✓ Synced. No manual refresh anywhere.sync() · pull loop → Room → UI
