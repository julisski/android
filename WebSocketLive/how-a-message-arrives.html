A second network protocol

How a message arrives

WebSocket Live teaches backend communication over a different protocol. Where the Network Parsing demo does HTTP request → response (ask once, get one answer), this keeps a single full-duplex connection open so either side can push messages at any time. It shows a live connection status, a running transcript, and a box to send text that an echo server sends straight back — built on OkHttp's WebSocket behind a ViewModel. This page is that flow as blocks of real code.

OkHttp WebSocket Flow<ConnectionEvent> pure reducer StateFlow + Compose Kotlin
The one sentence to hold onto A socket is a stream of events over time, not a single call. The connection's callbacks become a Flow<ConnectionEvent>; a pure reducer folds each event into one immutable LiveUiState; and the screen renders that state. Because the reducer is pure, the whole state machine is unit-testable with no coroutines.

Runs offline by default

The app ships in fake mode — the LiveViewModel constructor defaults to FakeWebSocketClient(), which scripts a connect and echoes whatever you send, so it builds, runs, and unit-tests with no network. Wiring in RealWebSocketClient() flips it to a live socket at wss://echo.websocket.events.

The shape of the stream

The event stream, folded into state

Unlike a one-shot request, the connection reports a sequence of things over time. Each one flows up through the layers: the socket's callbacks become events, the reducer folds them into a new state, and the screen recomposes. Sending text flows the other way — and is shown optimistically before the echo returns.

Step 0 — wiring the socket on

This project needs just one networking library: OkHttp, whose WebSocket is the full-duplex transport. There is no @Serializable state here, so — unlike Network Parsing — the kotlinx.serialization plugin is deliberately not applied.

build.gradle.kts · plugins + the one networking dep
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.compose)
    // Only the Android + Compose plugins; there is no @Serializable state here, so
    // kotlin.serialization is intentionally NOT applied.
}

dependencies {
    // OkHttp's WebSocket is the full-duplex transport this project demonstrates.
    implementation(libs.okhttp)
    // viewModel() + collectAsStateWithLifecycle() to observe StateFlow<LiveUiState>.
    implementation(libs.androidx.lifecycle.viewmodel.compose)
    implementation(libs.androidx.lifecycle.runtime.compose)
}
The INTERNET permission AndroidManifest.xml declares <uses-permission android:name="android.permission.INTERNET" /> actively, so flipping to the real socket works with no manifest edit. It's harmless in the default offline fake mode.

Step 1 · what can happen on the wire

The events & UI models

A WebSocket is a stream, so we model the closed set of things that can happen as a sealed ConnectionEvent. Because it's sealed, the reducer's when over it is exhaustive. Two small UI models go alongside it: the coarse ConnectionStatus chip and one transcript line, ChatMessage.

ConnectionEvent.kt · the wire events
sealed interface ConnectionEvent {
    data object Connecting : ConnectionEvent                    // handshake started
    data object Open : ConnectionEvent                          // open and ready
    data class MessageReceived(val text: String) : ConnectionEvent  // a text frame arrived
    data class Closed(val reason: String) : ConnectionEvent     // closed cleanly
    data class Failure(val error: String) : ConnectionEvent     // network dropped / handshake rejected
}
ConnectionEvent.kt · the UI models
// The coarse status shown as a header chip, derived from the event stream.
enum class ConnectionStatus { Disconnected, Connecting, Connected, Failed }

// One transcript line. fromMe = true if WE sent it; false if it arrived from the server.
data class ChatMessage(val text: String, val fromMe: Boolean)

Step 2 · the connection, Real and Fake

The connection — callbacks become a Flow

The protocol hides behind a small LiveConnection interface: an events() stream plus the actions connect() / send() / close(). The real implementation wires OkHttp's WebSocketListener callbacks into a MutableSharedFlow; the fake scripts the same events offline.

WebSocketClient.kt · the contract
interface LiveConnection {
    fun events(): Flow<ConnectionEvent>   // everything the socket reports, as a stream
    fun connect()                         // open (async; watch events for Connecting/Open)
    fun send(text: String)                // send a text frame (no-op if not open)
    fun close()                           // close cleanly
}
WebSocketClient.kt · RealWebSocketClient (OkHttp)
class RealWebSocketClient(
    private val url: String = "wss://echo.websocket.events",
    private val httpClient: OkHttpClient = OkHttpClient(),
) : LiveConnection {

    private var webSocket: WebSocket? = null
    // extraBufferCapacity lets callbacks emit with tryEmit() — never suspending the OkHttp thread.
    private val events = MutableSharedFlow<ConnectionEvent>(extraBufferCapacity = 64)
    override fun events(): Flow<ConnectionEvent> = events.asSharedFlow()

    override fun connect() {
        events.tryEmit(ConnectionEvent.Connecting)
        val request = Request.Builder().url(url).build()
        webSocket = httpClient.newWebSocket(request, object : WebSocketListener() {
            override fun onOpen(webSocket: WebSocket, response: Response) {
                events.tryEmit(ConnectionEvent.Open)
            }
            override fun onMessage(webSocket: WebSocket, text: String) {
                events.tryEmit(ConnectionEvent.MessageReceived(text))
            }
            override fun onClosing(webSocket: WebSocket, code: Int, reason: String) {
                webSocket.close(NORMAL_CLOSURE, null)
                events.tryEmit(ConnectionEvent.Closed(reason.ifBlank { "closed" }))
            }
            override fun onFailure(webSocket: WebSocket, t: Throwable, response: Response?) {
                events.tryEmit(ConnectionEvent.Failure(t.message ?: "connection failed"))
            }
        })
    }

    override fun send(text: String) { webSocket?.send(text) }    // OkHttp queues the frame
    override fun close() { webSocket?.close(NORMAL_CLOSURE, "client closed"); webSocket = null }

    private companion object { const val NORMAL_CLOSURE = 1000 }  // WebSocket "normal closure" code
}
WebSocketClient.kt · FakeWebSocketClient + the switch
class FakeWebSocketClient : LiveConnection {
    private val events = MutableSharedFlow<ConnectionEvent>(extraBufferCapacity = 64)
    override fun events(): Flow<ConnectionEvent> = events.asSharedFlow()

    override fun connect() {
        events.tryEmit(ConnectionEvent.Connecting)
        events.tryEmit(ConnectionEvent.Open)
        events.tryEmit(ConnectionEvent.MessageReceived("Connected to the OFFLINE fake echo server."))
    }
    override fun send(text: String) {
        events.tryEmit(ConnectionEvent.MessageReceived(text))   // echo it straight back, like the real server
    }
    override fun close() { events.tryEmit(ConnectionEvent.Closed("client closed")) }
}

// THE SWITCH — change one argument to toggle offline (default) vs the live wss:// server.
fun provideLiveConnection(useFake: Boolean = true): LiveConnection =
    if (useFake) FakeWebSocketClient() else RealWebSocketClient()
Why tryEmit and MutableSharedFlow OkHttp delivers callbacks on its own thread, which must never block. A MutableSharedFlow with spare buffer capacity lets each callback emit with the non-suspending tryEmit() — so socket events flow into the app without ever stalling the network thread.

Step 3 · the state machine

The pure reducer — fold one event into the next state

The heart of the app is a single pure function: given the current LiveUiState and one ConnectionEvent, return the next state. No coroutines, no side effects — so the entire state machine is trivially unit-testable.

LiveViewModel.kt · LiveUiState + reduceLiveState
data class LiveUiState(
    val status: ConnectionStatus = ConnectionStatus.Disconnected,
    val messages: List<ChatMessage> = emptyList(),     // the transcript, oldest first
)

fun reduceLiveState(state: LiveUiState, event: ConnectionEvent): LiveUiState =
    when (event) {
        is ConnectionEvent.Connecting -> state.copy(status = ConnectionStatus.Connecting)
        is ConnectionEvent.Open       -> state.copy(status = ConnectionStatus.Connected)
        is ConnectionEvent.MessageReceived ->
            state.copy(messages = state.messages + ChatMessage(event.text, fromMe = false))
        is ConnectionEvent.Closed     -> state.copy(status = ConnectionStatus.Disconnected)
        is ConnectionEvent.Failure    -> state.copy(status = ConnectionStatus.Failed)
    }
Pure = testable Because reduceLiveState takes a state and an event and returns a state — with no I/O — the unit tests check the whole machine with plain assertions: reduceLiveState(LiveUiState(), Open).status == Connected. No dispatcher, no socket.

Step 4 · the conductor

The ViewModel — collect the stream, fold it, expose it

The ViewModel collects the connection's events() for its whole lifetime, folding each event into the state with the pure reducer, and publishes the result as a StateFlow. Sending optimistically appends your own line; the echo arrives later through the same stream.

LiveViewModel.kt · collect in init + send()
class LiveViewModel(
    private val connection: LiveConnection = FakeWebSocketClient(),   // injected; defaults to offline fake
) : ViewModel() {

    private val _uiState = MutableStateFlow(LiveUiState())
    val uiState: StateFlow<LiveUiState> = _uiState.asStateFlow()

    init {
        // Collect the event stream for the ViewModel's whole lifetime, folding each
        // event into the UI state with the pure reducer.
        viewModelScope.launch {
            connection.events().collect { event ->
                _uiState.update { current -> reduceLiveState(current, event) }
            }
        }
    }

    fun connect() = connection.connect()

    fun send(text: String) {
        if (text.isBlank()) return
        connection.send(text)                                        // ask the server (echo)
        _uiState.update { it.copy(messages = it.messages + ChatMessage(text, fromMe = true)) }  // optimistic
    }

    fun disconnect() = connection.close()
}
Optimistic UI send() appends your outgoing line immediately so the UI feels instant; the server's echo returns as a separate MessageReceived event and is appended (as fromMe = false) by the reducer. Two lines, two sources, one transcript.

Step 5 · where it appears on screen

The screen — open once, render the stream

The stateful LiveScreen observes the state, owns the input text, and opens the socket exactly once with LaunchedEffect(Unit). The stateless LiveContent draws the status chip, the transcript, and the input row — so previews can drive it with hand-made state, no real socket.

MainActivity.kt · LiveScreen (stateful)
@Composable
fun LiveScreen(modifier: Modifier = Modifier, viewModel: LiveViewModel = viewModel()) {
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()
    // Open the connection exactly once when the screen first appears (Unit key = run once).
    LaunchedEffect(Unit) { viewModel.connect() }
    var input by remember { mutableStateOf("") }
    LiveContent(
        uiState = uiState,
        input = input,
        onInputChange = { input = it },
        onSend = { viewModel.send(input); input = "" },   // send, then clear the box
        modifier = modifier,
    )
}
MainActivity.kt · LiveContent (stateless) — status, transcript, input
Text("Status: ${uiState.status}", color = when (uiState.status) {
    ConnectionStatus.Connected -> MaterialTheme.colorScheme.primary
    ConnectionStatus.Failed    -> MaterialTheme.colorScheme.error
    else                       -> MaterialTheme.colorScheme.onSurfaceVariant
})

LazyColumn(modifier = Modifier.fillMaxWidth().weight(1f)) {   // the transcript fills the middle
    items(uiState.messages) { message -> MessageRow(message) }
}

Row {                                                         // the input row
    OutlinedTextField(value = input, onValueChange = onInputChange, /* … */)
    Button(onClick = onSend, enabled = input.isNotBlank()) { Text("Send") }
}
Sent vs received, at a glance MessageRow renders fromMe lines right-aligned in the primary colour (“you: …”) and server lines left-aligned (“server: …”), so the two-way nature of the socket is visible in the transcript.

The full trip, one step at a time

Step through the chain from opening the socket to seeing your echoed message — in offline fake mode, which mirrors the real server exactly.

Scenario: the screen opens, you connect, then send “hello”
1 / 7
  1. 1
    Open once. LiveScreen runs LaunchedEffect(Unit) { viewModel.connect() }, which calls connection.connect().MainActivity.kt · LaunchedEffect(Unit)
  2. 2
    Connecting → Open. The connection emits Connecting then Open. The collector folds them: status becomes Connecting, then Connected.WebSocketClient.kt · connect() → reduceLiveState
  3. 3
    Greeting. A MessageReceived("Connected to the OFFLINE fake echo server.") arrives; the reducer appends it as a fromMe = false line.LiveViewModel.kt · reduceLiveState (MessageReceived)
  4. 4
    You type “hello”, tap Send. onSend calls viewModel.send("hello").MainActivity.kt · Button(onClick = onSend)
  5. 5
    Optimistic append. send() calls connection.send("hello") and immediately appends your line as fromMe = true — the UI updates instantly.LiveViewModel.kt · send()
  6. 6
    The echo returns. The (fake or real) server sends “hello” back, arriving as a MessageReceived("hello") event.WebSocketClient.kt · send() echoes
  7. 7
    Fold → recompose. The collector folds the echo into a fromMe = false line; the StateFlow emits and the transcript recomposes with both lines.LiveViewModel.kt + MainActivity.kt
The point One connection stayed open the whole time. Your message and the server's reply are two separate events on that single socket — exactly the full-duplex behaviour HTTP can't express in one call.

Two protocols, side by side

WebSocket vs HTTP

The sibling Network Parsing demo uses HTTP. Putting them next to each other is the fastest way to feel why a socket is a different tool.

 HTTP (Network Parsing)WebSocket (this app)
ShapeOne request → one responseOne open connection, many frames over time
Who can speakClient asks; server answersEither side pushes at any time (full-duplex)
Modelled asA single suspend call returning dataA Flow<ConnectionEvent> stream
UI stateLoading / Success / ErrorA status chip + a growing transcript
LibraryRetrofit + kotlinx.serializationOkHttp WebSocket
Good forFetching/CRUD on demandLive chat, tickers, presence, notifications
Same skeleton, different transport Both apps use the identical teaching spine: a Real/Fake switch for offline runs, a ViewModel exposing a StateFlow, and a stateless Compose screen. Only the data layer differs — request/response vs an event stream — which is exactly the comparison this pair is built to teach.
