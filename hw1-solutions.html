Part A. Concepts — short answer

Two to four sentences each, using the real names from the code.

A1 — setContent & stateless UI6 pts

What is the job of setContent { } inside onCreate? Why is Greeting(name: String) a stateless composable, and what is modifier for?

setContent bridges the Activity to Compose: it sets the Activity's content to the Compose UI tree described in the lambda — the modern replacement for setContentView(R.layout.…). Greeting is stateless because it owns no state of its own (no remember {}, nothing it mutates); it is a pure function of its input name, so the same name always draws the same thing. The modifier parameter lets the caller control layout/decoration from the outside — here MainActivity passes Modifier.padding(innerPadding) so the text clears the system bars.

A2 — “Intent” without an Intent8 pts

How does DetailKey(message) + backStack.add(DetailKey("…")) replace the old Intent-extra pattern, and what are two advantages?

Instead of building an Intent, calling putExtra("msg", …) / startActivity, and reading getStringExtra("msg") on the other side, the argument becomes a typed field on a key: DetailKey(val message: String). backStack.add(DetailKey("Hello from Home!")) pushes that key onto the back stack and DetailScreen reads key.message directly. Advantage 1 — type safety: message is a compile-checked String field, so a wrong name or type is a compile error, not a runtime null from getStringExtra. Advantage 2 — no Activity boilerplate: navigation is just adding to a list of keys in one Activity (no second Activity, no manifest entry), and because the key is @Serializable the whole back stack is saved/restored for free.

A3 — pass the id, not the object8 pts

Give two reasons the id travels in the key instead of the whole Item.

1 — The key is persisted. It is @Serializable and saved into the back stack across rotation/process death, so keeping it a small, stable Int is cheap and robust; serializing a whole Item is heavier and easy to let go stale. 2 — Single source of truth. itemById(key.itemId) re-resolves the id against the one sampleItems source every time, so the detail always reflects the current data; carrying a copy of the Item in the key would duplicate data that could drift out of sync.

A4 — the back stack8 pts

What is the back stack? What do add(key) vs removeLastOrNull() do, and why is removeLastOrNull() safer than removeLast()?

The back stack is the ordered list of keys for the screens you've navigated through; the top (last) key is the screen currently shown (NavDisplay renders the top). add(key) pushes a new key on top = navigate forward; removeLastOrNull() pops the top key = go back (the previous screen reappears). removeLastOrNull() is safer than removeLast() because on an empty stack it returns null instead of throwing NoSuchElementException — so a stray Back press at the root can't crash the app.

B1 — extend the greeting (SingleActivity)8 pts

Add a second parameter to Greeting (course), display two lines, and fix the setContent call and the @Preview.

File: SingleActivity-solution/app/src/main/java/com/example/singleactivity/MainActivity.kt

import androidx.compose.foundation.layout.Column   // two lines need a container

@Composable
fun Greeting(name: String, course: String, modifier: Modifier = Modifier) {
    Column(modifier = modifier) {                  // Column stacks the lines vertically
        Text(text = "Hello $name!")                // line 1
        Text(text = "Welcome to $course.")         // line 2 — uses the new `course` arg
    }
}

// in setContent { … }:
Greeting(
    name = "Android",
    course = "Mobile Dev",                         // the new second argument
    modifier = Modifier.padding(innerPadding)
)

// uncommented + updated preview (still compiles):
@Preview(showBackground = true)
@Composable
fun GreetingPreview() {
    SingleActivityTheme {
        Greeting("Android", "Mobile Dev")          // both args; modifier defaults
    }
}

Why a Column: a composable function can only emit one root element at a time here, so two Texts must be wrapped in a layout container; Column stacks them vertically. Adding the course parameter forces every call site (the setContent call and the preview) to supply it — the compiler points you to each one.

B2 — add an argument to the key (Intent)10 pts

Give DetailKey a second field count: Int, pass a value when Home navigates, and show "message (count)" on the Detail screen.

File: Intent-solution/app/src/main/java/com/example/intent/MainActivity.kt

// 1) the key gains a typed field:
data class DetailKey(val message: String, val count: Int) : NavKey

// 2) Home passes a count when navigating:
onOpenDetail = { backStack.add(DetailKey("Hello from Home!", 3)) }

// 3) the entry forwards both fields down:
entry<DetailKey> { key ->
    DetailScreen(
        message = key.message,
        count = key.count,
        onBack = { backStack.removeLastOrNull() }
    )
}

// 4) the screen shows "message (count)":
@Composable
fun DetailScreen(message: String, count: Int, onBack: () -> Unit, modifier: Modifier = Modifier) {
    Column(modifier = modifier) {
        Text(text = "$message ($count)")           // → "Hello from Home! (3)"
        Button(onClick = onBack) { Text("Back") }
    }
}

// 5) fix the preview's call site:
DetailScreen(message = "Preview message", count = 1, onBack = {})

This is A2 in action: adding a typed argument to a navigation key. Because DetailKey is a data class, the new count field is type-checked everywhere — the compiler flags the navigation call, the entry block, the screen, and the preview until each passes the new argument.

B3 — grow the model (NavDetailList)10 pts

Add rating: Int to Item, add one new entry, and show the rating on both the Detail screen and each list row. Explain why the navigation code didn't change.

File: NavDetailList-solution/app/src/main/java/com/example/navdetaillist/MainActivity.kt

// 1) add the field:
data class Item(val id: Int, val title: String, val blurb: String, val rating: Int)

// 2) a rating on every row + one NEW entry:
private val sampleItems = listOf(
    Item(1, "Mercury", "The smallest planet…", 3),
    Item(2, "Venus",   "The hottest planet…",  2),
    Item(3, "Earth",   "The only planet…",     5),
    Item(4, "Mars",    "The red planet…",      4),
    Item(5, "Jupiter", "The largest planet…",  5),
    Item(6, "Saturn",  "The ringed gas giant…", 5),   // new entry
)

// 3) Detail screen — show the rating:
Text(text = "Rating: ${item.rating}/5", style = MaterialTheme.typography.bodyMedium)

// 4) each list row — show the rating:
Text(text = "★ ${item.rating}/5", style = MaterialTheme.typography.bodySmall)
Why navigation didn't have to change The key only carries the id (DetailKey(itemId)) and the Detail screen re-resolves the full Item via itemById() from the single data source — the routing never touches Item's shape, so adding a field only affects the data and the screens that render it.
B4 — navigate sideways (NavDetailList)8 pts

On the Detail screen, add a “Next” button that opens the detail of the next item (wrapping from the last back to the first) by pushing another DetailKey.

File: NavDetailList-solution/app/src/main/java/com/example/navdetaillist/MainActivity.kt

// 1) Detail screen gains an onNext callback + a Next button:
@Composable
fun DetailScreen(item: Item, onBack: () -> Unit, onNext: () -> Unit, modifier: Modifier = Modifier) {
    Column(modifier = modifier.fillMaxSize().padding(16.dp)) {
        Text(text = item.title,  style = MaterialTheme.typography.headlineSmall)
        Text(text = item.blurb,  style = MaterialTheme.typography.bodyLarge)
        Text(text = "Rating: ${item.rating}/5")
        Button(onClick = onNext) { Text("Next") }            // ← push the NEXT item's detail
        Button(onClick = onBack) { Text("Back to list") }
    }
}

// 2) the entry computes the next id (wrapping) and pushes a new DetailKey:
entry<DetailKey> { key ->
    DetailScreen(
        item = itemById(key.itemId),
        onBack = { backStack.removeLastOrNull() },
        onNext = {
            val idx  = sampleItems.indexOfFirst { it.id == key.itemId }
            val next = sampleItems[(idx + 1) % sampleItems.size]   // % wraps last → first
            backStack.add(DetailKey(next.id))                      // PUSH another DetailKey
        },
    )
}

// 3) the preview gains a no-op onNext:
DetailScreen(item = sampleItems.first(), onBack = {}, onNext = {})
What you observe After tapping “Next” 3 times you have 5 screens on the stack (List + 4 Details), so pressing Back walks back through every Detail one at a time instead of jumping to the list — because each “Next” pushes a new DetailKey rather than replacing the current one. (If you didn't want that pile-up, you'd pop the current detail before pushing, or replace the top key.)
