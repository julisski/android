Every common on-device store, in one app

How a note reaches storage

Storage Showcase is the capstone teaching app: it demonstrates every common on-device storage technique side by side, one per tab. This page follows a single note — one row you type — all the way through Room into the SQLite file on disk and back to the screen, as blocks of real code in order. Then it shows the exact same write-once / read-as-a-Flow pattern for the app's other three stores: Preferences DataStore, a typed DataStore, and internal files.

The one sentence to hold onto Across all four stores the shape is the same: you write once (an insert, an edit{}, an updateData{}, a file write), and because the read side is a reactive Flow turned into a StateFlow, the screen updates for you — no manual reload anywhere.

The shape of the trip

Tapping Add note sends the note down through the layers onto disk (the write path). Then a single reactive Flow carries the result back up to the screen (the return trip, in green). Initialization · before any data moves

Step 0 — turning four stores on

Three plugins and a handful of dependencies switch on the whole persistence stack: KSP generates Room's SQLite code, the kotlinx.serialization plugin lets the typed DataStore encode JSON, and the DataStore libraries supply both flavors.

build.gradle.kts · plugins
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.compose)
    // KSP runs Room's annotation processor and GENERATES the concrete *_Impl classes
    // (StorageDatabase_Impl / NoteDao_Impl) at build time.
    alias(libs.plugins.ksp)
    // kotlinx.serialization compiler plugin — lets @Serializable data classes (UserProfile)
    // be encoded to JSON, which the TYPED DataStore's Serializer uses.
    alias(libs.plugins.kotlin.serialization)
}
build.gradle.kts · the local persistence stack
// Room: structured/relational storage (notes + categories, with a relation & migration).
implementation(libs.androidx.room.runtime)
implementation(libs.androidx.room.ktx)              // suspend + Flow DAO return types
ksp(libs.androidx.room.compiler)                    // KSP generates the SQLite-backed impls
// DataStore: simple key/value settings (Preferences) + a typed object store (datastore).
implementation(libs.androidx.datastore.preferences)
implementation(libs.androidx.datastore)
// JSON for the typed DataStore's Serializer<UserProfile>.
implementation(libs.kotlinx.serialization.json)
One required flag AGP 9's built-in Kotlin normally forbids adding sources via the kotlin.sourceSets DSL — which is exactly how KSP registers its generated Room code. gradle.properties sets android.disallowKotlinSourceSets=false so the generated code is allowed in; without it assembleDebug fails.

Step 1 · the thing that travels

The element — a note, with a real relationship

The note is a Note @Entity — one row in the notes table. Unlike a flat one-table demo, each note belongs to a Category through a foreign key, so the schema is genuinely relational. Richer Kotlin types (a List<String> of tags, a Priority enum) ride along too — stored through the converters in the next step.

data/Models.kt · Category (the parent)
@Entity(
    tableName = "categories",
    // unique name so OnConflictStrategy.IGNORE makes "create the default category"
    // idempotent — re-inserting "Inbox" on every launch is silently skipped.
    indices = [Index(value = ["name"], unique = true)],
)
data class Category(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val name: String,
)
data/Models.kt · Note (the child)
@Entity(
    tableName = "notes",
    foreignKeys = [
        ForeignKey(
            entity = Category::class,
            parentColumns = ["id"],
            childColumns = ["categoryId"],
            onDelete = ForeignKey.CASCADE,            // deleting a category deletes its notes
        )
    ],
    indices = [Index("categoryId"), Index("title")],  // FK child column SHOULD be indexed
)
data class Note(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val title: String,
    val body: String,
    val done: Boolean = false,                 // INTEGER 0/1
    val categoryId: Long,                      // FK -> categories.id (indexed)
    val priority: Priority = Priority.NORMAL,  // enum -> TEXT via Converters
    val tags: List<String> = emptyList(),      // List -> JSON TEXT via Converters
    val createdAt: Long = 0L,                  // epoch millis (INTEGER); set by the ViewModel
)
data/Models.kt · CategoryWithNotes (a @Relation read object)
data class CategoryWithNotes(
    @Embedded val category: Category,
    @Relation(parentColumn = "id", entityColumn = "categoryId")
    val notes: List<Note>,                     // Room runs a 2nd query to fill this
)
Why index the foreign-key column Room warns if you don't. Without @Index("categoryId"), every parent delete or update does a full table scan of notes to find the children. The index turns that into a fast lookup.

Step 2 · making non-primitives storable

Type converters — List and enum into a column

SQLite stores only TEXT / INTEGER / REAL / BLOB / NULL. A List<String> and a Priority enum are richer than that, so a @TypeConverter pair turns each into a supported type and back. Room calls them automatically whenever it reads or writes those columns.

data/Converters.kt
class Converters {

    // --- List<String> <-> JSON TEXT --------------------------------------
    @TypeConverter
    fun fromTags(tags: List<String>): String = Json.encodeToString(tags)

    @TypeConverter
    fun toTags(json: String): List<String> =
        if (json.isBlank()) emptyList() else Json.decodeFromString(json)

    // --- Priority enum <-> TEXT (store the NAME) -------------------------
    @TypeConverter
    fun fromPriority(p: Priority): String = p.name

    @TypeConverter
    fun toPriority(value: String): Priority =
        runCatching { Priority.valueOf(value) }.getOrDefault(Priority.NORMAL)
}
Store the name, never the ordinal fromPriority persists p.name ("HIGH"), not p.ordinal (2). If you stored the ordinal and later reordered the enum, every stored value would silently point at the wrong constant. toPriority also falls back to NORMAL on unknown text, so a bad value can never crash a read — exactly what ConvertersTest.kt verifies.

Step 3 · the door

The DAO — reactive reads, suspend writes

A DAO is an interface of read/write methods; Room generates the implementation and checks your SQL at compile time. Reads return a Flow (live: re-emits on every change); writes are suspend (Room runs them off the main thread). One read uses @Transaction to assemble the relation as a single consistent snapshot.

data/NoteDao.kt
@Dao
interface NoteDao {

    // --- reactive reads (Flow) -------------------------------------------
    @Query("SELECT * FROM notes ORDER BY id DESC")
    fun observeNotes(): Flow<List<Note>>

    @Query("SELECT * FROM notes ORDER BY title COLLATE NOCASE ASC")
    fun observeNotesByTitle(): Flow<List<Note>>

    /** A relation read — one category + its notes. @Transaction = consistent snapshot. */
    @Transaction
    @Query("SELECT * FROM categories ORDER BY name COLLATE NOCASE ASC")
    fun observeCategoriesWithNotes(): Flow<List<CategoryWithNotes>>

    @Query("SELECT COUNT(*) FROM notes")
    fun observeCount(): Flow<Int>

    // --- suspend writes (off the main thread, on Room's executor) --------
    @Insert
    suspend fun insert(note: Note): Long                 // returns new rowid

    @Update
    suspend fun update(note: Note)

    @Delete
    suspend fun delete(note: Note)

    /** IGNORE: a category with a duplicate name is skipped (returns -1). */
    @Insert(onConflict = OnConflictStrategy.IGNORE)
    suspend fun insertCategory(category: Category): Long
}
⬇ The write

insert is suspend: Room runs the disk write on its own executor, so you call it from a coroutine and never freeze the UI. It returns the new id straight from SQLite.

⬆ The reactive reads

Two ordered Flow queries (newest-first and title A–Z) back the swappable sort order; observeCount() is a live COUNT(*). Subscribe once; Room re-emits on every table change.

Step 4 · the store, and evolving it

The database — holder, singleton, and a real migration

The abstract @Database ties the entities and DAO to one SQLite file (storage.db), registers the converters with @TypeConverters, and is built once as a thread-safe singleton. It is version = 2 and ships a migration — the part most demos skip.

data/StorageDatabase.kt
@Database(
    entities = [Note::class, Category::class],
    version = 2,
    exportSchema = false,
)
@TypeConverters(Converters::class)                 // register the List/enum converters
abstract class StorageDatabase : RoomDatabase() {

    abstract fun noteDao(): NoteDao

    companion object {
        @Volatile private var INSTANCE: StorageDatabase? = null

        /** v1 -> v2: add the `tags` column. ALTER TABLE needs a default for a NOT NULL column. */
        val MIGRATION_1_2 = object : Migration(1, 2) {
            override fun migrate(db: SupportSQLiteDatabase) {
                db.execSQL("ALTER TABLE notes ADD COLUMN tags TEXT NOT NULL DEFAULT '[]'")
            }
        }

        fun getInstance(context: Context): StorageDatabase =
            INSTANCE ?: synchronized(this) {
                INSTANCE ?: Room.databaseBuilder(
                    context.applicationContext,
                    StorageDatabase::class.java,
                    "storage.db",
                )
                    .addMigrations(MIGRATION_1_2)   // wired for v1 -> v2 upgraders
                    .build()
                    .also { INSTANCE = it }
            }
    }
}
Shipping the migration is mandatory A fresh install is created at v2 directly, so MIGRATION_1_2 only runs for users upgrading from v1. But you must still ship it: with no migration path Room throws IllegalStateException("A migration from 1 to 2 was required but not found") and the app crashes on launch for everyone who had v1.

Store 2 of 4 · simple settings

Preferences DataStore — the Settings tab

The modern replacement for SharedPreferences for a few simple settings. One file-scoped DataStore exposes typed keys as a reactive Flow (reads) and a transactional edit { } (writes). Same write-once / read-as-a-Flow shape as Room — just key/value instead of rows.

data/SettingsRepository.kt
// One app-wide DataStore named "settings", created lazily at file scope.
private val Context.settingsDataStore: DataStore<Preferences> by preferencesDataStore(name = "settings")

class SettingsRepository(context: Context) {

    private val dataStore = context.applicationContext.settingsDataStore

    private companion object {
        val KEY_DARK_THEME = booleanPreferencesKey("dark_theme")
        val KEY_SORT_ORDER = stringPreferencesKey("sort_order")
    }

    // READ: a reactive Flow derived from .data
    val darkTheme: Flow<Boolean> = dataStore.data.map { it[KEY_DARK_THEME] ?: false }

    // WRITE: a suspend, transactional edit { }
    suspend fun setDarkTheme(enabled: Boolean) {
        dataStore.edit { it[KEY_DARK_THEME] = enabled }
    }

    suspend fun setSortOrder(order: SortOrder) {
        dataStore.edit { it[KEY_SORT_ORDER] = order.name }   // store NAME, never ordinal
    }
}
One instance per file The preferencesDataStore(name = "settings") delegate is declared once at file scope. Creating a second DataStore for the same file throws IllegalStateException — so it lives outside the class as a single app-wide instance.

Store 3 of 4 · one typed object

Typed DataStore — the Profile tab

When settings are really one structured object, a typed DataStore beats a bag of loose keys: you get a schema, real defaults, and compile-time type safety. You supply a Serializer<T>; here it serializes a @Serializable UserProfile to JSON.

data/ProfileRepository.kt · the object + serializer
@kotlinx.serialization.Serializable
data class UserProfile(
    val displayName: String = "Guest",
    val accentSeed: Int = 0,
    val notesCreated: Int = 0,
)

object UserProfileSerializer : Serializer<UserProfile> {
    override val defaultValue = UserProfile()        // used when the file doesn't exist yet

    override suspend fun readFrom(input: InputStream): UserProfile =
        try {
            Json.decodeFromString(UserProfile.serializer(), input.readBytes().decodeToString())
        } catch (e: SerializationException) {
            throw CorruptionException("Cannot read UserProfile JSON.", e)
        }

    override suspend fun writeTo(t: UserProfile, output: OutputStream) {
        output.write(Json.encodeToString(UserProfile.serializer(), t).encodeToByteArray())
    }
}
data/ProfileRepository.kt · read & write
private val Context.profileDataStore: DataStore<UserProfile> by dataStore(
    fileName = "user_profile.json",
    serializer = UserProfileSerializer,
)

class ProfileRepository(context: Context) {
    private val dataStore = context.applicationContext.profileDataStore

    val profile: Flow<UserProfile> = dataStore.data          // reactive, type-safe read

    suspend fun setDisplayName(name: String) {
        dataStore.updateData { it.copy(displayName = name.ifBlank { "Guest" }) }
    }

    suspend fun incrementNotesCreated() {                    // called every time a note is added
        dataStore.updateData { it.copy(notesCreated = it.notesCreated + 1) }
    }
}
Two stores update on one tap When you add a note, the ViewModel calls dao.insert(...) and profileRepo.incrementNotesCreated() — so a single user action writes a Room row and bumps a field in the typed DataStore. Watch notesCreated climb on the Profile tab.

Store 4 of 4 · free-form bytes

Internal files — the Files tab

For blobs rather than rows or key/value, write straight to internal app storage. filesDir is persistent and private; cacheDir is re-creatable scratch the system may evict. Unlike Room and DataStore, raw java.io calls are blocking and are not auto-moved off the main thread — so the ViewModel wraps them in withContext(Dispatchers.IO).

data/FileStore.kt
class FileStore(context: Context) {

    private val app = context.applicationContext
    private val scratchFile: File get() = File(app.filesDir, "scratch.txt")

    /** Read the persistent scratch note (empty if never written). */
    fun readScratch(): String = if (scratchFile.exists()) scratchFile.readText() else ""

    /** Persist the scratch note to filesDir (survives restart, gone on uninstall). */
    fun writeScratch(text: String) { scratchFile.writeText(text) }

    /** Write a throwaway cache file the system is free to evict. */
    fun writeCacheBlob(): File =
        File.createTempFile("blob", ".tmp", app.cacheDir).apply { writeText("cacheable scratch data") }

    /** Clear all cache files (what "Clear cache" in system settings does). */
    fun clearCache() { app.cacheDir.listFiles()?.forEach { it.delete() } }
}

Step 5 · the conductor

The ViewModel — every store becomes a StateFlow

StorageViewModel is where each persistence Flow becomes observable UI state via stateIn(...), and where user intents become writes on viewModelScope. Two details are worth slowing down on: flatMapLatest (which swaps the Room query when the persisted sort order changes) and the deliberate asymmetry in threading.

Reads: persistence Flows → UI StateFlows

StorageViewModel.kt · notes (with flatMapLatest)
// The sort-order setting (from Preferences DataStore) SWAPS which Room query feeds the list.
val notes: StateFlow<List<Note>> = settings.sortOrder
    .flatMapLatest { order ->
        when (order) {
            SortOrder.NEWEST_FIRST -> dao.observeNotes()
            SortOrder.TITLE_ASC    -> dao.observeNotesByTitle()
        }
    }
    .stateIn(viewModelScope, started, emptyList())

val darkTheme: StateFlow<Boolean> = settings.darkTheme.stateIn(viewModelScope, started, false)
val profile: StateFlow<UserProfile> = profileRepo.profile.stateIn(viewModelScope, started, UserProfile())

Writes: intents on viewModelScope

StorageViewModel.kt · addNote()
fun addNote(title: String, body: String, categoryId: Long, tags: List<String>, priority: Priority) {
    val clean = title.trim()
    if (clean.isEmpty()) return
    viewModelScope.launch {
        dao.insert(
            Note(
                title = clean, body = body.trim(), categoryId = categoryId,
                tags = tags, priority = priority,
                createdAt = System.currentTimeMillis(),   // stamp metadata here
            )
        )
        profileRepo.incrementNotesCreated()   // also bumps the typed profile object
    }
}

// Raw FILE I/O, by contrast, IS wrapped in IO (Room/DataStore are not):
fun saveScratch() {
    viewModelScope.launch {
        withContext(Dispatchers.IO) { fileStore.writeScratch(_scratch.value) }
        refreshFileInfo()
    }
}
Why the threading is asymmetric Room suspend/Flow work runs on Room's own executor, and DataStore does its own IO — so neither is wrapped in withContext(Dispatchers.IO). Raw java.io file calls are plain blocking calls, so those are wrapped. Knowing which APIs are already main-safe is half of using them correctly.

Step 6 · where it starts and ends on screen

The screen — four tabs, one StateFlow pattern

MainActivity is a single screen with a bottom navigation bar — one tab per mechanism. Every tab reads ViewModel StateFlows with collectAsStateWithLifecycle() and shows the exact on-disk file its data is saved to, so the abstraction stays concrete.

MainActivity.kt · the tabs + theme driven by the persisted flag
setContent {
    val vm: StorageViewModel = viewModel(factory = StorageViewModel.factory(application))
    // Drive the theme from the PERSISTED Preferences-DataStore dark-theme flag.
    val darkTheme by vm.darkTheme.collectAsStateWithLifecycle()
    StorageShowcaseTheme(darkTheme = darkTheme, dynamicColor = false) {
        StorageShowcaseApp(vm)            // Scaffold + NavigationBar: Notes / Settings / Profile / Files
    }
}
MainActivity.kt · NotesTab — capture, write, render
val notes by vm.notes.collectAsStateWithLifecycle()        // live rows from Room

Button(
    onClick = {
        val catId = selectedCategoryId ?: return@Button
        val tags = tagsText.split(",").map { it.trim() }.filter { it.isNotEmpty() }
        vm.addNote(title, body, catId, tags, priority)     // -> ViewModel -> Room insert
        title = ""; body = ""; tagsText = ""               // clear local input
    },
    enabled = title.isNotBlank() && selectedCategoryId != null,
) { Text("Add note (Room INSERT)") }

LazyColumn {
    items(notes, key = { it.id }) { note ->                // re-renders when the Flow re-emits
        NoteCard(note = note, /* … */)
    }
}
"Saved to ·" is shown on every tab A small SavedToBanner prints the real absolute path (resolved by StorageLocations.kt) at the top of each tab — so learners see precisely where each store writes: databases/storage.db, datastore/settings.preferences_pb, datastore/user_profile.json, files/scratch.txt.

The end of the journey

What survives — per store

Everything except the cache lives in the app's private internal storage, so it outlives in-memory state. The one deliberate exception is cacheDir: the system or the user may evict it at any time, which is exactly why you never put anything important there.

Re-open = the read path, alone On relaunch no write happens, yet every tab fills in. That is the return trip running by itself: each screen subscribes, the Flows emit the persisted state, and collectAsStateWithLifecycle() turns each into a recomposition. The dark-theme you chose is already applied before the first frame, because the theme itself reads the persisted flag.
