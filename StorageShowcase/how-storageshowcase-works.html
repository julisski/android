Project walkthrough

How Storage Showcase works

Storage Showcase is a single Jetpack Compose teaching app that demonstrates every common on-device storage technique in one place. It has four tabs, one storage mechanism each, and everything you change survives an app restart because it is written to the device. This page is a file-by-file walkthrough of how the project is built and how data flows from a tap on the screen down to the disk and reactively back up again.

Read this before the code The app is not saying "always use all four." It is comparing four separate choices: Room for structured records, Preferences DataStore for simple settings, Typed DataStore for one schema-backed object, and filesDir/cacheDir for private bytes. Pick one based on the shape of the data.

The one idea to take away Each tab writes to a different place on disk, but they all reach the screen the same way: a persistence Flow → a UI StateFlow → collectAsStateWithLifecycle() → recomposition. Pick the storage that matches your data shape; the reactive plumbing above it never changes.

The shape of the app

Architecture: five layers, one direction

The app is a clean, unidirectional stack. The UI never touches a database or a file directly — it only reads StateFlows and calls functions on the ViewModel. The ViewModel never formats anything for the screen — it just turns persistence into observable state and forwards user intents to the repositories. Reads flow up; user actions flow down.

The crown jewel

The reactive spine

This is the single most important pattern in the project. Once you understand it, all four tabs read the same way. Two journeys matter: how data on disk reaches the screen (the read path), and how a tap persists data and the screen updates without anyone telling it to (the write path).

⬆ Read path — disk → screen
  1. A storage engine exposes a Flow that re-emits whenever its data changes (Room watches the table; DataStore watches the file).
  2. The repository/DAO surfaces that Flow (e.g. dao.observeNotes(), settings.darkTheme).
  3. The ViewModel converts it to a hot, cached StateFlow with stateIn(viewModelScope, …, initialValue) — so it always has a value and is shared across collectors.
  4. A composable reads it with collectAsStateWithLifecycle(), which only collects while the screen is visible.
  5. A new emission flips the backing Compose State, which recomposes just the parts that read it.
⬇ Write path — tap → disk → back up
  1. The UI calls a plain function on the ViewModel, e.g. vm.addNote(…).
  2. The ViewModel launches a coroutine on viewModelScope and calls a suspend write on the repository/DAO.
  3. The write lands on disk (a Room INSERT, a DataStore edit{}/updateData{}, or a file write on Dispatchers.IO).
  4. That write makes the engine’s Flow re-emit — the same Flow the read path is already collecting.
  5. The StateFlow updates, the screen recomposes. The UI is never updated by hand.

Why flatMapLatest and not map The sort setting doesn’t transform a value — it chooses a different ongoing Flow (a different live query). flatMapLatest subscribes to the newest inner Flow and cancels the previous one, so you’re never accidentally listening to both the old and new query at once.

Mechanism 1 · the Notes tab

Room — structured, relational storage

SQLite@Entity@Dao @Relation@TypeConverterMigration

Room is an object-mapping layer over SQLite. You describe tables as annotated data classes, queries as annotated interface methods, and Room generates the SQLite-backed implementation at build time (that is what KSP does). The Notes tab uses Room because notes are structured records with a relationship and queries — exactly what a relational store is for.

Tables: two entities with a real relationship

Category is the parent; Note is the child and references its category through a foreign key. The non-primitive columns (the tag list and the priority enum) are handled by type converters.

data/Models.kt
enum class Priority { LOW, NORMAL, HIGH }

@Entity(
    tableName = "notes",
    foreignKeys = [
        ForeignKey(
            entity = Category::class,
            parentColumns = ["id"],
            childColumns = ["categoryId"],
            onDelete = ForeignKey.CASCADE,   // delete a category -> its notes go too
        )
    ],
    indices = [Index("categoryId"), Index("title")],
)
data class Note(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val title: String,
    val body: String,
    val done: Boolean = false,                 // INTEGER 0/1
    val categoryId: Long,                      // FK -> categories.id (indexed)
    val priority: Priority = Priority.NORMAL,  // enum -> TEXT via Converters
    val tags: List<String> = emptyList(),      // List -> JSON TEXT via Converters
    val createdAt: Long = 0L,                  // epoch millis; set by the ViewModel
)
  • @ForeignKey … onDelete = CASCADE enforces referential integrity in SQLite: a note’s categoryId must point at an existing category, and deleting a category deletes its notes automatically.
  • @Index("categoryId") indexes the foreign-key column. Room warns if you don’t — without it, every parent delete/update does a full table scan of notes.
  • The enum is stored by name and the list as JSON — neither is a primitive SQLite column, so they need converters (next).

The @Relation read object

To read “a category and all its notes” in one shot, Room fills a POJO with a second query. The DAO method that returns it must be @Transaction so parent and children are one consistent snapshot.

data/Models.kt
data class CategoryWithNotes(
    @Embedded val category: Category,
    @Relation(parentColumn = "id", entityColumn = "categoryId")
    val notes: List<Note>,
)
Honest caveat: demonstrated, not yet wired CategoryWithNotes — and the DAO methods observeCategoriesWithNotes() and findCategoryByName() — are included to teach the @Relation pattern, but the current UI doesn’t call them: the Notes tab reads notes and categories as two flat lists. They’re a worked example to wire up, not part of the live data flow.

Type converters: teaching SQLite about richer types

SQLite only stores TEXT / INTEGER / REAL / BLOB / NULL. A @TypeConverter is a pair of functions Room calls to map your type to a supported one and back.

data/Converters.kt
class Converters {
    // List<String> <-> JSON TEXT
    @TypeConverter fun fromTags(tags: List<String>): String = Json.encodeToString(tags)
    @TypeConverter fun toTags(json: String): List<String> =
        if (json.isBlank()) emptyList() else Json.decodeFromString(json)

    // Priority enum <-> TEXT — store the NAME, never the ordinal
    @TypeConverter fun fromPriority(p: Priority): String = p.name
    @TypeConverter fun toPriority(value: String): Priority =
        runCatching { Priority.valueOf(value) }.getOrDefault(Priority.NORMAL)
}
Why name, not ordinal Storing the enum’s ordinal (its position) would silently corrupt every saved row the day someone reorders or inserts an enum constant. Storing p.name is stable across source changes, and toPriority falls back to NORMAL for unknown text instead of crashing.

The DAO: reactive reads, suspend writes

A @Dao is an interface — Room generates the implementation. The return type decides behaviour: Flow is a live, re-emitting query; suspend is a one-shot that runs off the main thread on Room’s own executor.

data/NoteDao.kt
@Dao
interface NoteDao {
    // reactive reads — re-emit whenever the table changes
    @Query("SELECT * FROM notes ORDER BY id DESC")
    fun observeNotes(): Flow<List<Note>>

    @Query("SELECT * FROM notes ORDER BY title COLLATE NOCASE ASC")
    fun observeNotesByTitle(): Flow<List<Note>>

    @Query("SELECT COUNT(*) FROM notes")
    fun observeCount(): Flow<Int>

    @Transaction
    @Query("SELECT * FROM categories ORDER BY name COLLATE NOCASE ASC")
    fun observeCategoriesWithNotes(): Flow<List<CategoryWithNotes>>

    // suspend writes — one-shot, off the main thread
    @Insert suspend fun insert(note: Note): Long
    @Update suspend fun update(note: Note)
    @Delete suspend fun delete(note: Note)

    // duplicate category name is silently skipped (returns -1)
    @Insert(onConflict = OnConflictStrategy.IGNORE)
    suspend fun insertCategory(category: Category): Long
}

The database holder, singleton, and a real migration

The abstract @Database ties the entities + DAO to one SQLite file (storage.db), registers the converters, and is built once as a thread-safe singleton.

data/StorageDatabase.kt
@Database(entities = [Note::class, Category::class], version = 2, exportSchema = false)
@TypeConverters(Converters::class)
abstract class StorageDatabase : RoomDatabase() {
    abstract fun noteDao(): NoteDao

    companion object {
        @Volatile private var INSTANCE: StorageDatabase? = null

        // v1 -> v2: add the `tags` column. ALTER TABLE ADD COLUMN needs a default
        // for a NOT NULL column, so existing rows decode to an empty tag list.
        val MIGRATION_1_2 = object : Migration(1, 2) {
            override fun migrate(db: SupportSQLiteDatabase) {
                db.execSQL("ALTER TABLE notes ADD COLUMN tags TEXT NOT NULL DEFAULT '[]'")
            }
        }

        fun getInstance(context: Context): StorageDatabase =
            INSTANCE ?: synchronized(this) {
                INSTANCE ?: Room.databaseBuilder(
                    context.applicationContext, StorageDatabase::class.java, "storage.db",
                ).addMigrations(MIGRATION_1_2).build().also { INSTANCE = it }
            }
    }
}
Why shipping the migration is mandatory A fresh install is created at version 2 directly, so MIGRATION_1_2 only runs for users upgrading from a v1 build. But you must still ship it: if a v1 user updates and no migration path exists, Room throws IllegalStateException (“A migration from 1 to 2 was required but not found”) and the app crashes on launch.
The @Volatile + synchronized double-check getInstance is the classic thread-safe singleton: check the volatile field, and only if it’s null take the lock and check again, so two threads racing on first launch can’t build two databases pointing at the same file.

Mechanism 2 · the Settings tab

Preferences DataStore — simple key/value settings

key/valueFlow readsedit { } writesreplaces SharedPreferences

For a handful of loose settings, Preferences DataStore is the modern replacement for SharedPreferences: it’s fully asynchronous (no apply()/commit() on the main thread), reads are a reactive Flow, and writes are transactional. The Settings tab stores two things — the dark-theme flag and the notes sort order.

data/SettingsRepository.kt
enum class SortOrder { NEWEST_FIRST, TITLE_ASC }

// one app-wide DataStore named "settings", created lazily at FILE scope
private val Context.settingsDataStore: DataStore<Preferences>
    by preferencesDataStore(name = "settings")

class SettingsRepository(context: Context) {
    private val dataStore = context.applicationContext.settingsDataStore

    private companion object {
        val KEY_DARK_THEME = booleanPreferencesKey("dark_theme")
        val KEY_SORT_ORDER = stringPreferencesKey("sort_order")
    }

    // reactive reads
    val darkTheme: Flow<Boolean> = dataStore.data.map { it[KEY_DARK_THEME] ?: false }
    val sortOrder: Flow<SortOrder> = dataStore.data.map { prefs ->
        runCatching { SortOrder.valueOf(prefs[KEY_SORT_ORDER] ?: SortOrder.NEWEST_FIRST.name) }
            .getOrDefault(SortOrder.NEWEST_FIRST)
    }

    // transactional writes
    suspend fun setDarkTheme(enabled: Boolean) { dataStore.edit { it[KEY_DARK_THEME] = enabled } }
    suspend fun setSortOrder(order: SortOrder) { dataStore.edit { it[KEY_SORT_ORDER] = order.name } }
}
  • Typed keys. booleanPreferencesKey(…) / stringPreferencesKey(…) tie a name to a type, so reads and writes can’t disagree about what a key holds.
  • The by preferencesDataStore delegate is file-scoped on purpose. One instance per file for the whole process — creating a second DataStore for the same file throws IllegalStateException. That’s why it’s a top-level Context extension, not something you new up per screen.
  • Defaults live in the read. ?: false and the runCatching … getOrDefault mean a missing or unparseable value yields a sane default rather than a crash — and again the enum is stored by name.
This is half of the sort-swap trace setSortOrder is step 3 of the reactive-spine trace; the sortOrder Flow here is what the ViewModel’s flatMapLatest listens to. One write, and Room’s query swaps — across two different storage mechanisms.

Mechanism 3 · the Profile tab

Typed DataStore — one object with a schema

@SerializableSerializer<T>JSONupdateData { }

When your settings are really one typed object, a typed DataStore beats a bag of loose keys. You supply a Serializer<T> that turns the object into bytes and back — here, JSON via kotlinx.serialization. Reads are a Flow<UserProfile> with real field defaults; writes are the transactional updateData { }.

data/ProfileRepository.kt
@Serializable
data class UserProfile(
    val displayName: String = "Guest",
    val accentSeed: Int = 0,
    val notesCreated: Int = 0,
)

object UserProfileSerializer : Serializer<UserProfile> {
    override val defaultValue = UserProfile()   // returned when the file doesn't exist yet

    override suspend fun readFrom(input: InputStream): UserProfile =
        try {
            Json.decodeFromString(UserProfile.serializer(), input.readBytes().decodeToString())
        } catch (e: SerializationException) {
            throw CorruptionException("Cannot read UserProfile JSON.", e)
        }

    override suspend fun writeTo(t: UserProfile, output: OutputStream) {
        output.write(Json.encodeToString(UserProfile.serializer(), t).encodeToByteArray())
    }
}

private val Context.profileDataStore: DataStore<UserProfile>
    by dataStore(fileName = "user_profile.json", serializer = UserProfileSerializer)

class ProfileRepository(context: Context) {
    private val dataStore = context.applicationContext.profileDataStore
    val profile: Flow<UserProfile> = dataStore.data            // reactive, type-safe read

    suspend fun setDisplayName(name: String) =
        dataStore.updateData { it.copy(displayName = name.ifBlank { "Guest" }) }
    suspend fun incrementNotesCreated() =
        dataStore.updateData { it.copy(notesCreated = it.notesCreated + 1) }
}
  • Type safety end to end. You read and write a real UserProfile, not get("displayName") string lookups. The compiler catches typos and type mismatches.
  • updateData { it.copy(...) } is read-modify-write in one transaction. incrementNotesCreated reads the current count and writes count + 1 atomically — no lost updates if two writes race.
  • CorruptionException is the contract for “this file is unreadable.” Throwing it (instead of letting a random exception escape) lets DataStore apply its corruption-handling policy rather than crashing.
Preferences vs. Typed — same library, different fit Both are DataStore and both expose a Flow. Reach for Preferences when you have a few independent flags; reach for Typed when the data is a cohesive object with a schema and invariants. Profile is one object (displayName, accentSeed, notesCreated), so it’s typed.

Mechanism 4 · the Files tab

Internal files — free-form blobs

filesDircacheDirjava.ioDispatchers.IO (manual)

For bytes or text that aren’t rows or key/value pairs, write straight to internal storage. The Files tab keeps a persistent scratch note in filesDir and a throwaway blob in cacheDir to show the difference. The FileStore itself is plain, blocking java.io — which is the key teaching point.

data/FileStore.kt
class FileStore(context: Context) {
    private val app = context.applicationContext
    private val scratchFile: File get() = File(app.filesDir, "scratch.txt")

    fun readScratch(): String = if (scratchFile.exists()) scratchFile.readText() else ""
    fun writeScratch(text: String) { scratchFile.writeText(text) }        // survives restart, gone on uninstall

    fun writeCacheBlob(): File =
        File.createTempFile("blob", ".tmp", app.cacheDir).apply { writeText("cacheable scratch data") }

    fun clearCache() { app.cacheDir.listFiles()?.forEach { it.delete() } }  // what "Clear cache" does

    fun info(): FileInfo = FileInfo(
        path = scratchFile.absolutePath,
        scratchBytes = if (scratchFile.exists()) scratchFile.length() else 0L,
        cacheBytes = app.cacheDir.listFiles()?.sumOf { it.length() } ?: 0L,
    )
}
The one place you must thread by hand Room and DataStore move work off the main thread for you. Raw java.io calls are blocking and are not — so the ViewModel wraps every FileStore call in withContext(Dispatchers.IO). Forgetting that is how you get jank or an ANR.
filesDir

Persistent, private, no permission, encrypted on Android 10+. Removed only on uninstall (and is included in Auto Backup). The scratch note lives here.

cacheDir

Re-creatable scratch space. The system may evict it under storage pressure and the user can clear it anytime — so always existence-check before reading. Never included in Auto Backup.

Showing the learner exactly where data goes

A small helper resolves the real absolute path of every store, so each tab can print “💾 Saved to …”. It’s the concrete payoff of the whole project — you can see the four different files.

The glue · part 1

StorageViewModel — where persistence becomes UI state

One AndroidViewModel owns all four data sources and exposes them to the UI as StateFlows. It needs the Application context to build the database and the DataStores, which is exactly what AndroidViewModel provides. It survives configuration changes (rotation), so the live queries aren’t torn down and rebuilt every rotate.

StorageViewModel.kt — construction & reads
class StorageViewModel(app: Application) : AndroidViewModel(app) {
    private val dao = StorageDatabase.getInstance(app).noteDao()
    private val settings = SettingsRepository(app)
    private val profileRepo = ProfileRepository(app)
    private val fileStore = FileStore(app)

    val paths: StoragePaths = StoragePaths.of(app)            // shown in the UI
    private val started = SharingStarted.WhileSubscribed(5_000)

    // Room — the sort SETTING chooses which live query feeds the list
    val notes: StateFlow<List<Note>> = settings.sortOrder
        .flatMapLatest { order ->
            when (order) {
                SortOrder.NEWEST_FIRST -> dao.observeNotes()
                SortOrder.TITLE_ASC    -> dao.observeNotesByTitle()
            }
        }
        .stateIn(viewModelScope, started, emptyList())

    val darkTheme: StateFlow<Boolean>     = settings.darkTheme.stateIn(viewModelScope, started, false)
    val sortOrder: StateFlow<SortOrder>   = settings.sortOrder.stateIn(viewModelScope, started, NEWEST_FIRST)
    val profile:   StateFlow<UserProfile> = profileRepo.profile.stateIn(viewModelScope, started, UserProfile())
    val noteCount: StateFlow<Int>         = dao.observeCount().stateIn(viewModelScope, started, 0)
}
  • stateIn(scope, started, initial) converts a cold Flow into a hot StateFlow: it always has a current value (the initial until the first real emission) and multiplexes one upstream collection to many UI collectors.
  • SharingStarted.WhileSubscribed(5_000) keeps the upstream query alive for 5 seconds after the last collector leaves. That window means a quick rotation or tab switch reuses the existing query instead of tearing it down and restarting it.
  • flatMapLatest is the only place two mechanisms meet — a Preferences value selects a Room query. Everything else is a straight repository Flow → stateIn.

Writes: launched on viewModelScope, persisted, never hand-applied

Every user action is a plain function that fires a coroutine and calls a suspend write. Note how a single intent can touch two stores, and how file I/O is the only one wrapped in Dispatchers.IO.

StorageViewModel.kt — writes
fun addNote(title: String, body: String, categoryId: Long, tags: List<String>, priority: Priority) {
    val clean = title.trim()
    if (clean.isEmpty()) return
    viewModelScope.launch {
        dao.insert(Note(title = clean, body = body.trim(), categoryId = categoryId,
                        tags = tags, priority = priority, createdAt = System.currentTimeMillis()))
        profileRepo.incrementNotesCreated()      // also bumps the typed profile object
    }
}

fun cycleSortOrder() = viewModelScope.launch {
    settings.setSortOrder(
        if (sortOrder.value == SortOrder.NEWEST_FIRST) SortOrder.TITLE_ASC else SortOrder.NEWEST_FIRST
    )
}

// raw file I/O — the ONE thing that must be moved off the main thread by hand
fun saveScratch() = viewModelScope.launch {
    withContext(Dispatchers.IO) { fileStore.writeScratch(_scratch.value) }
    refreshFileInfo()
}
Note vs. file state The Room/DataStore values are derived from disk via stateIn, so there’s no local copy to keep in sync. The scratch text is different: it’s an editable draft, so the ViewModel holds it in a MutableStateFlow (_scratch) that updates as you type and is only flushed to disk on “Save”.

Constructing it: a tiny factory

Because the ViewModel needs the Application, it’s created through a one-method ViewModelProvider.Factory — no DI framework required.

StorageViewModel.kt — factory
companion object {
    fun factory(app: Application): ViewModelProvider.Factory = object : ViewModelProvider.Factory {
        @Suppress("UNCHECKED_CAST")
        override fun <T : ViewModel> create(modelClass: Class<T>): T = StorageViewModel(app) as T
    }
}

The glue · part 2

The UI layer — one screen, four tabs

MainActivity is the whole UI. It obtains the ViewModel, wraps everything in the app theme (driven by the persisted dark-mode flag), and renders a Scaffold with a bottom NavigationBar. Each tab is a composable that only reads StateFlows and calls ViewModel functions — it knows nothing about Room, DataStore, or files.

MainActivity.kt — entry point
class MainActivity : ComponentActivity() {

        enableEdgeToEdge()

            StorageShowcaseTheme(darkTheme = darkTheme, dynamicColor = false) {
                StorageShowcaseApp(vm)
            }
        }
    }

Why collectAsStateWithLifecycle()

It collects the StateFlow only while the screen is at least STARTED, and pauses when the app is backgrounded — so a database that keeps changing in the background won’t needlessly recompose an invisible UI or keep work alive. Pair it with WhileSubscribed in the ViewModel and the upstream query stops too.

Theming closes the loop

StorageShowcaseTheme takes the darkTheme boolean read from the Settings store. Toggle the switch → DataStore write → darkTheme StateFlow emits → the theme recomposes the whole app — and it’s still dark after a restart. dynamicColor = false keeps the dark/light contrast obvious on Android 12+.

Where the “Saved to …” banner comes from Each tab shows vm.paths.roomDb / .preferences / .profile / .scratchFile in a monospace banner. That’s StorageLocations surfaced through the ViewModel — the app literally points at the file it’s writing.

Two gotchas worth knowing

KSP version is compound. 2.2.10-2.0.2 — the prefix must equal the Kotlin version; the suffix is KSP’s own release for that Kotlin. A mismatch fails the build.

The gradle.properties line. AGP 9’s built-in Kotlin forbids adding sources via the kotlin.sourceSets DSL — but that’s exactly how KSP registers Room’s generated code. So the project sets android.disallowKotlinSourceSets=false; without it, assembleDebug fails.

Build scaffolding worth a glance The wrapper pins Gradle 9.4.1 and the build daemon runs on a JDK 21 toolchain (gradle-daemon-jvm.properties), even though the app compiles to Java 17 bytecode. R8 keep rules now live in src/main/keepRules/rules.keep (the AGP 9 replacement for proguard-rules.pro), and room-testing is on the classpath for a future MigrationTestHelper test that isn’t written yet — today the only test is the JVM ConvertersTest. (The leftover template res/values/colors.xml palette is unused; the Compose theme uses ui/theme/Color.kt.)

What survives what

The whole point of the app is that changes persist. But “persist” has degrees — here’s exactly which store survives which event. (Auto Backup applies because the manifest sets allowBackup="true" with default rules.)

Test & run

The converters are pure logic, so they get a fast JVM unit test (no device, no Room). It guards the round-trips that everything stored in Room depends on.

src/test/…/ConvertersTest.kt
class ConvertersTest {
    private val c = Converters()

    @Test fun tags_roundTrip() {
        val tags = listOf("home", "urgent", "with,comma")     // comma inside a tag survives JSON
        assertEquals(tags, c.toTags(c.fromTags(tags)))
    }

    @Test fun priority_roundTripByName() {
        Priority.entries.forEach { p -> assertEquals(p, c.toPriority(c.fromPriority(p))) }
        assertEquals(Priority.NORMAL, c.toPriority("NOPE"))   // unknown text -> NORMAL, never crashes
    }
}

The mental model to keep Choose the store by the shape of the data — relational rows (Room), loose settings (Preferences), one typed object (typed DataStore), or raw bytes (files). The reactive plumbing above all four is identical: a persistence Flow becomes a StateFlow becomes recomposition, and every write goes to disk first so the screen can never disagree with what’s stored.
