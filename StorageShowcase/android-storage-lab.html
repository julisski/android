Start here - no experience needed

The simplest version: where can an app keep things?

An app often needs to remember things - a setting you picked, a note you typed, a photo you took. There is not one big box for all of it. Instead, Android gives an app a few different "places" to keep data, and each place is good for a different kind of thing - a bit like how, at home, you keep your keys, your photo albums, and your scratch paper in different spots. Here are the main places, in plain words.

If you only remember one sentence: use DataStore for small settings, Room for lists of records you query, filesDir for private files that must last, cacheDir for files you can rebuild, and MediaStore or the system picker when the user should see, keep, or choose the file.
First, what does "volatile" mean? Volatile data disappears the moment the power goes off - that is the phone's memory (RAM), where things live only while the app is running, like a variable or a screen's temporary state. Non-volatile data is written to the phone's storage chip (or a server) and is still there after a restart. Every place below is non-volatile - it all survives the phone turning off and on. (The cache is the one twist: it is non-volatile, but the system may delete it to free up space.)
Want to see this in real code? The runnable Storage Showcase app puts every mechanism on this page into one Compose app (one tab each), and How Storage Showcase works is a file-by-file walkthrough of how it's wired together.

1. A short list of settings

For small choices and short answers - "is dark mode on?", "what name did they type?". You keep a little list of labels and their values. It is the simplest place: this setting equals that value.

Good for: on/off switches, the user's name, the last tab they opened.

Android calls this DataStore (the older version was SharedPreferences).

Where on the phone: your app's own private area in internal storage. Volatile? Non-volatile - it survives the phone restarting.
🗄️

2. A filing cabinet for lots of records

When you have many similar things - notes, to-do items, contacts - and you need to find, sort, or filter them, you want a database. Picture a spreadsheet: each thing is a row, and you can search through them.

Good for: a notes app's notes, a to-do list, saved messages.

Android's tool for this is Room (a helper for the database built into every phone).

Where on the phone: a database file in your app's private area in internal storage. Volatile? Non-volatile - survives a restart.
📁

3. Your app's own private folder

Sometimes the data is just a file - a chunk of text, or a picture your app made. You can save it in a folder only your app is allowed to open, like a drawer only you have the key to.

Good for: a saved document, a drawing, data your app reads back later.

Android calls this internal storage (the folder is named filesDir).

Where on the phone: your app's private folder in internal storage. Volatile? Non-volatile - survives a restart.
🧽

4. A scratch folder you can lose

For things you can always make again - like a smaller copy kept around to make the app feel faster - there is a "cache". The phone may empty it whenever it needs space, so never keep anything important here. Think of scratch paper you would not mind tossing.

Good for: thumbnail images, a downloaded copy of something that is also online.

Android calls this the cache folder (cacheDir).

Where on the phone: your app's cache folder in internal storage. Volatile? Non-volatile, but the phone may erase it at any time to free up space.
🖼️

5. The phone's shared photo & music library

If you save a photo, video, or song the user should see in their Gallery or Music app - not just hidden inside your app - it goes into the phone's shared library that all apps can use. Like adding a photo to the family album everyone can open.

Good for: a photo the user took in your app, a video you exported.

Android calls this MediaStore.

Where on the phone: the shared storage area that all apps can use. Volatile? Non-volatile - survives a restart, and even your app being uninstalled.
🤝

6. Let the user pick or place a file

Sometimes you do not store the file yourself - you ask the user. "Pick a photo for your avatar." "Choose where to save this PDF." The phone shows a picker, the user chooses, and your app gets just the one file they pointed at.

Good for: choosing a profile picture, importing a document, exporting a file.

Android calls these the Photo Picker (for pictures and videos) and the Storage Access Framework (for other documents).

Where on the phone: wherever the file already lives - the phone's storage, or even a cloud app like Google Drive. Volatile? Non-volatile - you are borrowing the user's file, not keeping your own copy.
☁️

7. Out on the internet (a server)

Everything above lives on this one phone. If the data must also show up on the user's other devices, or be shared with other people, it has to live on the internet instead - on a server or a cloud database. That is a bigger topic, but it is why "where does this live?" is sometimes "not on the phone at all."

Good for: an account that follows you between phones, shared documents.

No single Android name - you would use a backend such as Firebase or your own server.

Where on the phone: nowhere - it lives on another computer (a server) out on the internet. Volatile? Non-volatile - stored on the server's disk.
How do I pick one? Three plain questions usually decide it:
1. One tiny setting, or a pile of records? A tiny setting goes in the settings list; a pile of records goes in the database.
2. Just for your app, or should the user see it in other apps? Just yours stays in your private folder or database; if the user should see it, use the shared library - or let the user pick.
3. Must it stick around, or can you remake it? Must stick around: settings, files, or the database. Can remake it: the cache.
Does my data stick around? Your settings, files, and database stay until the app is uninstalled. The cache can disappear at any time. A photo you place in the shared library stays even after your app is gone - it belongs to the user now.

The useful mental model

Android storage is a set of scoped places, not one shared disk.

Picking where to put data comes down to five plain questions: Who owns it - your app or the user? Who should be able to see it? Is it neat rows, or just a blob of bytes? Should it survive the app being uninstalled? And should the user pick the file or photo themselves?

Rule of thumb: default to the narrowest storage surface that satisfies the feature. Private app state belongs in the app sandbox. User-visible media belongs in MediaStore. Arbitrary user documents belong behind the system picker.
Private app data Room, DataStore, filesDir, cacheDir. Other apps cannot browse it. Removed when the app is uninstalled.
Shared user data Photos, videos, audio, downloads, and documents. Use system APIs so Android mediates access.
Granted access Photo Picker and SAF return content URIs. Your app receives access to exactly what the user selected.

Storage type guide

Understand each storage type, not just the API name.

The important choice is not "file versus database." It is whether the data is private or shared, structured or raw bytes, disposable or durable, app-owned or user-owned, and chosen by the user or created by your app.

Use in code reviews
Is it queryable structured app data?Use Room when you need rows, indexes, relations, filtering, sorting, transactions, or migrations.
Is it a small private setting or one typed app object?Use Preferences DataStore for simple keys; use typed or Proto DataStore for a single schema-backed object.
Is it private app-owned bytes?Use filesDir for durable bytes, cacheDir for rebuildable bytes, and noBackupFilesDir for data that should not restore.
Should the user see it after uninstall?Use shared storage. Photos, videos, and audio belong in MediaStore. Arbitrary documents usually belong behind SAF.
Is the user selecting existing media or a document?Prefer Photo Picker for visual media and SAF for documents. They return content URIs and avoid broad storage access.
Are you about to request broad storage access?Stop and prove the app is a file manager, backup/restore tool, antivirus, or another narrowly eligible core use case.

Room and SQLite

Room: a real database for your app's structured data.

Reach for Room when your data looks like a spreadsheet - many records you sort, filter, or link together. You fetch it with a query, and because a query can return a Flow<List<Note>>, the screen gets a fresh list automatically every time the rows change. No manual refresh.

DataStore

            
DataStore: the modern home for small settings.

            
Use DataStore for little pieces of private state, like a dark-theme switch. Preferences DataStore holds simple key-value settings; typed DataStore holds one whole object (such as a user profile). Both read as a Flow and save safely in the background, which makes them the modern replacement for the older SharedPreferences.

Files and cache

            
Files keep private bytes; the cache holds bytes you can lose.

            
When your data is just a file - a text draft, an exported JSON - and you do not need database-style queries, write it to your app's private files folder. Use the cache folder only for things you can recreate, because the system or the user can wipe it at any time. Reading and writing files is slow, so always do it off the main (UI) thread.

MediaStore

            
MediaStore: the shared library for photos, video, and audio.

            
When the user should see a file in the Gallery or Files app - a photo you saved, a video you exported - it goes into MediaStore, the device's shared media library. You do not write to a raw path; you hand the file to the system and get back a URI that points to it. On Android 10 and higher, adding your own media needs no permission; reading other apps' media needs the Photo Picker or a media permission.

Storage Access Framework

            
Storage Access Framework: let the user pick the file.

            
For documents your app does not own - open this PDF, save my export here - you open the system file picker and let the user choose. Because the user picked it, your app needs no storage permission; it simply gets a URI for that one file or folder. (Android 11 and higher block picking whole sensitive folders such as the storage root, Downloads, and Android/data.)

Backup, privacy, and security

            
Saved is not the same as secret.

            
The app sandbox stops other apps from casually reading your files, but the files themselves are still plain, unencrypted data - "private" does not mean "encrypted". If something is truly sensitive, you have to encrypt it yourself. And whether a file is copied to cloud backup is a separate choice you control.
